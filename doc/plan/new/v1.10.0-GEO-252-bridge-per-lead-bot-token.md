# Plan: Bridge 事件通知使用 Per-Lead Bot Token

**Version**: v1.10.0
**Issue**: GEO-252
**Date**: 2026-03-24
**Source**: Linear issue GEO-252 (Bridge 事件通知无法投递到 Lead Discord Bot)
**Status**: codex-approved

## Problem

GEO-246 给每个 Lead 创建了独立的 Discord Bot（Peter → `PETER_BOT_TOKEN`、Oliver → `OLIVER_BOT_TOKEN`），但 Bridge 的 `ClaudeDiscordRuntime`、`ForumPostCreator`、`ForumTagUpdater` 都使用同一个全局 `DISCORD_BOT_TOKEN`（即 `config.discordBotToken`）。

如果这个 token 对应的 bot 没有权限访问 Peter/Oliver 的 control channel 或 forum channel，事件就投递失败。

## Solution: Per-Lead `botTokenEnv` in `LeadConfig`

在 `projects.json` 的 `LeadConfig` 中添加 `botTokenEnv` 字段（存 env var **名称**，不存明文 token），Bridge 在启动时 resolve 为实际 token，用于 runtime 创建和 Forum 操作。

### 为什么选方案 1（per-lead token）而非方案 2/3

- **方案 2**（ClaudeBot 加入所有 channel）：耦合——所有 Lead 依赖同一个 bot，权限管理混乱
- **方案 3**（infra bot）：多了一层不必要的抽象
- **方案 1**：与 GEO-246 多 Lead 架构一致，每个 Lead 用自己的身份，最干净

### Secret 管理策略

现有模式：bot token 存在 env vars / `~/.flywheel/.env` 中，`BridgeConfig.discordBotToken` 从 `process.env.DISCORD_BOT_TOKEN` 读取。

新模式：`projects.json` 中的 `botTokenEnv` 只存 **env var 名称**（如 `"PETER_BOT_TOKEN"`），`loadProjects()` 在解析时通过 `process.env[lead.botTokenEnv]` resolve 为实际 token 并存入 `lead.botToken`。

**闭环保证**：
- Secret 仍然只在 env vars 中，不进 config 文件
- `loadProjects()` 显式 strip JSON input 中的 raw `botToken`，防止 secret 绕过 env var 通道
- 如果 `botTokenEnv` 已配置但 env var 不存在，`claude-discord` lead fail-fast（不静默 fallback）
- 与现有 `DISCORD_BOT_TOKEN` 模式一致
- Config 文件可以安全地被 review 或 version control

## Changes

### 1. `LeadConfig` 添加 `botTokenEnv` + resolved `botToken`

**File**: `packages/teamlead/src/ProjectConfig.ts`

分离 JSON input 和 resolved 形状：`LeadConfig` 是最终使用的类型（含 resolved `botToken`），`botTokenEnv` 是 JSON 输入字段。

```typescript
export interface LeadConfig {
  agentId: string;
  forumChannel: string;
  chatChannel: string;
  match: { labels: string[] };
  runtime?: "openclaw" | "claude-discord";
  controlChannel?: string;
  /** Env var name for this lead's Discord bot token (e.g., "PETER_BOT_TOKEN"). */
  botTokenEnv?: string;
  /** Resolved bot token (populated at load time from botTokenEnv). NOT from JSON input. */
  botToken?: string;
}
```

**`loadProjects()` 变更**：
- `botTokenEnv` 为 optional string，不做 required 校验（backward compatibility）
- 如果 `botTokenEnv` 存在，resolve `process.env[lead.botTokenEnv]` 并赋值给 `lead.botToken`
- **Misconfiguration detection**：如果 `botTokenEnv` 已配置但对应 env var 不存在：
  - `runtime === "claude-discord"` → **throw** (fail-fast，避免静默回退到全局 token)
  - 其他 runtime → warn 但不阻塞
- **Strip raw `botToken`**：如果 JSON input 中包含 `botToken` 字段，显式 `delete lead.botToken` 后再从 `botTokenEnv` resolve。防止 secret 通过 config 文件传入。

```typescript
// In loadProjects() validation, after runtime checks:
if (typeof lead.botTokenEnv === "string" && lead.botTokenEnv.length > 0) {
  const resolved = process.env[lead.botTokenEnv];
  if (!resolved) {
    if (runtime === "claude-discord") {
      throw new Error(
        `Project "${entry.projectName}" leads[${i}]: botTokenEnv="${lead.botTokenEnv}" is set but env var is not defined`,
      );
    }
    console.warn(
      `[loadProjects] "${entry.projectName}" leads[${i}]: botTokenEnv="${lead.botTokenEnv}" not found in env — will fall back to DISCORD_BOT_TOKEN`,
    );
  }
  lead.botToken = resolved;
} else {
  // Strip any raw botToken from JSON input
  delete lead.botToken;
}
```

### 2. `createLeadRuntime()` 使用 `lead.botToken`

**File**: `packages/teamlead/src/bridge/plugin.ts` (lines 31-53)

```typescript
export async function createLeadRuntime(
  lead: LeadConfig,
  config: BridgeConfig,
): Promise<LeadRuntime> {
  if (lead.runtime === "claude-discord") {
    if (!lead.controlChannel) {
      throw new Error(`Lead "${lead.agentId}" has runtime=claude-discord but missing controlChannel`);
    }
    // GEO-252: use per-lead botToken, fall back to global DISCORD_BOT_TOKEN
    const token = lead.botToken ?? config.discordBotToken;
    if (!token) {
      throw new Error(
        `Lead "${lead.agentId}" has runtime=claude-discord but no botToken (botTokenEnv=${lead.botTokenEnv ?? "unset"}) and DISCORD_BOT_TOKEN is not set`,
      );
    }
    const { ClaudeDiscordRuntime } = await import("./claude-discord-runtime.js");
    return new ClaudeDiscordRuntime(lead.controlChannel, token);
  }
  // ... openclaw unchanged
}
```

**Backward compatibility**: `lead.botToken ?? config.discordBotToken` — 没有 `botTokenEnv` 的旧配置仍然 fallback 到全局 token。

### 4. Out-of-scope: `/api/forum-tag` endpoint

`plugin.ts:303-345` 的 `/api/forum-tag` REST endpoint 使用 `config.discordBotToken`。这是一个 **infra-level API**（接受任意 `thread_id` + `tag_ids`），不与特定 Lead 关联。故意保留全局 token，不属于 GEO-252 scope。

### 5. Out-of-scope: `CleanupService`

`plugin.ts:744-745` 的 `FetchDiscordClient(config.discordBotToken)` 用于清理 archived threads。同样是 infra-level 操作，保留全局 token。

### 6. `BridgeConfig.discordBotToken` 保留

`BridgeConfig.discordBotToken` 不删除，作为：
- `/api/forum-tag` endpoint 的 token
- `CleanupService` 的 token
- Per-lead token 的 fallback
- 不需要改 `config.ts`

## Risks

- **Low**: Backward-compatible — 旧配置无 `botTokenEnv` 仍 fallback 到 `DISCORD_BOT_TOKEN`
- **Low**: `loadProjects()` 新增 fail-fast 只对 `claude-discord` + 已配置 `botTokenEnv` 的 lead 生效
- **Mitigated**: Raw `botToken` in JSON input 被 strip，防止 secret 绕过 env var 通道
