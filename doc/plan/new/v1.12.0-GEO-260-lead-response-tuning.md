# Plan: Lead 回复行为调优 — 精准回答 CEO 问题

**Version**: v1.12.0
**Issue**: GEO-260
**Date**: 2026-03-26
**Status**: codex-approved

---

## 问题

CEO 问 Peter "现在在跑的有哪些 issue"，Peter 返回了三个分类（running, awaiting_review, failed），把全部状态 dump 出来。CEO 只问了 "在跑的"，不需要其他信息。

**根因**:
1. Agent prompt 中没有指导 Lead 如何匹配 CEO 问题意图。当前「沟通风格」section 只说了 "简洁" 和 "消化而非转发"，但没有教 Lead **精准匹配问题范围**。
2. `product-lead/agent.md` 的 API 示例缺少 `leadId` 参数，Lead 可能查到跨 scope 的数据（GEO-259 后端已支持，但 product-lead agent.md 未更新）。

## 方案

### Prompt 层改动（两部分）

**Part A**: 新增「精准回答」规则 — 教 Lead 匹配 CEO 问题意图。两个 Lead 都需要。
**Part B**: 修复 API 示例加 `leadId` — 仅 `product-lead` 需要（`ops-lead` 在 GEO-259 时已更新）。

不改 Bridge API — 当前 `mode=active` 返回 running + awaiting_review，Lead 在回复中按 `status` 字段过滤即可。

**为什么不加 Bridge API `status` 过滤参数**:
- Lead 拿到 JSON 后，每条 session 已有 `status` 字段，在回复中过滤是自然的 LLM 行为
- 问题不是 API 能力不足，而是 prompt 没有明确要求精准匹配
- 加 API 参数是 overengineering — 当 prompt 解决不了时再考虑

## Step 1: 新增精准回答规则（Part A — 两个文件）

在两个 agent 文件的「沟通风格」section 之后、「事件处理」section 之前，新增以下 subsection：

### 1.1 新增内容

```markdown
### 精准回答（严格执行）

**回答 CEO 的问题时，只回答 CEO 问的内容。不要 dump 全部状态。**

| CEO 问的 | 你应该返回的 | 你不应该返回的 |
|---------|------------|-------------|
| "在跑的有哪些" / "running" | 只列 status=running 的 session | ❌ awaiting_review, failed, completed |
| "需要我 review 的" | 只列 status=awaiting_review 的 session | ❌ running, failed |
| "最近失败了哪些" | 从 recent sessions 中筛出 status=failed 的（注: 数据基于最近 N 条，非全量） | ❌ running, awaiting_review |
| "整体状态" / "汇报一下" | 完整摘要，按状态分类 | ✅ 全部返回 |
| "GEO-XX 什么情况" | 该 issue 的当前状态 + 关键信息 | ❌ 其他 issue |

**紧急附加提醒规则**:
- 如果有 **failed session** 且 CEO 没问到，可以在回答末尾附加提醒
- 必须明确区分: "**你问的**: ..." 和 "**另外提醒**: ..."
- 非紧急信息（如 awaiting_review）不要主动附加

**示例**:

> CEO: "在跑的有哪些"
> Lead: "**你问的**: 当前有 2 个 session 正在运行：
> 1. GEO-301「优化搜索排序」— 已运行 12 分钟
> 2. GEO-305「修复导出 bug」— 已运行 3 分钟
>
> **另外提醒**: GEO-298 执行失败了，需要你看一下。"

> CEO: "汇报一下"
> Lead: "当前状态：
> - Running (2): GEO-301, GEO-305
> - Awaiting Review (1): GEO-299
> - Recent Failures (1): GEO-298 — tailwind config 冲突"
```

### 1.2 两个 agent 文件精准回答内容一致

Product Lead 和 Ops Lead 使用完全相同的精准回答规则文本。

## Step 2: 修复 product-lead API 示例添加 leadId（Part B — 仅 product-lead）

`ops-lead/agent.md` 在 GEO-259 时已更新 `leadId`，本步骤仅更新 `product-lead/agent.md` 使其收敛到相同目标状态。

### 2.1 新增 leadId 说明段

在「工具」section 的 `### Bridge API（通过 Bash curl）` 下、endpoint 表格前，加一行：

```markdown
**重要：所有批量查询必须带 `leadId=product-lead`，只查你负责范围内的 sessions。所有 action body 必须带 `"leadId": "product-lead"`。**
```

### 2.2 修改 Bridge API 表格

将 endpoint 表格更新为（对齐 ops-lead 已有格式）：

```
| Endpoint | Method | 用途 |
|----------|--------|------|
| `/api/sessions?leadId=product-lead` | GET | 活跃 sessions（你的 scope） |
| `/api/sessions?mode=recent&limit=10&leadId=product-lead` | GET | 最近 sessions |
| `/api/sessions?mode=stuck&leadId=product-lead` | GET | Stuck sessions |
| `/api/sessions?mode=by_identifier&identifier=GEO-XX` | GET | 按 identifier 查询（不需要 leadId） |
| `/api/sessions/{id}` | GET | Session 详情 |
| `/api/sessions/{id}/history?leadId=product-lead` | GET | 执行历史（你的 scope） |
| `/api/sessions/{id}/capture?lines=100` | GET | 抓取 Runner tmux 终端输出 |
| `/api/resolve-action?issue_id={id}&action={action}&leadId=product-lead` | GET | 确认 action 可执行 |
| `/api/actions/{action}` | POST | 执行 action（body 带 `"leadId":"product-lead"`） |
| `/api/linear/create-issue` | POST | 创建 Linear issue |
| `/api/linear/update-issue` | PATCH | 更新 Linear issue |
| `/api/config/discord-guild-id` | GET | 获取 Guild ID |
```

### 2.3 修改 Bubble DOWN curl 示例

将「Bubble DOWN — CEO 指令执行」中的 curl 示例更新，对齐 ops-lead 已有格式：

```bash
# Step 1: 确认可执行（带 leadId）
curl -s -H "Authorization: Bearer $TEAMLEAD_API_TOKEN" \
  "$BRIDGE_URL/api/resolve-action?issue_id=GEO-XX&action=approve&leadId=product-lead"
# 返回: {can_execute, execution_id, reason}

# Step 2: 执行 action（只在 can_execute=true 时，body 带 leadId）
curl -s -X POST -H "Authorization: Bearer $TEAMLEAD_API_TOKEN" \
  -H "Content-Type: application/json" \
  $BRIDGE_URL/api/actions/approve \
  -d '{"execution_id":"...", "identifier":"GEO-XX", "leadId":"product-lead"}'
```

### 2.4 修改 Bubble DOWN 表格 by_identifier 行

将 `"GEO-XX 什么情况"` 对应的查询更新为匹配 ops-lead 格式：

```
| "GEO-XX 什么情况" | `GET /api/sessions?mode=by_identifier&identifier=GEO-XX` (不需要 leadId) |
```

## 风险

| 风险 | 影响 | 缓解 |
|------|------|------|
| LLM 不遵守 prompt 指导 | CEO 仍然收到全部 dump | 用表格 + few-shot 示例增强约束力 |
| 过度精简导致遗漏紧急信息 | CEO 不知道有 failed session | "**另外提醒**" 规则 + 示例展示分层格式 |
| Bootstrap 分类偏置 | Bootstrap 按 Active/Pending/Failed 分类，可能强化 dump 习惯 | Few-shot 示例明确展示精准 vs 全量的对比 |
| Resume 旧 session 导致旧 prompt 生效 | Prompt 更新无效 | 验证步骤要求 fresh session（删除 session ID 文件） |
