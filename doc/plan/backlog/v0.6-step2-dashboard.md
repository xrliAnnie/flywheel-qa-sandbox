# v0.6 Step 2: Operations Dashboard

## Summary

为 Flywheel 添加实时 Web Dashboard，显示正在运行的 sessions、stuck sessions、关键指标。参考 OpenAI Symphony 的 dashboard 设计，但基于现有的 Express + SQLite 架构实现。

## Background

### 现状

- TeamLead bridge（Express, port 9876）已有完整的 REST API
- SQLite 存储了所有 session 和 event 数据
- 但没有任何可视化界面——只能通过 API 或日志查看状态

### 目标

打开浏览器访问 `http://localhost:9876/` 就能看到：
1. 正在运行的 sessions（issue、status、runtime、branch）
2. 等待审批的 sessions
3. Stuck sessions（运行超时）
4. 关键指标（running count、awaiting review、completed today、failed）

### 参考: Symphony Dashboard

Symphony 使用 Phoenix LiveView 实现。关键设计：

- **4 个指标卡片**: Running count, Retrying count, Total tokens, Runtime
- **Running Sessions 表格**: Issue | State | Session | Runtime/turns | Last update | Tokens
- **Retry Queue 表格**: Issue | Attempt | Due at | Error
- **实时更新**: PubSub push + 1s timer tick（更新 runtime 计数器）
- **Live/Offline 状态**: CSS-only detection via WebSocket connection state
- **CSS**: OpenAI 设计系统，glassmorphism，响应式 grid

我们的实现更简单：Express serve 静态 HTML + SSE 推送。

## Design

### Architecture

```
Browser ──GET /──> Express (inline HTML response)
Browser ──GET /sse──> Express (Server-Sent Events stream)
                         ↓ lazy poller (starts on first client, stops on last disconnect)
                     StateStore (SQLite)
```

不引入任何前端框架（React/Vue/etc）。Dashboard HTML 作为 TypeScript 字符串模板内联在代码中，避免静态文件复制问题。SSE 用 lazy 共享 poller 广播给所有连接。

### Session Status Taxonomy

Dashboard 需要正确处理所有现有 status 值：

| Status | 类别 | Dashboard 位置 | 图标 |
|--------|------|---------------|------|
| `running` | Active | Active Sessions, metrics | 🟢 |
| `awaiting_review` | Active | Active Sessions, metrics | 🟡 |
| `completed` | Terminal/Outcome | Recent Outcomes | ⬜ |
| `approved` | Terminal/Outcome | Recent Outcomes | ✅ |
| `blocked` | Terminal/Outcome | Recent Outcomes | ⛔ |
| `failed` | Terminal/Outcome | Recent Outcomes | 🔴 |
| `rejected` | Terminal/Outcome | Recent Outcomes | ❌ |
| `deferred` | Terminal/Outcome | Recent Outcomes | ⏸ |
| `shelved` | Terminal/Outcome | Recent Outcomes | 📦 |

**重要**：`rejected`、`deferred`、`shelved` 是由 action 系统写入的状态（见 `actions.ts` 的 `ACTION_TARGET_STATUS`）。Dashboard 必须包含它们，否则这些 session 会从 UI 中消失。

### Data Source

Dashboard 数据由服务端 `buildDashboardPayload()` 直接读取 `StateStore` 组装，**不经过 `/api/*` 路由**。这样避免 `apiToken` 认证问题和响应格式转换（`/api` 会移除 `issue_id` 并重命名 `issue_identifier` 为 `identifier`）。

| Dashboard 区域 | StateStore 方法 | 说明 |
|---------------|----------------|------|
| Running count | `getActiveSessions()` | 过滤 `status === 'running'` 计数 |
| Awaiting Review | `getActiveSessions()` | 过滤 `status === 'awaiting_review'` 计数 |
| Completed Today | `getTerminalSessionsSince(todayStart)` | **新增方法**，见 Task 1 |
| Failed count | `getTerminalSessionsSince(todayStart)` | 过滤 `status === 'failed'` 计数 |
| Active Sessions | `getActiveSessions()` | 全部返回（running + awaiting_review） |
| Recent Outcomes | `getRecentOutcomeSessions(10)` | **新增方法**，直接在 SQL 层过滤终态 |
| Stuck Sessions | `getStuckSessions(threshold)` | 运行超时的 session |

### Metric 定义

| 指标 | 定义 | 计算方式 |
|------|------|---------|
| Running | 当前 status='running' 的 session 数 | `getActiveSessions()` 过滤 |
| Awaiting Review | 当前 status='awaiting_review' 的 session 数 | `getActiveSessions()` 过滤 |
| Completed Today | 今日进入终态的 session 数（含 completed, approved, blocked, rejected, deferred, shelved） | **新方法** `getTerminalSessionsSince(todayStartUTC)` |
| Failed | 今日 status='failed' 的 session 数 | 同上方法，过滤 failed |

**时区约定**：`todayStart` 使用 UTC 日期边界（`YYYY-MM-DD 00:00:00`），与 `sqliteDatetime()` 的 ISO 格式一致。仪表盘显示 "Today (UTC)"。

### SSE (Server-Sent Events)

```
GET /sse
Content-Type: text/event-stream
Cache-Control: no-cache
Connection: keep-alive

event: state
data: { "metrics": {...}, "active": [...], "recent": [...], "stuck": [...] }

(immediately on connect, then every 2 seconds)
```

**架构：Lazy 共享 poller + 广播**

SSE poller 和 heartbeat timer 采用 lazy 启动策略：第一个 SSE client 连接时启动，最后一个 client 断开时停止。这样避免 0 client 时仍然每 2s 查询 SQLite。

```typescript
// Lazy SSE broadcaster — encapsulated in a helper class
class SseBroadcaster {
  private clients = new Set<express.Response>();
  private poller: NodeJS.Timeout | null = null;
  private heartbeat: NodeJS.Timeout | null = null;

  constructor(
    private store: StateStore,
    private stuckThresholdMinutes: number,
  ) {}

  addClient(res: express.Response): void {
    // Send initial state immediately
    const payload = buildDashboardPayload(this.store, this.stuckThresholdMinutes);
    res.write(`event: state\ndata: ${JSON.stringify(payload)}\n\n`);

    this.clients.add(res);
    if (this.clients.size === 1) this.startPolling();
  }

  removeClient(res: express.Response): void {
    this.clients.delete(res);
    if (this.clients.size === 0) this.stopPolling();
  }

  /** Called by startBridge close() for graceful shutdown.
   *  Ends all active SSE connections so server.close() can complete. */
  destroy(): void {
    this.stopPolling();
    for (const client of this.clients) {
      try {
        client.write(": server shutting down\n\n");
        client.end();
      } catch { /* already closed */ }
    }
    this.clients.clear();
  }

  private startPolling(): void {
    this.poller = setInterval(() => { /* broadcast state */ }, 2000);
    this.heartbeat = setInterval(() => { /* broadcast : heartbeat */ }, 30000);
  }

  private stopPolling(): void {
    if (this.poller) { clearInterval(this.poller); this.poller = null; }
    if (this.heartbeat) { clearInterval(this.heartbeat); this.heartbeat = null; }
  }
}
```

- **首帧立即推送**：新连接建立后立刻发送一次完整 state，不需要等 2s
- **Heartbeat**：每 30s 发送 SSE comment（`: heartbeat\n\n`）防止代理/浏览器超时断开
- **清理**：`req.on('close')` 时调用 `removeClient()`；`res.write()` 失败时也移除
- **SSE endpoint 不需要认证**：dashboard 只在本地访问，bridge 已经绑定 loopback

### CSS Design

参考 Symphony 的设计语言但更简洁：
- Dark theme（适合监控界面）
- Monospace font for data（tabular-nums）
- Status badges: 🟢 running, 🟡 awaiting_review, 🔴 failed, ⬜ completed, ✅ approved, ⛔ blocked, ❌ rejected, ⏸ deferred, 📦 shelved
- Responsive grid for metric cards
- 不用 glassmorphism（简单实用优先）

### 安全渲染约束

Dashboard 展示的所有 session 字段（`issue_title`、`last_error`、`decision_reasoning`、`summary` 等）来自事件 payload 或用户动作，属于外部输入。

**硬性约束**：
- 所有 session 派生文本**只能**通过 `textContent` 或等价的安全 DOM API 渲染
- **禁止**将数据库/事件字段直接插入 `innerHTML`
- 如果需要模板拼接，必须先通过 HTML escape 函数处理
- Client-side JS 必须包含一个 `escapeHtml()` 工具函数，用于任何不得不使用 innerHTML 的场景（如表格行批量更新）

这符合项目 CLAUDE.md 的非协商项："External input must be validated at system boundaries"。

### Task 0: 统一 Session Status 常量（前置任务）

**文件**: `packages/teamlead/src/StateStore.ts`（修改）

**问题**：当前 `StateStore` 的 `TERMINAL_STATUSES` 只包含 5 个状态：

```typescript
// 当前代码 (StateStore.ts:6-12)
const TERMINAL_STATUSES = new Set([
  "completed", "awaiting_review", "approved", "blocked", "failed",
]);
```

但 action 系统（`actions.ts`）会写入 `rejected`、`deferred`、`shelved` 三个额外的终态。这意味着如果一个 session 被 reject/defer/shelve 后，迟到的 `session_started` 事件可以把它的 status 重新写回 `running`，因为 monotonic guard 不认识这些新状态。

**修改**：

1. 将 `TERMINAL_STATUSES` 扩展为包含所有终态：

```typescript
const TERMINAL_STATUSES = new Set([
  "completed", "awaiting_review", "approved", "blocked", "failed",
  "rejected", "deferred", "shelved",
]);
```

2. 导出一个共享的 outcome status 常量，供 dashboard 查询和 badge 映射使用：

```typescript
/** All statuses that represent a final outcome (used by dashboard, queries). */
export const OUTCOME_STATUSES = [
  "completed", "approved", "blocked", "failed",
  "rejected", "deferred", "shelved",
] as const;
```

3. 更新新增的 `getTerminalSessionsSince()` 和 `getRecentOutcomeSessions()` 使用 `OUTCOME_STATUSES` 构建 SQL IN 子句，而不是硬编码字符串列表。这样未来新增状态只需在一处修改。

**测试**：
- 已有 StateStore 测试可能需要更新，确认 `rejected`/`deferred`/`shelved` → `running` 的转换被 monotonic guard 拦截
- 新增测试：`upsertSession` 不允许 `rejected` → `running` 转换

getTerminalSessionsSince(sinceTs: string): Session[] {
  const placeholders = OUTCOME_STATUSES.map(() => "?").join(", ");
  const stmt = this.db.prepare(
    `SELECT * FROM sessions
     WHERE status IN (${placeholders})

### Task 2: Dashboard Data Assembler

**文件**: `packages/teamlead/src/bridge/dashboard-data.ts`（新建）

```typescript
import type { StateStore, Session } from "../StateStore.js";

export interface DashboardMetrics {
  running: number;
  awaiting_review: number;
  completed_today: number;
  failed_today: number;
}

export interface DashboardSession {
  execution_id: string;
  issue_identifier?: string;
  issue_title?: string;
  project_name: string;
  status: string;
  started_at?: string;
  last_activity_at?: string;
  branch?: string;
  last_error?: string;
  decision_route?: string;
  commit_count?: number;
  lines_added?: number;
  lines_removed?: number;
}

export interface DashboardPayload {
  metrics: DashboardMetrics;
  active: DashboardSession[];    // running + awaiting_review
  recent: DashboardSession[];    // last 10 terminal sessions
  stuck: DashboardSession[];     // stuck (running but stale)
  generated_at: string;          // ISO timestamp
}

export function buildDashboardPayload(
  store: StateStore,
  stuckThresholdMinutes: number,
): DashboardPayload
```

**逻辑**：
1. `store.getActiveSessions()` → 分拆 running / awaiting_review
2. `store.getTerminalSessionsSince(todayStartUTC)` → 计算 completed_today（非 failed 终态）/ failed_today
3. `store.getRecentOutcomeSessions(10)` → 最近 10 条终态 session（SQL 层过滤）
4. `store.getStuckSessions(stuckThresholdMinutes)` → stuck sessions
5. 组装 payload，strip 掉 dashboard 不需要的字段（`issue_id` 等）

`buildDashboardPayload` 是同步函数（sql.js 查询是同步的）。

**`todayStartUTC` 计算**：

```typescript
function todayStartUTC(): string {
  const now = new Date();
  return `${now.getUTCFullYear()}-${String(now.getUTCMonth() + 1).padStart(2, "0")}-${String(now.getUTCDate()).padStart(2, "0")} 00:00:00`;
}
```

**关键约束**：`createBridgeApp()` 的返回类型保持 `express.Application` 不变，不破坏现有调用方和测试。SSE cleanup 由 `startBridge()` 内部管理。

**替代方案（更简洁）**：将 `SseBroadcaster` 作为可选参数传入 `createBridgeApp()`。当不传入时（测试场景），`/sse` 返回单次快照；当传入时（`startBridge`），`/sse` 使用 broadcaster 的 streaming 行为。

```typescript
export function createBridgeApp(
  store: StateStore,
  projects: ProjectEntry[],
  config: BridgeConfig,
  broadcaster?: SseBroadcaster,     // optional, provided by startBridge
): express.Application {
  // ...
  app.get("/sse", (req, res) => {
    if (broadcaster) {
      // Streaming mode
      res.setHeader("Content-Type", "text/event-stream");
      res.setHeader("Cache-Control", "no-cache");
      res.setHeader("Connection", "keep-alive");
      res.flushHeaders();
      broadcaster.addClient(res);
      req.on("close", () => broadcaster.removeClient(res));
    } else {
      // Snapshot mode (tests)
      res.setHeader("Content-Type", "text/event-stream");
      const payload = buildDashboardPayload(store, config.stuckThresholdMinutes);
      res.write(`event: state\ndata: ${JSON.stringify(payload)}\n\n`);
      res.end();
    }
  });
}
```

`startBridge()` 的 `close()` 调用 `broadcaster.destroy()`。

**注意事项**：
- Dashboard 路由必须在 catch-all 404 handler 之前注册
- `createBridgeApp()` 返回类型保持 `express.Application`，不破坏现有测试
- `broadcaster` 参数可选，测试中不传也能正常工作

## Scope

### In Scope
- 统一 `TERMINAL_STATUSES` / `OUTCOME_STATUSES` 常量（修复现有 bug：`rejected`/`deferred`/`shelved` 缺少 monotonic guard）
- 内联 HTML dashboard（TypeScript 字符串模板）
- SSE 实时推送（lazy 共享 poller，2s 间隔，首帧立即推送，graceful shutdown）
- Metric cards + Active sessions table + Recent outcomes + Stuck sessions
- 暗色主题、响应式布局
- `StateStore.getTerminalSessionsSince()` 新增聚合查询
- `StateStore.getRecentOutcomeSessions()` 新增 SQL 层过滤查询
- 安全 DOM 渲染（textContent / escapeHtml）
- 自动化测试（unit + route integration + broadcaster lifecycle + 安全渲染）

### Out of Scope
- 用户认证（loopback only，不需要）
- Action 按钮（approve/reject 等——未来可加）
- 多项目切换（目前只有一个项目，未来再加）
- Token/cost 追踪（subscription model，不需要）
- Terminal TUI（不如 web dashboard 实用）
- 前端框架（React/Vue 等——过度工程）
- PR URL 展示（Session 没有 PR URL 字段，需要额外 API 集成）
- Executor 字段（Session 没有 executor 字段）
- Retry attempt/due_at 元数据（当前不支持结构化重试队列）

## Dependencies

- Express.js（已有）
- StateStore（已有，需小幅扩展）

## Risks

1. **SSE 连接数**: 如果很多浏览器 tab 打开 dashboard，每个都会加入 broadcast set。Lazy poller 确保 0 client 时不占资源，有 client 时只有一次 SQLite 查询。对于本地使用不是问题。
2. **SQLite 读取频率**: 每 2s 查一次（不管连接数）。sql.js 是同步读取，性能足够。
3. **`completed_today` 近似性**: 使用 `last_activity_at >= todayStartUTC` 而非 dedicated `completed_at` 字段。可能在 session 跨天更新时不精确，但对运营仪表盘足够。
4. **HTML 内联维护性**: dashboard HTML 作为 TS 字符串维护，IDE 没有 HTML 语法高亮。如果 dashboard 变复杂，可能需要迁移到独立文件 + copy 步骤。当前 300-400 行规模可接受。
5. **Status 枚举同步**: 新增 action status（如未来添加 `escalated`）时只需更新 `OUTCOME_STATUSES` 常量和 dashboard 的 status badge 映射。共享常量确保 SQL 查询自动同步。
