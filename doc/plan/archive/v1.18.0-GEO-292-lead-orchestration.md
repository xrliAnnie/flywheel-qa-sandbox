## Overview

为 Discord Lead（Peter/Oliver/Simba）增加编排可见性：PR number 追踪 + 细粒度执行阶段追踪。借鉴 Orchestrator 的 9 步 step template 设计思路，适配 Runner pipeline 定义 10 个 session_stage transport 状态。借鉴 AgentsMesh 的 Autopilot 断路器设计思想（仅设计模式，不复制代码，遵守 BSL-1.1 license）。

分两个 Step 独立交付。Step 1 是 PR number 追踪（1-2 天），Step 2 是 Session Stage 追踪（5-7 天，含 Runner 侧上报机制）。

## Step 1: Artifact Recording — PR Number 追踪

### 目标

Session 完成后，Lead 通过 `GET /api/sessions` 查询 session 时能直接看到 PR number，不用翻 GitHub。

### 现状

- `LandingStatus.prNumber` 已存在于 `packages/core/src/decision-types.ts:18`
- `emitCompleted()` 已将 `evidence.landingStatus` 传递到 event payload
- `event-route.ts:302` 的 `patchSessionMetadata()` 提取了 `decision_route`、`commit_count` 等，但**跳过了** `evidence.landingStatus.prNumber`
- Session 表没有 `pr_number` 字段
- `DirectEventSink.emitCompleted()` 同样没有提取 `prNumber`

### 实现

#### 1.1 StateStore Migration

`packages/teamlead/src/StateStore.ts` — `migrate()` 方法尾部添加：

```typescript
// GEO-292: PR number tracking
try {
    this.db.run("ALTER TABLE sessions ADD COLUMN pr_number INTEGER");
} catch {
    /* exists */
}
```

#### 1.2 Session Interface 更新

`packages/teamlead/src/StateStore.ts` — `SessionUpsert` 和 `Session` interface 加：

```typescript

```

#### 1.3 upsertSession SQL 更新

`upsertSession()` 的 INSERT/UPDATE SQL 加 `pr_number` 字段，`COALESCE(excluded.pr_number, pr_number)` 模式。

#### 1.4 persistTransition SQL 更新

`persistTransition()` 的 INSERT/UPDATE SQL 加 `pr_number` 字段，与 `upsertSession()` 相同的 COALESCE 模式。`persistTransition` 是 FSM 生产路径（由 `applyTransition()` 调用），必须与 `upsertSession` 同步。

#### 1.5 patchSessionMetadata fieldMap 更新

`patchSessionMetadata()` 的 `fieldMap` 对象加 `pr_number: "pr_number"` 条目。当前 `patchSessionMetadata` 使用显式 whitelist（`fieldMap` 对象），不在 whitelist 中的字段会被静默忽略。

#### 1.6 rowToSession 更新

`rowToSession()` 加 `pr_number` 字段映射：

```typescript

```

不加此行，即使数据写入 DB，`getSession()` / `getActiveSessions()` 等查询函数返回的 Session 对象也不会包含该字段。

#### 1.7 Event Handler 提取

`packages/teamlead/src/bridge/event-route.ts` — `session_completed` handler 中，在 `patchSessionMetadata()` 调用里增加：

```typescript
pr_number: asNumber(
    (evidence?.landingStatus as Record<string, unknown> | undefined)?.prNumber
),
```

同时在 legacy `upsertSession()` 路径（`else` 分支，line 320）也加 `pr_number` 提取。

注意：`evidence.landingStatus` 在 event payload 中是 `Record<string, unknown>` 类型（JSON 反序列化后），需要 type narrowing。

#### 1.8 DirectEventSink 同步

`packages/teamlead/src/DirectEventSink.ts` — `emitCompleted()` 的 `upsertSession()` 调用里增加：

```typescript

```

DirectEventSink 是 bridge-local 事件路径（注释 "Mirrors event-route.ts logic"），必须与 HTTP ingest 路径保持一致。注意 DirectEventSink 中 `result.evidence` 是强类型 `ExecutionEvidence`，不需要 type narrowing。

#### 1.9 API 返回

`packages/teamlead/src/bridge/tools.ts` — `omitIssueId()` 函数已返回 Session 全部字段（`...rest`），所以**不需要额外改动** — 新字段自动包含在 API response 中。**前提是 `rowToSession()` 已更新**（1.6）。

## Step 2: Session Stage Tracking — 细粒度 Pipeline 追踪

### 目标

Lead 通过 `GET /api/sessions` 能看到 Runner 执行的**细粒度 pipeline 阶段**，知道 Runner 走到哪一步，基于此做 proactive 干预决策。

### Stage 定义

**10 个 session_stage transport 状态**，分三类：

| 类型 | Stage 值 | 说明 |
|------|---------|------|
| Lifecycle | `started` | Runner 已启动，尚未进入 pipeline。auto-set by session_started |
| Pipeline (Runner-reported) | `brainstorm` `research` `plan` `design_review` `implement` `test` `code_review` | 7 个 Runner 主动上报的 pipeline 阶段 |
| Inferred | `pr_created` `ship` | 2 个 Bridge 自动推断的阶段（Runner 也可主动上报） |

**与 Orchestrator step_templates 的关系**：

Orchestrator 有 9 步 executor template（Verify → Brainstorm → Research → Plan+DR → Implement → CR → User Approval → Ship → Post-Ship）。session_stage 借鉴其设计但不是 1:1 映射：
- Verify / User Approval / Post-Ship 不需要独立 stage（分别由 Runner 启动、approve action、Lead 处理）
- Plan+DR 拆成 `plan` + `design_review` 两个 stage
- 新增 `test` stage（Orchestrator 无此步但 Runner pipeline 需要）
- 新增 `started` 和 `pr_created` lifecycle/inferred 状态

**Stage 顺序值**（用于 auto-inference "只进不退" 判断）：

```typescript
export const STAGE_ORDER: Record<string, number> = {
    started: 0, brainstorm: 1, research: 2, plan: 3, design_review: 4,
    implement: 5, test: 6, code_review: 7, pr_created: 8, ship: 9,
};
```

### 设计借鉴

**AgentsMesh Autopilot 断路器**（设计模式层面借鉴，遵守 BSL-1.1）：
- **Stage 停留超时 → Lead 主动 capture**：类似 Autopilot 的 no-progress 检测。每个 stage 有预期时长，超时 Lead 应 capture 检查。
- **Stage 记录时间戳**：`stage_updated_at` 用于计算 stage 停留时间。

### 实现

Step 2 分两个 Phase 交付。Phase A 是 Bridge-side infrastructure（可独立运行，依赖 auto-inference + 手动 POST），Phase B 是 Runner-side stage reporting（让 7 个 pipeline 阶段自动可见）。

##### 2.7 Stage Constants

`packages/teamlead/src/bridge/stage-utils.ts`（新文件）：

```typescript
export const VALID_STAGES = new Set([
    "started", "brainstorm", "research", "plan", "design_review",
    "implement", "test", "code_review", "pr_created", "ship",
]);

export const STAGE_ORDER: Record<string, number> = {
    started: 0, brainstorm: 1, research: 2, plan: 3, design_review: 4,
    implement: 5, test: 6, code_review: 7, pr_created: 8, ship: 9,
};
```

event-route.ts 和 actions.ts import 这个模块。flywheel-comm 也 import（或自带副本——stage-utils 无外部依赖，可以在 flywheel-comm 中 inline 使用避免跨包依赖）。

##### 2.8 Event Handler — stage_changed

`packages/teamlead/src/bridge/event-route.ts` — 新增 `stage_changed` event type handler：

```typescript
} else if (event.event_type === "stage_changed") {
    const stage = asString(payload.stage);
    if (stage && VALID_STAGES.has(stage)) {
        store.patchSessionMetadata(event.execution_id, {
            session_stage: stage,
            stage_updated_at: now,
            last_activity_at: now,
        });
    }
}
```

**Stage 覆写规则**：
- **Runner 主动上报（stage_changed）**：直接覆写，不检查顺序。理由：Runner 可能跳过 stage（如 bug fix 直接从 started → implement），强制顺序会阻塞。Runner 最了解自己在哪个阶段。
- **Auto-inference（session_completed, approve）**：只进不退，用 STAGE_ORDER 比较。理由：自动推断是启发式，不应覆盖 Runner 的明确报告。
- 总是更新 `stage_updated_at` 和 `last_activity_at`

##### 2.10 session_completed 自动推断

`event-route.ts` 的 `session_completed` handler 中，在 `patchSessionMetadata()` 后加：

```typescript
// Auto-infer stage from landing status (only advance, never regress)
const landingStatusObj = evidence?.landingStatus as Record<string, unknown> | undefined;
const landingStatusValue = asString(landingStatusObj?.status);
if (prNumber) {
    const inferredStage = landingStatusValue === "merged" ? "ship" : "pr_created";
    const currentSession = store.getSession(event.execution_id);
    const currentOrder = STAGE_ORDER[currentSession?.session_stage ?? ""] ?? -1;
    const inferredOrder = STAGE_ORDER[inferredStage] ?? -1;
    if (inferredOrder > currentOrder) {
        store.patchSessionMetadata(event.execution_id, {
            session_stage: inferredStage,
            stage_updated_at: now,
        });
    }
}
```

同时在 `DirectEventSink.emitCompleted()` 中加相同的推断逻辑。

##### 2.11 Approve Action 自动设置 ship

#### Phase B: Runner-Side Stage Reporting（2-3 天）

##### 2.13 Event Ingest HTTP 契约

`flywheel-comm stage set` 的 HTTP 请求必须符合现有 event ingest endpoint（`/events`，`packages/teamlead/src/bridge/event-route.ts:100-112`）的要求。

**必填字段**（5 个）：

```json
{
    "event_id": "<generated UUID>",
    "execution_id": "<from FLYWHEEL_EXEC_ID env>",
    "issue_id": "<from FLYWHEEL_ISSUE_ID env>",
    "project_name": "<from FLYWHEEL_PROJECT_NAME env>",
    "event_type": "stage_changed",
    "source": "flywheel-comm",
    "payload": { "stage": "implement" }
}
```

**认证**：event ingest endpoint 受 `tokenAuthMiddleware(config.ingestToken)` 保护（`plugin.ts:302-305`）。flywheel-comm 需要通过 Authorization header 携带 ingest token。

**端点路径**：`POST ${FLYWHEEL_BRIDGE_URL}/events`（不是 `/events/ingest`）。

##### 2.17 flywheel-comm `stage` 命令

`packages/flywheel-comm/src/commands/stage.ts` — 新增 CLI 命令：

```bash
flywheel-comm stage set implement
flywheel-comm stage set code_review
```

实现：
1. 从环境变量读取：`FLYWHEEL_EXEC_ID`、`FLYWHEEL_ISSUE_ID`、`FLYWHEEL_PROJECT_NAME`、`FLYWHEEL_BRIDGE_URL`、`FLYWHEEL_INGEST_TOKEN`
2. 校验 stage 值在 VALID_STAGES 中（client-side 快速失败）
3. 生成 `event_id`（`crypto.randomUUID()`）
4. POST 到 `${FLYWHEEL_BRIDGE_URL}/events`。**Authorization header 可选**：如果 `FLYWHEEL_INGEST_TOKEN` 存在则加 `Bearer ${token}` header，不存在则不加 header。理由：`TEAMLEAD_INGEST_TOKEN` 在现有代码中是可选配置（`scripts/lib/setup.ts` 条件传递，`TeamLeadClient` 仅在 token 存在时加 header），stage CLI 应保持同样语义。
5. Body 包含 5 个 required fields + payload
6. **Fail-open 语义**：HTTP 失败（网络错误、Bridge 不可达、非 2xx 响应）时输出 warning 到 stderr 但 exit 0。Stage telemetry 不能打断 Runner 主流程。最多 1 次重试（2s timeout），不做 unbounded retry。
7. 成功输出 `Stage: implement`

##### 2.18 flywheel-comm CLI 注册 + async 改造

`packages/flywheel-comm/src/index.ts` — 两个改动：

1. **switch 加 `stage` case** + 更新 `printUsage()` 加 `stage` 命令描述。

2. **main() 改 async**。当前 `main()` 和所有 `runX()` 函数都是同步的（`index.ts:36-84`）。`flywheel-comm stage` 是 HTTP 异步命令，需要 `await`。改造方式：
   - `main()` → `async main()`
   - `runStage()` → `async runStage()`，返回 `Promise<void>`
   - 文件末尾加 `main().catch(err => { console.error(err.message); process.exit(1); })`
   - 现有同步命令（ask/check/pending 等）不受影响 — async function 中调用同步代码是 no-op。

## Runner 跳步规则

不是所有 task 都走全 pipeline。Runner agent.md 定义何时跳步，Bridge 不做 gate enforcement。

| Task 类型 | 典型 stage 路径 |
|-----------|----------------|
| 新功能（完整 pipeline） | started → brainstorm → research → plan → design_review → implement → test → code_review → pr_created → ship |
| Bug fix | started → implement → test → code_review → pr_created → ship |
| 纯文档 | started → research → implement → pr_created → ship |
| Refactoring | started → plan → implement → test → code_review → pr_created → ship |

## 不做的事

- **Gate enforcement**：不阻塞 stage 倒退或跳跃。Runner 明确上报可自由设置。只有 auto-inference 遵守"只进不退"。
- **Per-stage 超时阈值 + Lead 干预规则**：Step 3 范围。本 plan 只做基础设施 + CLI 工具 + 最小调用路径。
- **Session artifacts 表**：pr_number 直接加到 Session 字段。如果未来需要更多 artifact 类型再考虑独立表。
- **断路器自动暂停**：不自动暂停 Runner。只提供 stage 数据让 Lead 自行判断。
- **Bootstrap/Dashboard/HookPayload DTO 更新**：低优先级展示面。本期只保证 `GET /api/sessions` 可查。
