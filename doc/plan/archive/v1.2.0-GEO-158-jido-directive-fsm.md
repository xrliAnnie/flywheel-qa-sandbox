# Plan: Port Jido Directive Pattern + FSM Strategy

**Version**: v1.2.0
**Issue**: GEO-158
**Date**: 2026-03-15
**Source**: `doc/exploration/new/GEO-158-jido-directive-fsm.md`, `doc/research/new/GEO-158-fsm-directive-implementation.md`
**Status**: codex-approved

## Overview

将 Jido 的两个核心架构模式 port 到 Flywheel TypeScript 代码库：

1. **WorkflowFSM** — 声明式 transition map，替代 StateStore 的 monotonic guard + forceStatus()
2. **Directive Pattern** — audit-only directives。`WorkflowFSM.transition()` **内置**自动生成 AuditDirective（不依赖 onEnter），由 `applyTransition()` drain 写入 session_events

**选型**：Option B（FSM + Lightweight Directives）。不重构 event-route.ts / Blueprint.ts 的现有副作用代码。Directive 范围收敛到 **audit-only**（不含 notify/state_update），避免与现有通知路径混淆。

### Architecture

```mermaid

    subgraph "Core Package (flywheel-core)"
        FSM[WorkflowFSM]
        DT[Directive Types<br/>AuditDirective only]
    end

    subgraph "TeamLead Package"
        ATF[applyTransition<br/>统一入口]
        DE[DirectiveExecutor<br/>audit-only]

        HS[HeartbeatService]
    end

    AR -->|applyTransition()| ATF
    ER -->|applyTransition()| ATF
    HS -->|applyTransition()| ATF
    ATF -->|validate + guard| FSM
    ATF -->|persist status| SS
    ATF -->|drain audit| DE
    DE -->|insertEvent| SS
```

### Design Principles (Codex Round 1 反馈)

1. **统一 transition 入口** — 所有状态变更走 `applyTransition()`，不再有 `upsertSession(session, fsm)` 和 `fsm.transition()` 两条路径。`upsertSession()` 只负责非状态字段的 read-model 更新。
2. **首次 event 也验证** — 缺失 session 时隐式从 `pending` 开始，`applyTransition("pending", target)` 确保 FSM 校验覆盖所有路径。
3. **ACTION_DEFINITIONS 是 action 可见性的唯一来源** — `tools.ts` 和 dashboard 通过 `allowedActionsForState(state)` 纯函数查询（从 `ACTION_DEFINITIONS` 派生），不再维护独立的 `ACTION_SOURCE_STATUS` 映射。
4. **FSM 实例归属明确** — `createBridgeApp()` 接收 `transitionOpts` 参数，`startBridge()` 负责构建并传入，测试也显式传入。

#### Task 1.1: WorkflowFSM class

**File**: `packages/core/src/workflow-fsm.ts` (NEW)

```typescript
export interface TransitionContext {
  executionId: string;
  issueId: string;
  projectName: string;
  trigger: string;
  payload?: Record<string, unknown>;
}

export interface TransitionResult {
  ok: boolean;
  newState: string;
  directives: Directive[];
  error?: string;
}

export type GuardFn = (ctx: TransitionContext) => boolean;
export type OnEnterFn = (ctx: TransitionContext) => Directive[];

export class WorkflowFSM {
  constructor(
    private transitions: Record<string, string[]>,
    private guards?: Record<string, GuardFn>,
    private onEnter?: Record<string, OnEnterFn>,
  ) {}

  transition(currentState: string, targetState: string, ctx?: TransitionContext): TransitionResult
  allowedTransitions(currentState: string): string[]
  isTerminal(state: string): boolean
  canTransition(from: string, to: string): boolean
}
```

核心 `transition()` 逻辑：
1. 检查 `targetState` 是否在 `transitions[currentState]` 中
2. 如果有 guard（key = `"from → to"`），执行 guard，失败返回 error
3. 如果有 onEnter（key = `targetState`），执行生成 directives
4. 返回 `{ ok: true, newState, directives }` 或 `{ ok: false, error }`

#### Task 2.1: DirectiveExecutor class (audit-only)

**File**: `packages/teamlead/src/DirectiveExecutor.ts` (NEW)

```typescript
import type { Directive } from "flywheel-core";
import type { StateStore } from "./StateStore.js";

export interface DirectiveResult {
  type: string;
  success: boolean;
  error?: string;
}

export class DirectiveExecutor {
  constructor(private store: StateStore) {}

  /** Drain directives sequentially (audit order matters). */
  async drain(directives: Directive[]): Promise<DirectiveResult[]> {
    const results: DirectiveResult[] = [];
    for (const d of directives) {
      results.push(await this.execute(d));
    }
    return results;
  }

  private async execute(d: Directive): Promise<DirectiveResult> {
    switch (d.type) {
      case "audit":
        return this.handleAudit(d);
      default:
        return { type: d.type, success: false, error: `Unknown directive: ${d.type}` };
    }
  }

  private handleAudit(d: AuditDirective): DirectiveResult {
    // Write to session_events as event_type = "state_transition"
    this.store.insertEvent({
      event_id: randomUUID(),
      execution_id: d.executionId,
      issue_id: d.issueId,
      project_name: d.projectName,
      event_type: "state_transition",
      payload: { from: d.fromState, to: d.toState, trigger: d.trigger },
      source: "fsm",
    });
    return { type: "audit", success: true };
  }
}
```

**设计决策（Codex Round 1 反馈）**：
- `drain()` 用 sequential for loop（不是 `Promise.all`），保证 audit 写入顺序确定
- 只处理 `audit` directive，其他类型返回 error（可扩展）

#### Task 2.2: applyTransition() 统一入口

**File**: `packages/teamlead/src/applyTransition.ts` (NEW)

**Codex Round 1 核心反馈**：所有状态变更走统一入口，避免 `upsertSession(session, fsm)` 和 `fsm.transition()` 两条路径语义分裂。

```typescript
import type { WorkflowFSM, TransitionContext, TransitionResult } from "flywheel-core";
import type { StateStore } from "./StateStore.js";
import type { DirectiveExecutor } from "./DirectiveExecutor.js";

export interface ApplyTransitionOpts {
  store: StateStore;
  fsm: WorkflowFSM;
  executor?: DirectiveExecutor;
}

/**
 * 统一 transition 入口：validate → guard → persist → audit。
 * 所有状态变更必须走这个函数。
 *
 * 对于首次写入（无 existing session），隐式从 "pending" 开始。
 */
export function applyTransition(
  opts: ApplyTransitionOpts,
  executionId: string,
  targetStatus: string,
  ctx: TransitionContext,
  sessionFields?: Partial<SessionUpsert>,
): TransitionResult {
  const { store, fsm, executor } = opts;

  // Determine current state (missing session = "pending")
  const existing = store.getSession(executionId);
  const currentState = existing?.status ?? "pending";

  // FSM validate + guard + generate directives
  const result = fsm.transition(currentState, targetStatus, ctx);
  if (!result.ok) return result;

  // Persist via persistTransition (INSERT OR UPDATE, bypasses monotonic guard)
  store.persistTransition(executionId, targetStatus, {
    issue_id: ctx.issueId,
    project_name: ctx.projectName,
    ...sessionFields,
  });

  // Drain directives (audit) — best-effort
  if (executor && result.directives.length > 0) {
    executor.drain(result.directives).catch(() => {});
  }

  return result;
}
```

**关键设计**：
- 缺失 session → `currentState = "pending"`，确保首个 event 也被 FSM 验证（Codex Issue #1）
- `upsertSession()` 不再传 FSM 参数 — FSM 验证在 `applyTransition` 层完成
- audit directives 自动 drain（Codex Issue #2: 统一路径）

### Wave 3: StateStore 简化 + forceStatus deprecated

**目标**：StateStore `upsertSession()` 保持不变（不传 FSM）。FSM 验证由 `applyTransition()` 统一负责。标记 `forceStatus()` deprecated。

#### Task 3.1: StateStore — 新增 persistTransition() + patchSessionMetadata()

**设计决策（Codex Round 2 Issue #1）**：`upsertSession()` 的 monotonic guard 和 FSM 的 retry transition（`failed → running`）冲突。FSM 允许但 upsertSession 静默丢弃。

**解决方案**：新增 `persistTransition()` 方法供 `applyTransition()` 使用，绕过 monotonic guard（因为 FSM 已验证）。`upsertSession()` 的 monotonic guard 保留给未走 FSM 的遗留路径。

```typescript
/**
 * Persist a status change that has already been validated by FSM.
 * Bypasses monotonic guard — caller MUST have validated via WorkflowFSM.
 * Uses INSERT OR UPDATE to handle both first-time creation and subsequent transitions.
 */
persistTransition(executionId: string, status: string, fields: Partial<SessionUpsert>): void {
  // INSERT OR UPDATE — handles missing session (first event: pending → running)
  this.db.run(
    `INSERT INTO sessions (execution_id, issue_id, project_name, status, last_activity_at, ...)
     VALUES (?, ?, ?, ?, ?, ...)
     ON CONFLICT(execution_id) DO UPDATE SET
       status = excluded.status,
       last_activity_at = excluded.last_activity_at,
       ...COALESCE for other fields...`,
    [executionId, fields.issue_id, fields.project_name, status, sqliteDatetime(), ...],
  );
  this.save();
}

/**
 * Update non-status metadata fields only. Does NOT touch status.
 * Used after applyTransition() for read-model enrichment (commit_count, lines_added, etc.)
 */
patchSessionMetadata(executionId: string, fields: Partial<Omit<SessionUpsert, "status">>): void {
  // Build SET clause from non-null fields, explicitly excluding status
  // UPDATE sessions SET commit_count=?, lines_added=?, ... WHERE execution_id=?
}
```

**Codex Round 3 修复**：
- `persistTransition()` 用 INSERT OR UPDATE（不是纯 UPDATE），解决首次 event 无 session 行问题
- `patchSessionMetadata()` 只更新非 status 字段，event-route 在 `applyTransition()` 后用它补充 metadata

保留 `TERMINAL_STATUSES` 和 `OUTCOME_STATUSES` 常量不变。

```typescript
export interface ActionDefinition {
  action: string;
  fromStates: string[];
  targetState: string;
}

export const ACTION_DEFINITIONS: ActionDefinition[] = [
  { action: "approve", fromStates: ["awaiting_review"], targetState: "approved" },
  { action: "reject",  fromStates: ["awaiting_review"], targetState: "rejected" },
  { action: "defer",   fromStates: ["awaiting_review", "blocked"], targetState: "deferred" },
  { action: "retry",   fromStates: ["failed", "blocked", "rejected"], targetState: "running" },
  { action: "shelve",  fromStates: ["awaiting_review", "blocked", "failed", "rejected", "deferred"], targetState: "shelved" },
];

// Pure static helper — NO FSM instance needed, directly importable
export function allowedActionsForState(state: string): string[] {
  return ACTION_DEFINITIONS
    .filter(d => d.fromStates.includes(state))
    .map(d => d.action);
}

// Convenience: get target state for an action
export function getActionTarget(action: string): string | undefined {
  return ACTION_DEFINITIONS.find(d => d.action === action)?.targetState;
}
```

#### Task 4.1: actions.ts transitionSession 改用 applyTransition

**File**: `packages/teamlead/src/bridge/actions.ts` (MODIFY)

改动：

2. 不再直接调用 `store.forceStatus()` 或 `store.upsertSession()`
3. 标记 `ACTION_SOURCE_STATUS` deprecated（被 FSM 替代）

```typescript
export function transitionSession(
  store: StateStore,
  action: string,
  executionId: string,
  reason: string | undefined,
  transitionOpts: ApplyTransitionOpts,
): ActionResult {
  const session = store.getSession(executionId);
  if (!session) return { success: false, message: `No session found` };

  const actionDef = ACTION_DEFINITIONS.find(d => d.action === action);
  if (!actionDef) return { success: false, message: `Unknown action: ${action}` };
  const targetStatus = actionDef.targetState;

  const result = applyTransition(
    transitionOpts,
    session.execution_id,
    targetStatus,
    { executionId, issueId: session.issue_id, projectName: session.project_name, trigger: action },
    { issue_id: session.issue_id, project_name: session.project_name,
      last_activity_at: sqliteDatetime(), last_error: reason ?? null },
  );

  if (!result.ok) {
    return { success: false, message: result.error ?? "Transition rejected by FSM" };
  }

  return { success: true, message: `${session.issue_identifier ?? executionId} ${action} successfully` };
}
```
