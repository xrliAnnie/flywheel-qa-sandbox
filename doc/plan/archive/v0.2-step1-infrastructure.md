# Plan: v0.2 Step 1 — Infrastructure Components

## Context

v0.1.1 (PR #4) 完成了 interactive tmux runner。v0.2 架构文档（6 轮 Codex review 通过）定义了三个 Step：

- **Step 1: 基础设施** — 5 个独立组件（本计划）
- Step 2: 集成 — 连接组件成完整单 issue 流程
- Step 3: 并行 — DagDispatcher 重写

Step 1 的 5 个组件**互不依赖**，可并行开发。每个组件遵循 TDD (RED → GREEN → REFACTOR)。

**Architecture doc**: `doc/architecture/v0.2-architecture.md`

### Interface

```typescript
export interface HookEvent {
  token: string;        // callbackToken (UUID)
  sessionId: string;    // Claude's session ID (from query param)
  issueId: string;
  eventType: string;    // "SessionEnd"
  timestamp: number;
}

export class HookCallbackServer extends EventEmitter {
  constructor(port?: number)             // 0 = auto-assign
  async start(): Promise<number>         // returns assigned port
  async stop(): Promise<void>
  getPort(): number
  waitForEvent(token: string, eventType: string, timeoutMs: number): Promise<HookEvent | null>
  // Convenience — matches architecture doc's IHookCallbackServer.waitForCompletion()
  waitForCompletion(callbackToken: string, timeoutMs: number): Promise<HookEvent | null>
  // → delegates to waitForEvent(callbackToken, "SessionEnd", timeoutMs)
}
```

### HTTP Endpoint

```
POST /hook/complete?token=<uuid>&sessionId=<uuid>&issueId=<id>&eventType=SessionEnd
→ 200 "ok"

Missing required params → 400
GET → 405
Unknown path → 404
```

### Implementation

Node `http.createServer` binding to **`127.0.0.1` only** (loopback — no external access). Parse URL query params. Validate `token` is non-empty UUID format before accepting. Emit `"hook"` event. `waitForEvent()` listens with strict token+eventType matching + races against setTimeout.

### Hook Script Update

`scripts/hooks/flywheel-session-end.sh` — **full rewrite** to parse input first, then route to callback or marker:

```bash
#!/bin/bash
set -euo pipefail
# Flywheel SessionEnd hook — notifies orchestrator of session completion.
# Two modes: HTTP callback (v0.2) and marker file (v0.1.1 fallback).

INPUT=$(cat)
SESSION_ID=$(echo "$INPUT" | jq -r '.session_id // empty')

# Validate: must be a UUID format
if [ -z "$SESSION_ID" ] || ! echo "$SESSION_ID" | grep -qE '^[0-9a-f-]{36}$'; then
  exit 0
fi

# Primary: HTTP callback (v0.2) — env vars injected by TmuxRunner
if [ -n "${FLYWHEEL_CALLBACK_PORT:-}" ] && [ -n "${FLYWHEEL_CALLBACK_TOKEN:-}" ]; then
  curl -s -X POST \
    "http://127.0.0.1:${FLYWHEEL_CALLBACK_PORT}/hook/complete?token=${FLYWHEEL_CALLBACK_TOKEN}&sessionId=${SESSION_ID}&issueId=${FLYWHEEL_ISSUE_ID:-unknown}&eventType=SessionEnd" \
    --max-time 5 || true
fi

# Fallback: marker file (v0.1.1 compat) — only if marker dir exists
MARKER_DIR="${FLYWHEEL_MARKER_DIR:-/tmp/flywheel/sessions}"
if [ -d "$MARKER_DIR" ]; then
  echo "$INPUT" > "$MARKER_DIR/$SESSION_ID.done"
fi
```

**Key change vs v0.1.1**: Removed early `exit 0` on missing marker dir — input parsing and HTTP callback run unconditionally. Marker file write is conditional on dir existence.

Env vars `FLYWHEEL_CALLBACK_PORT`, `FLYWHEEL_CALLBACK_TOKEN`, `FLYWHEEL_ISSUE_ID` will be injected by TmuxRunner via `tmux new-window -e` (wired in Step 2).

### Ownership Note (Step 2 integration)

Per architecture doc: **TmuxRunner is the sole completion owner**. DagDispatcher creates HookCallbackServer and passes `port` to TmuxRunner as an env var primitive. TmuxRunner internally races `waitForCompletion(token)` vs `pane_dead` polling. Cross-package wiring (how TmuxRunner in `claude-runner` accesses the server) is a Step 2 decision. Step 1 only builds the component.

### Interface

```typescript
export type WorktreeExecFn = (
  cmd: string, args: string[], cwd: string
) => Promise<{ stdout: string }>;

export interface WorktreeConfig {
  baseDir?: string;           // default: ~/.flywheel/worktrees
  createTimeoutMs?: number;   // default: 120_000
  pruneTimeoutMs?: number;    // default: 10_000
}

export interface WorktreeInfo {
  projectName: string;
  issueId: string;
  worktreePath: string;       // absolute path
  branch: string;             // flywheel-{issueId}
  mainRepoPath: string;
}

export interface ExternalWorktree {
  path: string;
  branch: string | null;
  isDetached: boolean;
  isBare: boolean;
}

export class WorktreeManager {
  constructor(config?: WorktreeConfig, execFn?: WorktreeExecFn)
  async create(opts: { mainRepoPath, projectName, issueId, startPoint? }): Promise<WorktreeInfo>
  async remove(mainRepoPath: string, worktreePath: string): Promise<void>
  async isRegistered(mainRepoPath: string, worktreePath: string): Promise<boolean>
  async list(mainRepoPath: string): Promise<ExternalWorktree[]>
  async pruneOrphans(mainRepoPath: string, projectName: string): Promise<string[]>
}
```

### Implementation Details

- **Dependency injection**: `execFn` for testability (same pattern as `GitResultChecker`)
- **create()**: `mkdir -p parent` → `git worktree add <path> -b flywheel-{issueId} {startPoint}^{commit}` → `git -C <path> config --local push.autoSetupRemote true`
- **remove()**: macOS-safe 3-phase deletion (per architecture doc):
  1. `fs.rename()` to same-filesystem temp dir (avoids EXDEV cross-device error)
  2. `git -C <main> worktree prune` to clean git metadata
  3. `child_process.spawn('/bin/rm', ['-rf', tmpPath], { detached: true })` — background delete, not blocking caller
  - Why not `fs.rm()`: macOS hangs on .app bundle extended attributes. `/bin/rm` is native and more reliable.
  - **Error handling**: `rename()` and `git worktree prune` failures throw (these are critical operations). Only ENOENT (dir already gone → skip to prune) and background `rm` failures are swallowed (logged, not thrown).
- **isRegistered()**: parse `git worktree list --porcelain` output, check if `worktreePath` appears in registered worktrees
- **list()**: parse `git worktree list --porcelain` — blocks separated by blank lines, lines: `worktree <path>`, `HEAD <sha>`, `branch refs/heads/<name>` or `detached`, `bare`
- **pruneOrphans()**: list() → filter flywheel-* branches → check dir exists → remove missing → return pruned paths

**Coexistence**: Separate from existing `GitService.createGitWorktree()` (Cyrus legacy). Different interface, different code path. No shared state. GitService also handles `.worktreeinclude` file copying and setup scripts — these are Cyrus-specific concerns. WorktreeManager does NOT replicate them; if needed for v0.2, they will be factored separately in Step 2 integration.

### Interface

```typescript
export interface SkillContext {
  // Required — always available from Linear issue + project config
  issueId: string;
  issueTitle: string;
  issueDescription: string;
  projectName: string;
  // Optional — derived from project config or package.json (Step 2 wiring)
  testCommand?: string;      // default: "pnpm test"
  lintCommand?: string;      // default: "pnpm lint"
  buildCommand?: string;     // default: "pnpm build"
  testFramework?: string;    // default: "vitest"
}

export class SkillInjector {
  async inject(projectRoot: string, ctx: SkillContext): Promise<void>
}
```

**Data source note**: `issueId/Title/Description` come from Linear API (already available). `projectName` comes from `FlywheelConfig`. The optional command fields (`testCommand` etc.) do NOT currently exist in `FlywheelConfig` (`packages/config/src/types.ts`).

**Step 1 template strategy (option A — minimal)**: Templates use only currently-available fields. Optional fields (`testCommand`, `lintCommand`, `buildCommand`, `testFramework`) use hardcoded defaults when omitted (e.g., `"pnpm test"`, `"vitest"`). Templates are written to be useful with defaults alone — no placeholder will render as empty string or `{{...}}`. Step 2 wiring will extend `FlywheelConfig` or auto-detect from `package.json` scripts to supply project-specific values.

### The 5 Skills (from exploration doc)

| # | Name | Purpose |
|---|------|---------|
| 1 | `flywheel-context` | Project structure, tech stack, conventions |
| 2 | `linear-issue-context` | Current issue details (dynamic per session) |
| 3 | `flywheel-git-workflow` | Git + PR procedures (REQUIRED for result detection) |
| 4 | `flywheel-escalation` | Escalation conditions + procedures |
| 5 | `flywheel-tdd` | Test-first workflow enforcement |

### Implementation

- Templates in `skill-templates/*.ts` — each exports `const template = \`...\`` string
- `index.ts` barrel re-exports as `Record<SkillName, string>`
- `inject()`: for each skill → `mkdir -p .claude/skills/{name}/` → `writeFile(SKILL.md, render(template, ctx))`
- `render()`: simple `{{placeholder}}` → `String.replace()` (no deps). Missing optional fields → use defaults (e.g., `{{testCommand}}` → `"pnpm test"`)
- Template content sourced from `doc/exploration/new/v0.2-skill-system.md` sections 2.1-2.5
- **Git exclude injection**: After writing skills, resolve the real git dir via `git -C <projectRoot> rev-parse --git-dir`, then append `.claude/skills/` to `<gitDir>/info/exclude` (if not already present). This handles both normal repos (`.git` is a directory) and worktrees (`.git` is a file pointing to the main repo's `.git/worktrees/xxx`). Uses git's local exclude (not `.gitignore`) to prevent session-level skill files from being committed via `git add .`.

├── flywheel-git-workflow/SKILL.md

## Implementation Notes

1. **ESM imports**: use `.js` extension (`import { X } from "./X.js"`)
2. **No logging in Semaphore/ProjectLock**: generic utilities, no side effects
3. **WorktreeManager uses `createLogger`**: for git operation logging
4. **HookCallbackServer port 0**: auto-assign in all tests; **bind 127.0.0.1 only**
5. **GitService coexistence**: do NOT touch `GitService.ts` — separate code path. `.worktreeinclude` and setup scripts are Cyrus-specific, not replicated in WorktreeManager. Step 2 determines if they're needed.
6. **vitest config**: edge-worker has `afterEach(() => vi.clearAllMocks())` in setup
7. **WorktreeManager remove()**: 3-phase macOS-safe deletion (rename → prune → background rm). `rename()`/`prune` failures throw; only ENOENT and background rm are best-effort.
8. **SkillInjector git exclude**: Resolve `git rev-parse --git-dir` first (handles worktrees where `.git` is a file), then write to `<gitDir>/info/exclude`. Prevents git pollution without modifying version-controlled files.
9. **SkillContext defaults**: Optional fields use hardcoded defaults (`pnpm test`, `pnpm lint`, `pnpm build`, `vitest`). No placeholder renders as empty string. Step 2 wires project-specific values.
10. **Hook script URL**: Use `127.0.0.1` (not `localhost`) to match server's loopback binding.
