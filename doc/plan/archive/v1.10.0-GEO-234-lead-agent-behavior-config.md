## Step 1: 创建 Agent 文件

**源码位置**: `packages/teamlead/agents/product-lead.md`（Flywheel repo 内，唯一 source of truth）
**运行时位置**: `~/.claude/agents/product-lead.md`（supervisor 自动 symlink 维护，不手动创建）

**为什么 repo 内存源码、全局放 symlink**:
- Checked in，版本控制，code review 覆盖
- Supervisor 启动时自动同步，无手动部署风险
- Lead 运行在隔离工作区（`~/.flywheel/lead-workspace/`），全局 agent 路径不依赖产品 repo cwd

### 1.1 Frontmatter

```yaml
---
name: product-lead
description: Flywheel Product Department Lead — manages AI runners, monitors execution, communicates with CEO via Discord
model: opus
disallowedTools: Write, Edit, MultiEdit, Agent, NotebookEdit
permissionMode: bypassPermissions
---
```

**设计决策**:
- 使用 `disallowedTools` 替代 `tools` allowlist — 显式禁止所有写代码相关工具（Write, Edit, MultiEdit, Agent, NotebookEdit），同时保留 Discord MCP 工具（reply, react, edit_message, fetch_messages）和 Read/Bash/Grep/Glob。`tools` allowlist 会切断所有未列出的工具（包括 MCP），这对 Lead 是致命的。参考: [Claude Code Sub-agents Docs](https://code.claude.com/docs/en/sub-agents)
- `model: opus` — 需要深度推理（事件分析、escalation 判断）
- `permissionMode: bypassPermissions` — 无人值守运行，替代 CLI 的 `--dangerously-skip-permissions`

**写保护设计 — 双层防御**:
1. **工具层**: `disallowedTools` 禁用内建编辑工具（Write/Edit/MultiEdit/NotebookEdit）
2. **工作区隔离层**: Lead 运行在 `~/.flywheel/lead-workspace/`（非 repo 目录），不是产品代码库内。Bash 仍可用（curl + flywheel-comm），但 Lead 的工作目录不包含代码文件，显著降低误操作修改代码库的风险。
3. **Prompt 约束层**: Agent prompt 明确写"绝对不做: 写代码、修改代码库"。

**安全边界说明**: 工作区隔离降低误操作概率，但不构成硬安全边界 — Lead 理论上仍可通过绝对路径 + Bash 修改文件。真正的 filesystem 级隔离（容器、独立用户、只读挂载）超出本 issue scope，可作为未来安全加固方向。当前三层防御（工具限制 + 工作区隔离 + prompt 约束）足以满足 Phase 1 需求。

### 1.3 新增 Runner 通信 Section

```markdown
## Runner 通信 — flywheel-comm

你的 Runner（AI 工程师）通过 flywheel-comm CLI 与你通信。通信通过本地 SQLite 数据库进行。

### 命令参考

检查待回答问题:
\`\`\`bash
node $FLYWHEEL_COMM_CLI pending --lead $LEAD_ID --project $PROJECT_NAME
\`\`\`

回答 Runner 问题（获取 question-id 后）:
\`\`\`bash
node $FLYWHEEL_COMM_CLI respond --lead $LEAD_ID <question-id> "你的回答"
\`\`\`

发送主动指令给 Runner:
\`\`\`bash
node $FLYWHEEL_COMM_CLI send --from $LEAD_ID --to <exec-id> "指令内容"
\`\`\`

查看活跃 Runner sessions:
\`\`\`bash
node $FLYWHEEL_COMM_CLI sessions --project $PROJECT_NAME --active
\`\`\`

抓取 Runner 当前 tmux 输出:
\`\`\`bash
node $FLYWHEEL_COMM_CLI capture --exec-id <exec-id>
\`\`\`

### 检查时机

你必须主动检查 Runner 通信。没有人会提醒你 — 这是你的职责。

1. **处理完每个 Discord 事件后** → 检查是否有 pending 问题
2. **空闲等待时** → 定期检查 pending（至少每 5 分钟一次）
3. **CEO 提到某个 issue 时** → 先查看该 issue 的 session 状态 + capture
4. **收到 session_completed/session_failed 事件后** → 检查是否有未处理的问题

### Escalation 策略

**当前阶段（Phase 1）: 所有 Runner 问题都上报 CEO。**

当你发现 Runner 有 pending 问题时：
1. 读取问题内容
2. 在 Chat Channel 通知 CEO，附上问题摘要和上下文
3. 等 CEO 回复后，用 `respond` 命令将答案传回 Runner
4. **不要自行回答** — 即使你觉得自己知道答案

这个策略会逐步演进：未来你将被授权自行回答部分技术问题。
```

### 2.4 Agent 文件自动同步 + Fail Fast

Repo 内 `packages/teamlead/agents/product-lead.md` 是唯一 source of truth。Supervisor 启动时自动 symlink 到全局位置：

```bash
AGENT_SOURCE="${SCRIPT_DIR}/../agents/product-lead.md"
AGENT_TARGET="${HOME}/.claude/agents/product-lead.md"
mkdir -p "${HOME}/.claude/agents"

if [ -f "$AGENT_SOURCE" ]; then
  # 自动同步：symlink repo 源码到全局位置
  ln -sf "$(cd "$(dirname "$AGENT_SOURCE")" && pwd)/$(basename "$AGENT_SOURCE")" "$AGENT_TARGET"
  echo "[lead] Agent file synced: ${AGENT_TARGET} -> ${AGENT_SOURCE}"
else
  echo "[lead] ERROR: Agent source not found at ${AGENT_SOURCE}"
  echo "[lead] Ensure packages/teamlead/agents/product-lead.md exists."
  exit 1
fi
```

**单一 source of truth**: repo 内源码通过 symlink 映射到全局位置，不存在版本漂移风险。更新 agent 行为只需修改 repo 内文件 + `git pull`。

使用 bash 数组组装 `claude` 参数：

```bash
CLAUDE_ARGS=(--agent product-lead --channels "plugin:discord@claude-plugins-official")
```

**注意**: 不提供 `--dangerously-skip-permissions` fallback — `permissionMode: bypassPermissions` 在 agent frontmatter 中设置。

### 2.4 FLYWHEEL_COMM_CLI 缺失时的降级

当前脚本在 `flywheel-comm/dist/index.js` 不存在时打印 warning 继续启动。GEO-234 的 agent prompt 依赖 `$FLYWHEEL_COMM_CLI`。

**策略**: Degraded mode — Lead 仍能处理 Discord 事件和 CEO 指令，只是 Runner 通信不可用。

当前脚本的 dist 探测使用 `$(cd .../flywheel-comm/dist && pwd)`，在 `set -euo pipefail` 下如果目录不存在会直接退出，无法进入 degraded mode。需要改为 non-failing probe：

```bash
# 替换现有的 cd-based 探测（会被 set -e 中断）
COMM_DIST_DIR="${SCRIPT_DIR}/../../flywheel-comm/dist"
if [ -f "${COMM_DIST_DIR}/index.js" ]; then
  export FLYWHEEL_COMM_CLI="$(cd "$COMM_DIST_DIR" && pwd)/index.js"
  echo "[lead] Comm CLI: ${FLYWHEEL_COMM_CLI}"
else
  echo "[lead] WARNING: flywheel-comm not built. Runner communication disabled."
  echo "[lead] Lead will still handle Discord events and CEO commands."
  echo "[lead] Run 'pnpm -r build' to enable Runner communication."
fi
```

**关键改动**: 先用 `[ -f ]` 检查文件是否存在（non-failing），只在存在时才 `cd && pwd` 取绝对路径。这样 `set -e` 不会中断脚本执行。

**注意**: 后续引用使用 `${FLYWHEEL_COMM_CLI:-}` 而非 `$FLYWHEEL_COMM_CLI`，兼容 `set -u`。

Agent prompt 中也添加 guard 说明：
> "如果 `$FLYWHEEL_COMM_CLI` 为空或命令执行失败，说明 flywheel-comm 未部署。此时你仍可处理 Discord 事件和 CEO 指令，但无法与 Runner 通信。向 CEO 报告此情况。"
