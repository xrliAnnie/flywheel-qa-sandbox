# Plan: Lead Workspace 指向 Per-Lead 子目录

**Version**: v1.17.0
**Issue**: GEO-286
**Date**: 2026-03-29 (v2 — rewrite for per-Lead subdirectories)
**Source**: `doc/exploration/new/GEO-286-lead-workspace-project-dir.md`

## Summary

将 Lead 的工作目录从空的隔离目录改为项目中各自对应的子目录：
- Peter (product-lead) → `GeoForge3D/product/`
- Oliver (ops-lead) → `GeoForge3D/operations/`
- Simba (cos-lead) → `GeoForge3D/`（根目录）

Claude Code 向上遍历加载 CLAUDE.md，子目录运行仍获得项目级 context。Agent file 统一留在 `GeoForge3D/.lead/`（不移动）。

## Motivation

1. **Lead 对项目零 context** — 当前运行在空目录，无法读 CLAUDE.md
2. **Per-Lead 职责对齐** — Peter 管产品代码、Oliver 管运营、Simba 需全局视角
3. **Annie 明确要求** per-Lead 子目录，不要所有人都在根目录

## Research 结论

| 问题 | 结论 |
|------|------|
| Claude Code 在子目录能加载根 CLAUDE.md 吗？ | ✅ 向上遍历加载所有 ancestor CLAUDE.md |
| `.claude/` settings 在子目录能找到吗？ | ✅ 从 git root 查找 |
| Bridge API / Comm DB 受 cwd 影响吗？ | ❌ 全部绝对路径 |
| Agent source 查找受 cwd 影响吗？ | ❌ 用 `${PROJECT_DIR}` 而非 cwd |

## Design

### 核心改动

`claude-lead.sh` 新增 `--subdir` named flag：

```
Usage: ./scripts/claude-lead.sh <lead-id> <project-dir> [project-name] [--subdir <dir>]
```

- `--subdir <dir>`: Lead 的工作子目录（相对于 project-dir）。必须是已存在的目录，不允许 `..` 路径遍历。
- 省略 `--subdir` 时在 project-dir 根目录运行（显式空字符串 `--subdir ''` 会被拒绝）。
- 环境变量 `LEAD_WORKSPACE` 是最高优先级逃生舱：如果已设定，跳过 `--subdir` 的目录合法性校验（存在性、边界检查），直接使用该值作为工作目录。注意 CLI 语法解析仍会执行，`--subdir` 缺参数等语法错误不受 `LEAD_WORKSPACE` 影响。

**为什么用 `--subdir` flag 而不是第 4 个 positional 参数**（Codex review Round 1 feedback）：
第 3 个参数 `project-name` 已经是可选的。如果再加第 4 个可选 positional，`./claude-lead.sh product-lead /path product` 中的 `product` 会被误解为 project-name 还是 lead-subdir，CLI 有歧义。Named flag 消除歧义。

### Workspace 解析逻辑

```bash
# Parse --subdir flag (can appear anywhere after $2)
# If $3 doesn't start with --, treat as project-name
# All --flags are parsed as named flags

# Workspace resolution — LEAD_WORKSPACE env var has highest priority:
# (CLI syntax parsing already ran above — --subdir arg validated)
if [ -n "${LEAD_WORKSPACE:-}" ]; then
  # Escape hatch: skip --subdir path validation, use directly
elif [ -n "$LEAD_SUBDIR" ]; then
  # Reject .. traversal
  # Resolve to physical absolute path via cd + pwd -P (follows symlinks)
  # Verify resolved physical path is within PROJECT_DIR (also physical)
  # Fail-fast if directory doesn't exist
  LEAD_WORKSPACE="${RESOLVED}"
else
  LEAD_WORKSPACE="${PROJECT_DIR}"
fi
```

Per-Lead 调用示例：
```bash
# Peter → GeoForge3D/product/
./scripts/claude-lead.sh product-lead /path/to/GeoForge3D geoforge3d --subdir product

# Oliver → GeoForge3D/operations/
./scripts/claude-lead.sh ops-lead /path/to/GeoForge3D geoforge3d --subdir operations

# Simba → GeoForge3D/（根目录，不传 --subdir）
./scripts/claude-lead.sh cos-lead /path/to/GeoForge3D geoforge3d

# Flag order is flexible — these are equivalent:
./scripts/claude-lead.sh product-lead /path --subdir product geoforge3d
./scripts/claude-lead.sh product-lead /path geoforge3d --subdir product
```

### Path validation（Codex review Round 1 feedback）

`--subdir` 参数经过严格验证：
1. 拒绝包含 `..` 的路径
2. 要求目录必须已存在（不再 `mkdir -p` 静默创建）
3. 解析为绝对路径后验证仍在 `PROJECT_DIR` 内
4. 任何验证失败都 fail-fast

### Agent source 查找（简化）

去掉兼容层，只保留两级：

```bash
# Priority: 1) AGENT_SOURCE env var, 2) PROJECT_DIR/.lead/<lead-id>/agent.md, 3) fail-fast
if [ -n "${AGENT_SOURCE:-}" ]; then
  : # explicit override
elif [ -f "${PROJECT_DIR}/.lead/${LEAD_ID}/agent.md" ]; then
  AGENT_SOURCE="${PROJECT_DIR}/.lead/${LEAD_ID}/agent.md"
fi
```

**删除**: `${LEAD_WORKSPACE}/agent.md` 兼容层（Annie 明确不需要向后兼容）。

### Agent file 位置不变

Agent file 统一在 `${PROJECT_DIR}/.lead/`，不跟 workspace 走：
- `GeoForge3D/.lead/product-lead/agent.md`
- `GeoForge3D/.lead/ops-lead/agent.md`
- `GeoForge3D/.lead/cos-lead/agent.md`

理由：agent file 是身份配置，不是工作内容。分散增加维护成本。

### 安全性

与 v1 plan 相同 — 有意识的安全-功能取舍。Lead 在项目子目录运行，`disallowedTools` 禁止 Write/Edit，agent.md 禁止代码操作。

### 不改的东西

- GeoForge3D 目录结构 — 已有 `product/` 和 `operations/`
- Agent.md 文件内容和位置 — 不需要改
- Bridge API — 不涉及
- Discord plugin — 不涉及

## Implementation Steps

### Step 1: 修改 claude-lead.sh

**文件**: `packages/teamlead/scripts/claude-lead.sh`

1. **Argument parsing 重构** — 用 `shift 2` + `while` loop 解析 `$3+`：
   - 非 `--` 开头的第一个 arg → `PROJECT_NAME`
   - `--subdir <dir>` → `LEAD_SUBDIR`
   - 未知 flag/多余 positional → `[lead] ERROR` + exit 1
   - `--subdir` 缺参数 → 显式检查 `$#` 和 `$2` 形态，输出 `[lead] ERROR`

2. **Workspace 解析（在 agent auto-sync 之前）** — 带 physical path 校验：
   ```bash
   if [ -n "${LEAD_WORKSPACE:-}" ]; then
     # Escape hatch: highest priority, skip --subdir path validation
   elif [ -n "$LEAD_SUBDIR" ]; then
     # 1. Reject paths containing ..
     # 2. cd + pwd -P → physical absolute path (follows symlinks)
     # 3. Verify RESOLVED starts with PROJECT_DIR/ or equals PROJECT_DIR
     # 4. Directory must already exist — no mkdir -p
     LEAD_WORKSPACE="${RESOLVED}"
   else
     LEAD_WORKSPACE="${PROJECT_DIR}"
   fi
   ```
   `PROJECT_DIR` 自身也用 `pwd -P` 解析，保证两边比较时基准一致。

3. **Agent source 简化 — 去掉兼容层**:
   ```bash
   if [ -n "${AGENT_SOURCE:-}" ]; then
     : # explicit override
   elif [ -f "${PROJECT_DIR}/.lead/${LEAD_ID}/agent.md" ]; then
     AGENT_SOURCE="${PROJECT_DIR}/.lead/${LEAD_ID}/agent.md"
   fi
   ```

4. **删除旧 workspace 块**（原 "Workspace isolation" section）。

5. **更新文件头注释和示例** — 使用 `--subdir` flag。

### Step 2: Verification

1. 启动 Peter: `./scripts/claude-lead.sh product-lead /path/to/GeoForge3D geoforge3d --subdir product`
   - 日志应显示 `Working directory: .../GeoForge3D/product`
   - Agent file 从 `GeoForge3D/.lead/product-lead/agent.md` 正确复制
2. 启动 Oliver: `./scripts/claude-lead.sh ops-lead /path/to/GeoForge3D geoforge3d --subdir operations`
3. 启动 Simba: `./scripts/claude-lead.sh cos-lead /path/to/GeoForge3D geoforge3d`（无 `--subdir` → 根目录）
4. 确认 Lead 的 Discord 通信正常
5. Fresh session — 删除旧 session ID 文件

## Test Plan

- [ ] `--subdir product` 时 workspace 为 `PROJECT_DIR/product`（physical path）
- [ ] 不传 `--subdir` 时 workspace 为 `PROJECT_DIR`（Simba 场景）
- [ ] `--subdir ../escape` 被拒绝（path traversal guard）
- [ ] `--subdir nonexistent` 被拒绝（directory must exist）
- [ ] Symlink subdir 解析为 physical path，如果 physical path 在 PROJECT_DIR 外则被拒绝
- [ ] `LEAD_WORKSPACE` 环境变量覆盖有效（最高优先级，跳过 --subdir 路径校验）
- [ ] `LEAD_WORKSPACE` + malformed `--subdir`（缺参）仍报 CLI 语法错误（LEAD_WORKSPACE 不绕过语法解析）
- [ ] `AGENT_SOURCE` 环境变量覆盖仍有效
- [ ] Agent source 从 `${PROJECT_DIR}/.lead/<lead-id>/agent.md` 正确找到
- [ ] Agent source 不存在时 fail-fast 且错误信息清楚（`[lead] ERROR` 格式）
- [ ] `--subdir` 缺参数时输出 `[lead] ERROR` 格式错误信息
- [ ] `--subdir ''`（空字符串）被拒绝，不会静默回退到根目录
- [ ] 不设任何环境变量时脚本不 crash（set -u 安全）
- [ ] Lead 在子目录能读到项目根 CLAUDE.md
- [ ] Flag 顺序灵活：`--subdir product geoforge3d` 和 `geoforge3d --subdir product` 等价

## Risks

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|------------|
| Lead 通过 Bash 修改项目文件 | Low | Medium | disallowedTools + agent 行为规则 |
| 子目录不存在 | Low | High (无法启动) | Fail-fast + 明确错误信息 |
| Symlink 指向项目外 | Low | Medium | `pwd -P` physical path + prefix check |
| CLAUDE.md 不加载 | None | N/A | Research 确认向上遍历 |
| 旧 session resume 后 context 不更新 | Medium | Low | Fresh session rollout — 删旧 session ID 文件 |

## Rollback

`LEAD_WORKSPACE` 环境变量覆盖，或 revert commit。
