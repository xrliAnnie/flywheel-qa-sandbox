# Plan: Forum Tag Per-Lead 配置

**Version**: v1.10.0
**Issue**: GEO-253
**Date**: 2026-03-24
**Source**: Linear issue GEO-253 (Forum Tag 过滤器丢失 — STATUS_TAG_MAP 未配置)
**Status**: codex-approved

## Problem

Bridge 的 `ForumTagUpdater` 和 `ForumPostCreator` 依赖全局 `STATUS_TAG_MAP` 环境变量来映射 session status → Discord Forum tag ID。当前该变量未配置，导致所有 Forum tag 更新被跳过：

```
[Bridge] statusTagMap is empty — ForumTagUpdater will skip all tag updates.
```

GEO-246 后系统有两个 Forum channel（Product + Operations），Discord tag ID 是 **per-channel** 的（同名 tag 在不同 Forum 有不同 ID）。全局 `STATUS_TAG_MAP` 无法同时服务两个 Forum。

### 影响

Forum 帖子能创建，但没有状态标签 → CEO 无法通过标签过滤查看不同状态的任务。

## Solution: Per-Lead `statusTagMap` in `LeadConfig`

在 `projects.json` 的 `LeadConfig` 中添加 `statusTagMap` 字段。每个 Lead 的 Forum channel 配置自己的 status → tag ID 映射。

### 为什么不继续用全局 env var

| 方案 | 问题 |
|------|------|
| 单一 `STATUS_TAG_MAP` | tag ID per-channel，无法一个 map 覆盖两个 Forum |
| 双 env var（`STATUS_TAG_MAP_PRODUCT` / `STATUS_TAG_MAP_OPS`） | 命名耦合到具体 lead，扩展性差 |
| **Per-lead `statusTagMap`**（选此） | 与 GEO-246 multi-lead 架构一致，`projects.json` 已有 per-lead config |

### 与 GEO-252 关系

GEO-252（per-lead `botToken`）和 GEO-253 修改同一批调用点，但逻辑独立。GEO-253 不依赖 GEO-252，二者可独立实施、独立合并。

## Changes

### 1. `LeadConfig` 添加 `statusTagMap` 字段

**File**: `packages/teamlead/src/ProjectConfig.ts`

```typescript
export interface LeadConfig {
  agentId: string;
  forumChannel: string;
  chatChannel: string;
  match: { labels: string[] };
  runtime?: "openclaw" | "claude-discord";
  controlChannel?: string;
  /** Status → Discord Forum tag ID mapping for this lead's forum channel. */
  statusTagMap?: Record<string, string[]>;
}
```

**验证逻辑**（`loadProjects()`）：
- 如果 `statusTagMap` 未定义 → 不报错，fallback 到全局 `STATUS_TAG_MAP`（即该 lead 的 tag 行为由全局配置决定，不是禁用 tag）
- 如果 `statusTagMap` 定义了但为空 `{}` → 报错（空 map 是配置错误；要使用全局 fallback 就省略此字段，要自定义就填完整）
- 如果 `statusTagMap` 定义了 → 校验为 non-null、non-array object（与 `parseStatusTagMap()` 一致），每个 value 为 non-empty string[]，每个 tag ID 为 non-empty string

**语义规则**：`statusTagMap` 一旦定义，即为该 lead 的完整 tag 配置，完全替代全局 fallback。

**不做严格 key 集合校验**：不要求 `statusTagMap` 包含所有可能的 status key。原因：status 集合可能随功能演进变化（如新增 action），且 `ForumTagUpdater` 已对 unmapped status 返回 `"skipped"`，不会报错。未映射的 status 只是不会打 tag，不会影响功能。

**已知生产 status 列表**（供运维参考，非强制校验）：
`running`, `awaiting_review`, `approved`, `completed`, `failed`, `blocked`, `rejected`, `deferred`, `shelved`, `terminated`

### 4. 全局 `STATUS_TAG_MAP` 保留为 fallback

`config.ts` 中的 `parseStatusTagMap(process.env.STATUS_TAG_MAP)` 不删除。`ForumTagUpdater` / `ForumPostCreator` 构造函数仍接收全局 map。Per-call override 优先于构造器 map。

降级路径（调用方 → callee 内部）：
```
调用方: lead.statusTagMap → 传入 ctx.statusTagMap
callee: ctx.statusTagMap ?? this.statusTagMap (constructor/global) ?? {}
```

### 5. `plugin.ts` 启动告警优化

**File**: `packages/teamlead/src/bridge/plugin.ts` (~line 691-695)

当前：全局 `statusTagMap` 为空时打印 `Set STATUS_TAG_MAP env var to enable`。GEO-253 后 per-lead tag map 才是主配置源，全局为空是正常状态。

改为 3-state 诊断逻辑：

- **None configured**（全局空 + 无任何 lead 有 `statusTagMap`）→ `console.warn`，提示所有 tag 更新将被跳过
- **Partial**（全局空 + 部分 lead 有 `statusTagMap`）→ `console.warn`，列出缺失配置的 lead `agentId`，提示这些 lead 的 tag 更新将被跳过
- **All configured**（全局空 + 所有 lead 都有 `statusTagMap`）→ `console.log`，info 级别确认

```typescript
const allLeads = projects.flatMap(p => p.leads);
const leadsWithMap = allLeads.filter(l => l.statusTagMap != null);
const leadsWithoutMap = allLeads.filter(l => l.statusTagMap == null);

if (Object.keys(statusTagMap).length === 0 && leadsWithMap.length === 0) {
  console.warn(
    "[Bridge] No statusTagMap configured (global or per-lead) — ForumTagUpdater will skip all tag updates.",
  );
} else if (Object.keys(statusTagMap).length === 0 && leadsWithoutMap.length > 0) {
  console.warn(
    `[Bridge] Global statusTagMap is empty. ${leadsWithMap.length}/${allLeads.length} leads have per-lead statusTagMap. ` +
    `Leads missing config: ${leadsWithoutMap.map(l => l.agentId).join(", ")} — these will skip tag updates.`,
  );
} else if (Object.keys(statusTagMap).length === 0) {
  console.log(
    "[Bridge] Global statusTagMap is empty; all leads have per-lead statusTagMap configured.",
  );
}
```

当全局为空时，三种状态：全部未配、部分配置（列出缺失的 lead）、全部已配。

当全局非空时，额外检查多 Forum 场景的 fallback 风险：

```typescript
if (Object.keys(statusTagMap).length > 0 && leadsWithoutMap.length > 0) {
  // 多 Forum 部署下，全局 map 的 tag ID 只属于一个 Forum channel。
  // 未配置 per-lead map 的 lead 会 fallback 到全局，可能使用错误的 tag ID。
  const uniqueForums = new Set(allLeads.map(l => l.forumChannel));
  if (uniqueForums.size > 1) {
    console.warn(
      `[Bridge] Multiple forum channels detected but ${leadsWithoutMap.length} lead(s) lack per-lead statusTagMap ` +
      `and will fallback to global STATUS_TAG_MAP (which may contain wrong tag IDs): ` +
      `${leadsWithoutMap.map(l => l.agentId).join(", ")}`,
    );
  }
}
```

### 6. 运维：获取 Discord tag ID 并更新 `projects.json`

**操作步骤**（不需要代码改动）：

1. 调用 Discord API 获取每个 Forum channel 的 tag：
   ```bash
   # Product Forum
   curl -H "Authorization: Bot $PETER_BOT_TOKEN" \
     https://discord.com/api/v10/channels/1485787822119194755 | jq '.available_tags'

   # Operations Forum
   curl -H "Authorization: Bot $OLIVER_BOT_TOKEN" \
     https://discord.com/api/v10/channels/1485789340989915266 | jq '.available_tags'
   ```

2. 如果 tag 不存在，通过 Discord API 或 Server Settings 创建。以下是当前可触发 tag 更新的状态（参考上方 "已知生产 status 列表"）。建议至少覆盖高频状态，低频状态（如 `deferred`, `shelved`）可按需添加：
   - In Progress (`running`)
   - Awaiting Review (`awaiting_review`)
   - Approved (`approved`)
   - Completed (`completed`) — 可复用 Approved 的 tag ID
   - Failed (`failed`)
   - Blocked (`blocked`)
   - Terminated (`terminated`)

   注意：未映射的 status 不会报错，只会跳过 tag 更新（`ForumTagUpdater` 返回 `"skipped"`）。

3. 更新 `~/.flywheel/projects.json`（`statusTagMap` 定义后即为该 lead 的完整 tag 配置，全局 fallback 不再生效。未列出的 status 会被静默跳过）：
   ```json
   {
     "agentId": "product-lead",
     "forumChannel": "1485787822119194755",
     "chatChannel": "1485787822894878955",
     "match": { "labels": ["Product"] },
     "runtime": "claude-discord",
     "controlChannel": "1486024717235261462",
     "statusTagMap": {
       "running": ["<tag-id>"],
       "awaiting_review": ["<tag-id>"],
       "approved": ["<tag-id>"],
       "completed": ["<tag-id>"],
       "failed": ["<tag-id>"],
       "blocked": ["<tag-id>"],
       "terminated": ["<tag-id>"]
     }
   }
   ```

## Risks

- **Low**: 完全 backward compatible — 旧配置无 `statusTagMap` 仍 fallback 到全局 env var（或空 map）
- **Low**: 接口变更是 additive only（新增可选字段）
- **Low**: `actions.ts` 中 `lead` 已通过 `registry.resolveWithLead()` 获取（已确认）
- **Operational**: GEO-253 与 GEO-252 代码独立，但 **生产部署** 假设当前 Bridge 的全局 `DISCORD_BOT_TOKEN` 对所有目标 Forum 有编辑权限。如果 Peter/Oliver 的 Forum 只允许各自 bot 编辑（bot 权限按 channel 隔离），则 GEO-252（per-lead `botToken`）需要先部署或同时部署，否则 tag 更新仍会 403。建议在 runbook 中检查：GEO-253 rollout 前，确认 Bridge bot 有 `MANAGE_THREADS` 权限 on both Forums，或先合并 GEO-252。
