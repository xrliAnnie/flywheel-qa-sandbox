## Root Causes

### RC-1: Agent 跳过 interactive brainstorm
- **Category**: Process Gap
- **Chain**: Agent 收到 issue → 直接实现 → 方向错误 → PR 废掉
  - Why? Agent prompt 没有硬性要求 brainstorm
  - Why? Orchestrator 只说 "follow /spin" 没有具体 gate
  - Why? Memory 写了但执行文件没改
  - **Root cause**: 改 memory ≠ 改行为。执行文件（orchestrator.md）才改行为。
- **Evidence**: FLY-51 PR #116 closed (wrong direction)

### RC-2: Team lead 在 prompt 里放错误实现细节
- **Category**: Process Gap
- **Chain**: Team lead 写 "Root Cause: remain-on-exit" → Agent 照做 → 方向错
  - Why? Team lead 自己没理解清楚
  - Why? Team lead 没有先跟 Annie 确认
  - **Root cause**: Team lead 不懂就别装懂。不应该在 prompt 里加实现方案。
- **Evidence**: FLY-51 agent prompt 包含错误假设

### RC-3: QA 做代码分析不做 E2E
- **Category**: Process Gap
- **Chain**: QA 跑 pnpm test → 报 PASS → Annie 发现功能不对
  - Why? QA prompt 说 "analyze the fix"
  - Why? QA 协议没有强制 E2E
  - **Root cause**: QA 必须从 product spec 推导测试，验证用户可见行为，不是内部状态。
- **Evidence**: 多轮 QA 报 PASS 但 Annie 发现问题

### RC-4: QA 报 negative assertion 为 PASS
- **Category**: Missing Validation
- **Chain**: "Chat 没有 ClaudeBot 消息" → QA 报 PASS → 但 Peter 也没说话
  - Why? QA 验证 "错误的事情没发生" 而不是 "正确的事情发生了"
  - **Root cause**: PASS 必须是 positive assertion，不是 negative assertion。
- **Evidence**: QA V3 报 PASS 但 Chat 完全没有通知

### RC-5: FLY-47 架构方向反复 (打地鼠)
- **Category**: Architecture Gap
- **Chain**: Discord control channel → ClaudeBot token → Lead daemon → CommDB → MCP poller（转了 5 次）
  - Why? 每次遇到问题就想一个新方案
  - Why? 没有从第一性原理出发
  - **Root cause**: 应该先问 "Agent Team 怎么做的？" 然后照搬，不是自己发明方案。
- **Evidence**: FLY-47 PR #119 有 20+ commits，多次方向转换

### RC-6: Bridge 用 Discord 做内部通信
- **Category**: Architecture Gap
- **Chain**: Bridge → Discord control channel → Lead（绕路）
  - Why? 历史原因（OpenClaw 时代的设计）
  - Why? 没有人质疑 "为什么内部通信要走外部服务？"
  - **Root cause**: Lead 是 Claude Code session，CommDB 就是它的 inbox，不需要 Discord 中转。
- **Evidence**: allowBots 配置问题、self-filtering bug

### RC-7: QA 没有启动 Lead 就测 E2E
- **Category**: Missing Validation
- **Chain**: QA 测 "Bridge → Lead" 但 Lead 没在跑 → 链路断一半 → 报 PASS
  - Why? QA 没有前置条件检查
  - **Root cause**: E2E 测试必须验证前置条件（Lead 在线），否则不能开始。
- **Evidence**: 多轮 QA 报 PASS 但 Lead 没收到任何消息

### RC-8: Lead 声称做了没做的事
- **Category**: Missing Validation
- **Chain**: Peter 说 "PR 已合并" / "tmux 已关闭" 但实际没做
  - Why? Bridge 通知内容不够精确，Lead AI 自己推断
  - Why? Lead rules 没有 "不能声称未验证的事"
  - **Root cause**: Lead 不应该声称完成无法验证的操作。通知内容必须精确到操作可验证。
- **Evidence**: BUG-7 + BUG-8

## Lessons Learned

1. **改 memory ≠ 改行为。** Memory 是笔记，执行文件（orchestrator.md, qa-parallel-executor.md）才改行为。每次 Annie 给反馈，必须改执行文件。

2. **从第一性原理出发，不要打地鼠。** 遇到通信问题时应该先问 "Agent Team 怎么做的？CommDB 就是 mailbox"，而不是一个一个方案试。FLY-47 如果一开始就这样想，5 次架构转向可以避免。

3. **技术正确 ≠ 产品正确。** QA 报 PASS（GatePoller delivered=true）但 Annie 看不到通知。验证终点必须是"用户看到什么"，不是"系统内部状态"。

4. **Lead 是 Annie 的唯一沟通对象。** 所有 Annie 看到的信息必须来自 Lead（Peter/Oliver/Simba），不来自 system bot（ClaudeBot）。Bridge 是基础设施，不跟用户对话。

5. **QA 的价值在于发现代码审查找不到的问题。** Codex 4 轮 APPROVED 但 QA 发现了 timeout bug、self-filtering、Lead 误报等。真正的 E2E 不可替代。
