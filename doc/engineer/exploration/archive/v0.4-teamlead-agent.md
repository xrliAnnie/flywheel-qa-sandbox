# v0.4 TeamLead Agent — Brainstorm & Architecture

> Status: Phase 1 SHIPPED (PR #10) — Phase 1.5 (LLM Brain) pending
>
> 综合来源：Claude research + ChatGPT architecture analysis + Codex design review (Round 1-4) + OpenClaw & Codex CLI architecture research

---

## 1. Problem Statement

Flywheel v0.2 完成了并行执行 + Decision Layer + Slack 通知。但当前的 Slack 集成是**哑巴式的**：

- SlackNotifier 输出模板化的 Block Kit 消息（固定格式）
- SlackInteractionServer 只处理按钮点击（approve/reject/defer）
- CEO 无法在 Slack 里问 "GEO-95 改了什么"、"哪个 agent 卡住了"
- 没有跨 issue 的上下文感知（"GEO-95 改了 auth，GEO-96 也在改 auth"）
- 没有主动监控（agent 卡 15 分钟没人知道）

**目标**：一个 24/7 运行的 TeamLead daemon，作为 CEO 与 Flywheel 执行引擎之间的智能中介。

---

## 2. Design Principles

1. **不要重建 Agent Teams** — TeamLead 不是另一个 agent 框架，是一个事件驱动的 daemon + Slack 对话层
2. **确定性逻辑为主，LLM 按需调用** — 大部分逻辑是 "事件 X → 查状态 → 通知"，只有消息措辞和意图解析需要 LLM
3. **最小化 token 消耗** — Haiku 做分类，Sonnet 只做综合/对话，每次调用注入最小上下文
4. **Socket Mode** — 不暴露公网端口，不需要 Tailscale Funnel
5. **增量交付** — Phase 1 先用模板消息验证闭环，Phase 1.5 加 LLM 智能层
6. **统一事件接口** — run-issue.ts 和 run-project.ts 共享同一套事件发射机制

## 4. Key Design Decisions

### 4.1 SQLite: sql.js → better-sqlite3

**现状问题**：AuditLogger 使用 `sql.js`（WASM 实现），每次写入都 `db.export()` + `writeFileSync` 全量写盘。不支持 WAL mode，不支持多进程并发。

**Decision**：TeamLead 使用 `better-sqlite3`（原生 C++ binding），启用 WAL mode。独立 DB 文件 `~/.flywheel/teamlead.db`。

**Spike required**：在 Phase 1 开始前，先验证 `better-sqlite3` 在 macOS arm64 上的安装和运行。如果有问题，降级到 `libsql` 或 `sql.js` + 文件锁方案。

**通信方式**：Flywheel 通过 HTTP POST 事件到 TeamLead daemon（localhost only, 127.0.0.1）。

- 事件携带 `event_id`（UUID）实现幂等去重
- TeamLead 不在线时：Flywheel 的 HTTP POST 会失败，事件丢失（Phase 1 接受此限制；Phase 4 加本地 spool 文件缓冲）
- 无需 HMAC 签名：仅 localhost 通信，攻击面极小

### 4.2 Slack 传输：HTTP → Socket Mode

**Decision**：切换到 **Socket Mode**（`@slack/bolt` 原生支持）。

好处：
- 不需要公网 URL / Tailscale Funnel
- WebSocket 出站连接，穿透 NAT
- 同时接收 Events API + Interactivity 事件

要求：
- Slack App 需要启用 Socket Mode
- 需要 `SLACK_APP_TOKEN`（`xapp-...`）
- App 需要 `connections:write` scope

**Impact on existing code**：
- SlackInteractionServer 保持可用（不在 Phase 1 删除），TeamLead 的 Socket Mode bot 独立运行
- 现有 `run-issue.ts` 的 `waitForAction()` 阻塞模式不受影响
- 长期（Phase 2+）考虑用 `InteractionAdapter` 抽象统一 HTTP/Bolt 两种实现

**Slack 双通路切换策略**（Codex R3 #1）：

Phase 1 引入 feature flag `TEAMLEAD_OWNS_SLACK`（环境变量，默认 `false`）：

| Flag 状态 | 脚本 legacy Slack 路径 | TeamLead Slack 路径 |
|-----------|----------------------|-------------------|
| `false`（默认） | 正常运行（SlackNotifier + SlackInteractionServer） | TeamLead 只收事件入库，不发 Slack 通知 |
| `true` | **完全旁路**：SlackNotifier.notify() skip, waitForAction() 立即返回 null | TeamLead 负责所有 Slack 通知 + 动作处理 |

**关键约束**：
- 同一时刻只有一个路径发通知，杜绝重复消息
- `setupComponents()` 读取 `TEAMLEAD_OWNS_SLACK` 决定是否创建 SlackNotifier 实例
- 验收测试覆盖：每个 issue 仅 1 条 review 通知 + 1 次动作执行（不论 flag 状态）
- Phase 1 开发期间先 `false`（验证事件管道），功能完整后切 `true`（验证 TeamLead 闭环）

### 4.3 Agent 观测：三层信号

| 层级 | 信号源 | 粒度 | Phase |
|------|--------|------|-------|
| **L1: Pipeline events** | ExecutionEventEmitter (注入 Blueprint/DagDispatcher) | issue started/completed/failed/decision | Phase 1 |
| **L2: Claude Code hooks** | Claude Code lifecycle hooks (JSON) | tool use, permission request, errors | Phase 2 |
| **L3: tmux output** | `tmux pipe-pane` → log file | 原始 stdout，用于 "在做什么" 查询 | Phase 2 |

**统一事件接口**（Codex #3 feedback）：定义 `ExecutionEventEmitter` 接口，注入到 Blueprint 或 `setupComponents`，确保 `run-project.ts` 和 `run-issue.ts` 都发同一套事件。

### 4.4 Execution ID（Codex #4 + R3 #3 feedback）

当前 `sessionId` 在 `TmuxRunner.run()` 内部生成，调用前不可得。

**Solution**：引入 orchestrator 级 `execution_id`（UUID v4），**唯一 owner 是 caller 侧**：

- `run-issue.ts`：在调用 `blueprint.run()` 前生成 `executionId = randomUUID()`
- `DagDispatcher.dispatchOne()`：在调用 `blueprint.run()` 前生成 `executionId = randomUUID()`
- `BlueprintContext` 新增 `executionId: string` 必填字段（不是 Blueprint 自己生成）
- Blueprint 内部所有 `emitStarted/emitCompleted/emitFailed` 使用传入的 `executionId`
- `sessionId`（tmux session name）作为可选后补字段，在 runner 返回后填充

**集成测试要求**：
- started + completed 事件携带相同 `execution_id`
- started + failed 事件携带相同 `execution_id`（失败路径）

### 4.5 LLM 调用策略

| 场景 | Model | Phase |
|------|-------|-------|
| 通知消息 | **无 LLM**（模板 Block Kit，复用 SlackNotifier 逻辑） | Phase 1 |
| 按钮点击（approve/reject） | **无 LLM** | Phase 1 |
| 通知消息智能措辞 | Sonnet (~$0.01/call) | Phase 1.5 |
| CEO 自然语言问答 | Sonnet (~$0.01/call) | Phase 1.5 |
| 事件分类（security-sensitive?） | Haiku (~$0.001/call) | Phase 2 |
| 跨 issue 综合分析 | Sonnet (~$0.02/call) | Phase 3 |

**Phase 1 不使用 LLM** — 先用模板消息验证事件管道和 Socket Mode 闭环。

### 4.6 对话模式：Stateless per-message（Phase 1.5+）

每条 CEO 消息触发一次独立的 LLM 调用：

```
CEO 消息 → 加载上下文（从 SQLite）→ 构建 system prompt → Sonnet → 回复 + 可选动作
```

**不用长期 session**，原因：
- 便宜：只在有消息时消耗 token
- 安全：不会把无关 Slack 聊天内容泄入 LLM context
- 可靠：进程重启不丢 session 状态（context 从 DB 重建）

### 4.7 Retry 机制（Codex #8 feedback）

CEO 点 "Retry" 后，TeamLead 需要重新调度 issue。

**Decision**：不直接复用 `run-issue.ts`（有 hardcoded `KNOWN_ISSUES`，不通用）。抽象稳定的 `RunIssueService`（可参数化 fetchIssue/timeout/worktree），TeamLead 和 scripts 共享该服务。

Phase 1 scope：Retry 按钮暂时 disabled（或显示 "coming soon"）。Phase 4 实现 `RunIssueService` 支持独立 retry。

### 4.8 借鉴 Codex CLI Multi-Agent 与 OpenClaw 的 5 个改进

> 研究来源：[Codex CLI](https://github.com/openai/codex) `codex-rs/core/src/agent/control.rs` + [OpenClaw](https://github.com/openclaw/openclaw) `src/infra/heartbeat-runner.ts`

**核心发现**：TeamLead 的上下文传递机制与 Codex multi-agent 本质相同。

Codex 中，子 agent 完成后 runtime 调用 `inject_user_message_without_turn()` 把 `{agent_id, status, last_message}` 注入父 agent 的对话历史，父 LLM 自然看到并推理。

TeamLead 的流程是：Blueprint emitCompleted() → HTTP POST → SQLite → 组装到 Sonnet prompt → LLM 推理。

**信息流完全一样**：结构化状态通知 → 注入 LLM 上下文 → 推理决策。区别只是 transport（Codex 同进程内存注入 vs 我们跨进程 HTTP + DB），因为我们的子 agent 是独立 tmux 进程。

#### 改进 1：last_message / summary 模式（Codex `AgentStatus::Completed(last_message)`）

Codex 的终态携带 `last_message`（agent 最后说的话），不只是 "completed"。

**改进**：`emitCompleted()` 事件增加 `summary` 字段，从 `BlueprintResult.evidence` 中提取：
- 来源：`evidence.diffSummary` + `evidence.commitMessages` 拼接
- 示例："重构了 auth middleware 的 JWT 验证，增加了 token refresh 逻辑和 3 个单元测试"
- 让 TeamLead 能说 "GEO-95 重构了 JWT 验证" 而不只是 "GEO-95 改了 6 个文件"

`sessions` 表新增字段：

```sql
summary TEXT,  -- 自然语言总结（来自 evidence.diffSummary + commitMessages）
```

同理，`emitFailed()` 的 `error` 字段已有，但应确保包含足够的上下文（不只是 error message，还有最后在做什么）。

#### 改进 2：Structured Notification Tags（Codex `<subagent_notification>`）

Codex 用 XML tag `<subagent_notification>` 包裹注入的状态通知，让 LLM 区分"系统注入的状态"和"用户说的话"。

**改进**：Phase 1.5 的 Sonnet prompt 使用结构化 XML tags 注入 agent 状态：

```xml
<agent_status>
- GEO-95: completed | summary: "重构 auth JWT 验证" | commits: 3
- GEO-96: running | started: 10 min ago | no issues
- GEO-97: failed | error: "npm test timeout on auth.test.ts"
</agent_status>

<conversation_history>
CEO (5 min ago): "95 怎么样了"
TeamLead: "GEO-95 完成了，needs review"
</conversation_history>
```

#### 改进 3：Completion Watcher（Codex `maybe_start_completion_watcher()`）

Codex 为每个子 agent 启动 completion watcher，子 agent 完成时自动通知父。如果子 agent 意外消失（`not_found`），也会通知。

**改进**：TeamLead 的 EventIngestion 在收到 terminal 事件时立即评估（Phase 1 已有），但增加 Stuck Watcher：

- Phase 1（简化版）：`setInterval` 每 5 分钟检查 `sessions` 表中 `status = 'running' AND last_activity_at < NOW - 15min`
- 检测到 stuck → 发 Slack 告警（template Block Kit，不需要 LLM）
- Phase 3 升级为 ProactiveMonitor 做更复杂的分析

#### 改进 4：Fork Parent Context（Codex `fork_parent_spawn_call_id`）

Codex spawn 子 agent 时可以 fork 父的对话历史，子 agent 启动时就有项目背景。

**改进**：Phase 1.5+ 中，TeamLead 启动新 execution 前，从 SQLite 查询相关历史注入到 Claude Code 的初始上下文：

- 同一 issue 的 retry 历史（"上次 GEO-95 失败了，原因是 test timeout"）
- 相关文件冲突提示（"GEO-97 正在改 api/routes.ts，注意冲突"）
- 注入方式：通过 SKILL.md 或动态生成的 context 文件

#### 改进 5：OpenClaw Heartbeat 模式（`heartbeat-runner.ts`）

OpenClaw 每 30 分钟（可配置）运行 `HEARTBEAT.md` 清单，检查是否有需要主动处理的事务。无事发生返回 `HEARTBEAT_OK`（不发消息），有事则推送。

**改进**：Phase 1 引入简化版 Heartbeat — Stuck Watcher（见改进 3）。Phase 3 升级为完整 ProactiveMonitor：

- 定时检查清单：stuck agents / 文件冲突 / cost 超标 / daily summary 时间
- 无事发生：静默
- 有事：主动 Slack 推送
- 与 OpenClaw 一样，检查频率可配置（default 5min for stuck, 24h for summary）

## 5. Data Model

### 5.1 TeamLead SQLite Schema

```sql
PRAGMA journal_mode = WAL;

CREATE TABLE session_events (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    event_id TEXT UNIQUE NOT NULL,  -- UUID, 幂等去重
    ts TEXT NOT NULL DEFAULT (datetime('now')),
    execution_id TEXT NOT NULL,     -- orchestrator 级 ID（非 tmux sessionId）
    issue_id TEXT NOT NULL,
    project_name TEXT NOT NULL,
    event_type TEXT NOT NULL,
    -- 'session_started', 'session_completed', 'session_failed',
    -- 'decision_made', 'ceo_action'
    severity TEXT NOT NULL DEFAULT 'info',
    payload JSON,
    source TEXT NOT NULL  -- 'orchestrator', 'claude_hook', 'tmux_log', 'slack', 'monitor'
);

CREATE TABLE sessions (
    execution_id TEXT PRIMARY KEY,
    issue_id TEXT NOT NULL,
    issue_identifier TEXT,
    issue_title TEXT,
    project_name TEXT NOT NULL,
    status TEXT NOT NULL DEFAULT 'pending',
    -- 'pending', 'running', 'completed', 'failed', 'awaiting_review', 'approved', 'rejected'
    started_at TEXT,
    last_activity_at TEXT,
    tmux_session TEXT,
    worktree_path TEXT,
    branch TEXT,
    -- PR 信息：Phase 1 只填 evidence 已有字段，PR URL/number 由 Phase 1.5 从 gh CLI 补采
    last_error TEXT,
    decision_route TEXT,
    decision_reasoning TEXT,
    cost_usd REAL DEFAULT 0,
    commit_count INTEGER DEFAULT 0,
    files_changed INTEGER DEFAULT 0,
    lines_added INTEGER DEFAULT 0,
    lines_removed INTEGER DEFAULT 0,
    summary TEXT,            -- 自然语言总结（借鉴 Codex last_message 模式）
    diff_summary TEXT,
    commit_messages TEXT,  -- JSON array
    changed_file_paths TEXT  -- JSON array
);

-- Phase 1 不实现完整 outbox，直接 fire-and-forget 发 Slack
-- Phase 4 加 outbox table

CREATE TABLE conversation_threads (
    thread_ts TEXT PRIMARY KEY,
    channel TEXT NOT NULL,
    issue_id TEXT,
    summary TEXT,
    last_updated TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX idx_events_execution ON session_events(execution_id);
CREATE INDEX idx_events_issue ON session_events(issue_id);
CREATE INDEX idx_sessions_status ON sessions(status);
```

**Schema 原则**（Codex #11 feedback）：Phase 1 只使用 `BlueprintResult.evidence` 和 `DecisionResult` 中已有的字段。PR URL/number 等需要额外查询的字段推到 Phase 1.5。

### 6.2 Flywheel Integration (`packages/edge-worker/`)

> Codex #2 feedback：TeamLead 直接依赖 edge-worker（接受耦合），复用 handler 逻辑。
> Codex #3 feedback：统一事件发射接口，不仅限于 DagDispatcher。

```
packages/edge-worker/src/
├── ExecutionEventEmitter.ts        # 新接口 + TeamLeadClient 实现
│   interface ExecutionEventEmitter {
│     emitStarted(executionId, issueId, projectName, meta?): Promise<void>
│     emitCompleted(executionId, result: BlueprintResult, summary?: string): Promise<void>
│     emitFailed(executionId, error: string, lastActivity?: string): Promise<void>
│     flush(): Promise<void>  // drain pending events; NoOp returns resolved
│   }
│   // summary 来源：evidence.diffSummary + commitMessages 拼接
│   // 借鉴 Codex AgentStatus::Completed(last_message) 模式
│   class TeamLeadClient implements ExecutionEventEmitter {
│     constructor(baseUrl: string)  // http://127.0.0.1:{port}
│     // HTTP POST with event_id, fire-and-forget (catch errors silently)
│   }
│   class NoOpEventEmitter implements ExecutionEventEmitter { ... }
└── Blueprint.ts                    # 修改: constructor 接受可选 eventEmitter
```

**Blueprint 构造函数修改契约**（Codex R2 #1）：

```typescript
// Blueprint constructor — 新增最后一个可选参数
constructor(
  hydrator, gitChecker, getRunner, shell,
  worktreeManager?, skillInjector?, evidenceCollector?,
  skillsConfig?, decisionLayer?,
  eventEmitter?: ExecutionEventEmitter,  // ← 新增
)
```

**事件触发点**：
- `emitStarted()`: 在 `Blueprint.run()` 开头，worktree setup 之后（此时有 cwd + executionId）
- `emitCompleted()`: 在 `Blueprint.run()` 返回前，构建好 `BlueprintResult` 之后
- `emitFailed()`: 在 catch 块中，构建 error result 之后
- **started 事件不 await**（fire-and-forget），用 `.catch()` 静默吞错误，不影响 Blueprint 主流程
- **terminal 事件（completed/failed）使用 bounded await**（Codex R3 #2）：`await Promise.race([emitCompleted(...), sleep(1000)])`，最多等 1 秒。确保短生命周期进程（如 `run-issue.ts`）不会在退出前丢失尾部关键事件
- `TeamLeadClient` 提供 `flush(): Promise<void>` 方法，drain 内部 pending queue；脚本在 finally 块中调用 `await emitter.flush()`
- `setupComponents()` 在 `scripts/lib/setup.ts` 中注入 emitter（`TeamLeadClient` 或 `NoOpEventEmitter`），根据 `TEAMLEAD_URL` 环境变量决定

**修改 Blueprint**（而非 DagDispatcher）发射事件，因为：
- `run-issue.ts` 直接调用 `blueprint.run()`，绕过 DagDispatcher
- Blueprint 知道 execution 的完整生命周期
- DagDispatcher 仍然可以通过 `onNodeComplete` 做额外处理

**Project Config 映射**（Codex R2 #2）：

```typescript
// ProjectConfig — 最小契约
interface ProjectConfig {
  projectName: string;      // e.g., "geoforge3d"
  projectRoot: string;      // e.g., "/Users/xiaorongli/Dev/GeoForge3D"
  projectRepo?: string;     // e.g., "xrliAnnie/GeoForge3D"
}

// 来源：FLYWHEEL_PROJECTS 环境变量 (JSON) 或 ~/.flywheel/projects.json
```

**SlackActionAdapter**（Codex R2 #2）：

Bolt 的 `action` payload 结构与现有 `SlackAction` 不同。`SlackActionAdapter` 做转换：

```typescript
// Bolt payload → SlackAction
function adaptBoltAction(boltAction: BlockAction, body: BlockPayload): SlackAction {
  return {
    actionId: boltAction.action_id,
    issueId: parseIssueId(boltAction.action_id),
    action: parseAction(boltAction.action_id),
    userId: body.user.id,
    responseUrl: body.response_url ?? "",
    messageTs: body.message?.ts ?? "",
  };
}
```
