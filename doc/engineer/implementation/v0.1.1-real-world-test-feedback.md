### 发现的 Bug（v0.1.1 需要修复）

1. **tmux session name 前缀匹配**：TmuxRunner 用 `flywheel` 作为 session name，tmux 前缀匹配到了遗留的 `flywheel-e2e` session，导致 window 创建在错误的 session 里，viewer 也无法正确 attach。
   - 修复方向：使用唯一 session name（如 `flywheel-{timestamp}` 或 `flywheel-{issueId}`）

2. **GitResultChecker 不支持多 repo 项目**：GeoForge3D 的 parent repo 是一个 umbrella，Frontend 和 Backend 是独立 git repos。`assertCleanTree` 和 `captureBaseline` 检查的是 parent repo，但 Claude 在 sub-repos 里 commit。结果：0 commits detected → 误判 FAIL。
   - 修复方向：支持指定多个 git roots，或自动检测 sub-repos

3. **Viewer 窗口没有自动弹出**：因为 bug #1，osascript 打开的 Terminal 窗口 attach 到了错误的 session。

### CEO 反馈：当前 Flow 的核心问题

> "现在的情况是，它跑完之后就把 PR 丢在那里让我去看，然后它就关闭了。"

当前 v0.1.1 的流程：
```
Issue → Claude 工作 → 创建 PR → Session 关闭 → 结束
```

**CEO 期望的流程**：
```
Issue → Claude 工作 → 创建 PR → 通知 CEO → 等待 CEO 回复 → CEO 说 ship → Claude merge → 结束
                                                        ↗ CEO 说改 X → Claude 修改 → 再次通知
```

### CEO 的具体需求（请在 design docs 中 address）

1. **Session 不应该在 PR 创建后关闭**
   - Claude 创建 PR 后，应该进入"等待审批"状态，而不是退出
   - Session 保持活跃，随时可以接收新指令

2. **实时通知机制**
   - 当 Claude 完成工作（创建 PR）后，应该**立刻**通知 CEO
   - 优先渠道：Slack（v0.2 已有 transport）
   - 通知内容：issue ID、PR 链接、变更摘要、耗时

3. **交互式审批**
   - CEO 看完 PR 后，可以通过 Slack（或其他渠道）回复：
     - "Ship it" → Flywheel merge PR
     - "改 X" → Flywheel 继续修改，改完再通知
     - "放弃" → Flywheel close PR
   - 这是一个 loop，直到 CEO 满意

4. **实时查看进展**
   - CEO 想要能随时看到 Claude 在做什么
   - tmux viewer 需要可靠地自动打开
   - 或者提供一个简单的命令让 CEO 随时 attach

5. **避免 PR 堆积无人知晓**
   - 如果没有通知，多个 issue 并行跑时会产生大量 PR，CEO 不知道哪些需要 review
   - 需要一个 dashboard 或 digest 机制

### 对 v0.2 Design Docs 的具体建议

#### v0.2-decision-layer.md
- 补充 "PR Ready for Review" 作为一个 decision point
- 定义 Slack notification 的 message format（issue、PR link、summary、cost）
- 定义 CEO 回复的 action types（ship / revise / abandon）
- 定义 session persistence：Claude 创建 PR 后进入 WAITING_REVIEW 状态

#### v0.2-parallel-execution.md
- 补充多 repo 项目的处理策略
- 当多个 issue 并行时，notification 如何聚合（每个单独通知 vs batch digest）
- tmux session naming 策略（避免冲突，唯一标识）

#### v0.2-skill-system.md
- 补充 GeoForge3D 测试中发现的问题：Claude 没有 follow 项目自定义 workflow
- SKILL.md 应该能注入项目的 onboarding → plan → implement 流程
- 实际例子：GeoForge3D 有 `.claude/commands/frontend-implementation.md`、`.claude/commands/onboard.md` 等
