# Plan: Session Role/Lane Modeling

**Version**: v1.22.0
**Issue**: FLY-59
**Date**: 2026-04-06
**Source**: `doc/architecture/product-experience-spec.md` (§2.2, §4.3), `doc/architecture/capability-matrix.md`
**Status**: codex-approved

---

## Problem

RunDispatcher uses `issueId` as the inflight map key, enforcing a single active session per issue. The 409 conflict check in `POST /api/runs/start` and `handleRetry()` also deduplicate by issueId alone.

This blocks the QA Agent workflow defined in the Product Experience Spec (§2.2, §4.3): after a Main Runner creates a PR, the Lead should be able to spawn a QA Agent on the **same issue** to test in parallel. Currently the system rejects this as a duplicate.

## Solution

Add a `session_role` field (`'main' | 'qa' | 'review'`) to the session model. Change all deduplication logic from "per issue" to "per issue + role". Propagate role through the full stack: storage → dispatcher → API → events → Lead-facing surfaces. Default to `'main'` for backward compatibility.

## Field Naming Convention

- **camelCase `sessionRole`**: used in TypeScript interfaces for cross-service payloads and in-memory objects (`EventEnvelope`, `BlueprintContext`, `StartRequest`, `RetryRequest`, `BootstrapSession/Decision/Failure`)
- **snake_case `session_role`**: used in SQLite column name, `Session`/`SessionUpsert` interfaces (matching existing `session_stage`, `session_params`), AND `HookPayload` (matching existing `event_type`, `execution_id`, `issue_id` snake_case convention in that type)

## Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Role enum values | `main`, `qa`, `review` | Covers current use cases: Main Runner, QA Agent, future code review. Extensible as plain strings. |
| Default role | `'main'` | All existing sessions auto-classify as main. Zero migration needed. |
| Inflight map key | `${issueId}:${role}` composite string | Simple, fast Map lookup. No need for nested data structures. |
| Forum thread sharing | Same issue shares one thread regardless of role | QA results should appear alongside main execution updates. Annie reviews one thread per issue. |
| Concurrency counting | QA sessions count toward global `maxConcurrentRunners` | Prevents resource exhaustion. Can be refined later if needed. |
| EventFilter rules | No changes | EventFilter classifies by event type, not session role. Role is informational in the event payload. |
| resolve-action ambiguity | Accept optional `sessionRole` param; default `'main'` when omitted | Avoids breaking existing callers while enabling role-specific action resolution. |
| HookPayload construction | No helper extraction — update all 8 existing construction sites inline | Matches existing pattern; introducing a builder for just one new field is over-engineering. |

### Phase 2: Dispatcher Logic (1 file, core change)

#### 2a. `packages/teamlead/src/bridge/run-dispatcher.ts`

**Inflight map key change:**

```typescript
protected inflightKey(issueId: string, role: string): string {
  return `${issueId}:${role}`;
}
```

**RetryDispatcher.dispatch()** changes:
- Line 47: `this.inflight.has(req.issueId)` → `this.inflight.has(this.inflightKey(req.issueId, req.sessionRole ?? 'main'))`
- Line 63: `this.inflight.set(req.issueId, entry)` → `this.inflight.set(this.inflightKey(req.issueId, req.sessionRole ?? 'main'), entry)`
- Line 98: `this.inflight.delete(req.issueId)` → `this.inflight.delete(this.inflightKey(req.issueId, req.sessionRole ?? 'main'))`
- **BlueprintContext** (line 65-82): add `sessionRole: req.sessionRole` to the context object

**RunDispatcher.start()** changes:
- Line 150: Same composite key check
- Context build: add `sessionRole: req.sessionRole`

**New method on RetryDispatcher** (satisfies IRetryDispatcher contract):

```typescript
hasInflightForRole(issueId: string, role: string): boolean {
  return this.inflight.has(this.inflightKey(issueId, role));
}
```

**getInflightIssues()** — keep returning `Set<string>` of unique issueIds (extract from composite keys) for backward-compat callers:

```typescript
getInflightIssues(): Set<string> {
  const issueIds = new Set<string>();
  for (const key of this.inflight.keys()) {
    issueIds.add(key.split(":")[0]);
  }
  return issueIds;
}
```

**getInflightCount()** — change to `return this.inflight.size` directly (not via `getInflightIssues().size`), so same-issue-different-role entries count separately toward global concurrency.

## Backward Compatibility

- All existing sessions get `session_role = 'main'` via SQLite DEFAULT
- All APIs default to `'main'` when `sessionRole` is omitted
- Existing behavior is completely unchanged
- No data migration needed — SQLite ALTER TABLE with DEFAULT handles it
- `getInflightIssues()` still returns unique issueIds for backward compat
- resolve-action defaults to `'main'` when role not specified
- TeamLeadClient serializes `sessionRole` — old bridges that don't parse it simply ignore it

## Risk Assessment

| Risk | Mitigation |
|------|-----------|
| Inflight map key change breaks concurrent counting | `getInflightCount()` uses `this.inflight.size` directly — each entry counted separately |
| `getInflightIssues()` callers expect issueId-only | Preserved: extracts unique issueIds from composite keys |
| resolve-action ambiguity | Explicit role param with `'main'` default; no silent ambiguity |
| `/events` mirror path divergence | Both sender (TeamLeadClient) and receiver (event-route) updated; test scenario #7+#10 validate |
| Lead can't distinguish main/qa events | CommDBLeadRuntime (production) + bootstrap all updated |
| HookPayload missing role | All 8 manual construction sites updated; no helper needed |
| BootstrapDecision/Failure missing role | Both types updated alongside BootstrapSession |

## Not In Scope

- QA Agent spawn logic (FLY-49)
- EventFilter role-based routing
- Per-role concurrency limits
- Forum thread role annotations
