# Plan: Lead Resume 静默吞掉 flywheel-inbox events — 根因 + 修复

**Version**: v1.23.0
**Issue**: FLY-109
**Date**: 2026-04-15
**Source**: Linear FLY-109 + Stage 2 研究（claude-code source + inbox-mcp + claude-lead.sh + Bridge runtime）+ Codex Round 1 design review
**Status**: draft

## 1. 背景 (Problem)

**现象**：Lead Claude `--resume <session-id>` 重启后，inbox-mcp 推送的 `notifications/claude/channel` 消息被静默吞掉。CommDB 里 `read_at` 被写入（消息永远追不回），Lead REPL 里什么都没出现，连续 4 条同时丢失——不是偶发 race。

**影响**：Annie 发给 Lead 的指令只要落在重启窗口内就永久丢失。Workaround（`rm session-id && kickstart fresh`）不可接受——Annie Stage 2 明确要求找根因修真问题。

**Fresh 启动基本正常**——因此分歧必然在 resume 路径或启动时序上。

### 2.2 失败机制

**主因 C1 — Readiness lease 过早 + at-most-once 放大器（由 F1+F2+F3 组合产生）**

- Lead 进程启动 → `inbox-mcp` 立刻写 lease → Bridge 立刻把 lease 视为 "ready" 并开始投递
- 但此时：
  - `inbox-mcp` server 的 stdio transport 还没 `connect`，或者刚 connect 但 client 侧 React tree 还没 mount MCP，`setNotificationHandler(ChannelMessageNotificationSchema, ...)` 还没挂
  - `inbox-mcp` 的 poll loop 已经可能 fire 第一轮（或 Bridge 主动塞了新消息）
- `await server.server.notification(...)` 在 client 无 handler 时仍 resolve（MCP notifications 是 fire-and-forget），`stmtMarkRead.run` 立即落盘
- 后果：**任何 Lead 启动窗口内的消息都可能被永久吃掉**，resume 因为要 replay transcript、REPL mount 被 UI 阻塞，窗口更长更稳定——这与"连续 4 条同时丢失"的症状吻合

**次因 C2 — expect 脚本空分支自终止（由 F6 产生）**

- resume 时上游也会跑 `showSetupScreens()` 完整流程（F5）；DevChannelsDialog 可能出现
- expect 的 `bypass permissions` / `Type your message` 空 body 分支先命中就让整个 expect block 退出，`confirmed` 可能仍为 0
- `setAllowedChannels` 没被扩展 → `gateChannelServer` 对 `flywheel-inbox` 返回 `skip` → notification handler 永不挂 → C1 的 "client 侧无 handler" 变成永久真
- 这是 **独立的可靠性 bug**，即便 C1 修了也应当修

**次要观察**：原 plan 中的 A2 假设（"REPL MCP effect 和 showSetupScreens 的 await 之间有 ordering race"）与 F5 冲突，**删除**。

### 2.3 Fresh vs Resume 差异 — 推断，不是证据

一个**合理推断**（非源码直接证据）：fresh 没有 transcript replay，REPL mount 路径更短，`showSetupScreens` 完成到 MCP handler `setNotificationHandler` 注册的 wall-clock 窗口可能小于 inbox-mcp 第一轮 `pollOnce`（1 秒）；resume 时 REPL 要 replay 大量 message，handler 注册被推后，窗口被拉长。但源码只能证明调用**顺序**，证明不了具体时间。

**本 PR 的 correctness 不依赖该推断成立**——即便 fresh 也偶尔在 race 窗口内，修复方案（§3.1 #1 retry/ack）都能兜住。本节只是为了解释症状分布规律。

## 3. 方向选择 (Direction)

对 Codex Round 1/2 提到的方向：

| Direction | 描述 | 评估 |
|---|---|---|
| A | 改 claude-code 源码让 `--resume` 补跑 DevChannelsDialog / 补挂 handler | 非我们 fork，放弃 |
| **B** | inbox-mcp **durable retry + 显式 ack**（`delivered_at` + `flywheel_inbox_ack` tool） | **Phase 1 primary correctness fix**：这是唯一能覆盖 Lead restart/resume 主场景的机制 |
| C | 把 flywheel-inbox 从 dev channel 升级成 plugin-source channel | **Phase 2 跟踪**：F10 说明需要造 plugin wrapper，不适合本 PR 范围 |
| **D** | 修 expect 脚本空分支 bug（两阶段化 + shell 启动日志） | **Phase 1**：治 C2 独立 bug，与 B 互补 |
| R | Readiness hardening：lease 延后 + Bridge 启动期 pane marker 检查 | **Phase 1 opt-in hardening**：只覆盖 Bridge cold-start / late-register，**不覆盖 Lead restart/resume 主故障路径**（见下） |

### 3.0 Correctness 归属

FLY-109 主故障路径是 **"Lead 已经在跑、Bridge runtime 早已注册、然后 Lead restart/resume"**。此时 Bridge 不会重新跑 `createLeadRuntime`（`RuntimeRegistry` 里 runtime 常驻，Lead 进程起伏不触发 re-register，见 `packages/teamlead/src/bridge/runtime-registry.ts:9` + `event-route.ts:775` + `gate-poller.ts:191`）——**任何"注册时 gate"都对这条主路径无效**。因此：

- **本 PR 的 correctness 100% 由 Direction B（ack/retry）承担**
- Direction R 的价值缩减到：Bridge **首次启动** 和 **late-register**（新 Lead 加入）时多挡一层 startup-window race；**不是 FLY-109 主修复**
- 如果未来决定做"Lead 下线 → unregister runtime → 上线 → re-register"的动态 lifecycle，可作为 Phase 2 issue 跟踪；本 PR 不做

### 3.1 Phase 1 范围（本 PR）

三件事：

1. **Durable retry + 显式 ack（Direction B，主修复）**
   - CommDB `messages` 表加一列 `delivered_at DATETIME NULL`（migration 走现有 `applyMigrations()` 模式）
   - **不改动** `getUnreadInstructions()` 和 `markInstructionRead()` 的既有语义 —— CLI pull path (`packages/flywheel-comm/src/commands/inbox.ts:14`) 继续用旧语义
   - **新增 push 专用 helpers**（在 `db.ts`）：
     - `getPendingPushInstructions(to_agent)`：`read_at IS NULL AND (delivered_at IS NULL OR delivered_at < datetime('now', :retry_window))`
     - `markInstructionDelivered(id)`：写 `delivered_at = now`
     - `ackInstructionRead(id)`：幂等写 `read_at = now`（若已有则 no-op）
   - `inbox-mcp` poll loop 改用新 helpers：成功 `await notification()` → `markInstructionDelivered`；**不再立即写 `read_at`**
   - 新增 MCP tool `flywheel_inbox_ack({ message_id: string })`，Lead 端 Claude 模型**显式调用**（`message_id` 已通过 `params.meta` 暴露给模型，见 F8）
   - 默认 retry window 30s，通过 inbox-mcp 环境变量 `FLYWHEEL_INBOX_RETRY_WINDOW_SEC` 可覆盖
   - 重投语义 at-least-once：同 `message_id` 可能被推多次；Lead 靠 ack 幂等去重，ack 对同 id 重复调用 no-op
   - identity.md / system prompt 增加一段最小侵入指示：处理完一条 inbox message 后必须调 `flywheel_inbox_ack`。具体注入点与 team-lead 对齐

2. **Expect 两阶段化 + 启动日志（Direction D，独立 bug 修复）**
   - 重写 `claude-lead.sh:548-589` 为保守两阶段：
     1. **第一阶段**：`expect` 只等 dialog marker（`Loading development channels` / `I am using this for local development` / `WARNING: Loading development channels`）。超时由新环境变量 `FLYWHEEL_EXPECT_DIALOG_TIMEOUT_SEC` 控制，生产默认 30s，单测覆盖为 1-2s
     2. 看到 marker → `send "1\r"` → `set confirmed 1`
     3. **第二阶段**：`expect eof` —— 不再依赖任何 UI 字符串作为 post-dialog marker（Codex Round 2 指出 `"Type your message"` 在 claude-code 源码里查无此字面量）。`expect eof` 在子进程退出时命中；或给外层 shell 加一个自定义 sentinel 字符串，和 expect 匹配（若证明 eof 不合适）
     4. 第一阶段超时且 `confirmed=0` → 记 `DEV_CHANNELS_DIALOG_NOT_SEEN` 并 **不**盲发 `1\r`
   - 所有 match / timeout / confirmed state 写 `~/.flywheel/logs/lead-<agent>-startup.log`

3. **Readiness hardening（Direction R，opt-in；不作为主修复）**
   - `inbox-mcp`：`writeLease()` 移到 `await server.connect(transport)` 之后，且在 `setInterval(pollOnce, ...)` 调用之后；去掉 "transport 未 connect 即 ready" 这段 race 窗口
   - Bridge `createLeadRuntime`：增加可选 **startup** marker 等待（`Listening for channel messages from:`，来自 `ChannelsNotice.tsx:128-138`）。**明确作为 heuristic**：该 marker 只读 `allowedChannels / auth / policy`，**不证明** MCP handler 已挂，只证明 "REPL 已进入 channels listening UI 分支"。因此该 wait 只能**缩短** startup window，不能消灭 race
   - 20s soft timeout，超时降级为原 lease-only readiness；correctness 仍由 #1 的 retry/ack 承担
   - 不覆盖 Lead restart/resume 主场景（§3.0 已说明）

### 3.2 Phase 2（后续 issue，不进本 PR）

- 造 `flywheel-inbox` minimal plugin wrapper，切到 `--channels` 规范路径（F10 明确路径）
- 评估 channel subscribe handshake（Lead 主动调 tool 订阅后 server 才 push）

## 6. 风险与回滚 (Risks & Rollback)

| 风险 | 缓解 |
|---|---|
| Migration 在真实 CommDB 上失败 | 沿用现有 `applyMigrations` + `PRAGMA table_info` 模式，已被 `read_at / checkpoint / resolved_at` 多次验证；本地先对 backup DB 跑一遍 |
| Bridge readiness marker 20s 超时过严 / 过松 | 超时不 fail，只降级为旧 lease-based readiness；由 §3.1 #1 retry/ack 兜底 |
| Lead 模型"忘记"调 ack tool | 30s 超时重投；多次 retry 后若仍无 ack，inbox-mcp 日志告警（不自动 dead-letter，避免消息丢） |
| 重投导致同一指令被执行两次 | `message_id` 对 Lead 模型可见（已在 MCP meta），识别重复靠模型；ack tool 幂等；**此风险被接受，是 at-most-once→at-least-once 的必要 trade-off** |
| expect 改动破坏 fresh 启动 | A1 scenario + `expect -d` 本地 dry run |
| FLY-83 冲突 | §7 |

**回滚**：PR `git revert` 即可。`delivered_at` 列残留不影响旧代码（旧代码只读 `read_at`）。
