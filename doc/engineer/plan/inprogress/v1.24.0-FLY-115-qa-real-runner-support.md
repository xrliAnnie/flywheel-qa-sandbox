## 0. TL;DR

Extend `scripts/test-deploy.sh` + `scripts/test-teardown.sh` so a test slot hosts a **real Runner** against `xrliAnnie/flywheel-qa-sandbox` (a fork of flywheel). Add `scripts/inject-linear-issue.sh` to trigger `/api/runs/start` directly, bypassing Linear webhook.

Round 1 Codex review uncovered a blocking correctness gap: `/api/runs/start` spawns the Runner inside a FLY-95 `WorktreeManager` worktree whose start point is hard-coded to `origin/main` (`packages/edge-worker/src/WorktreeManager.ts:126`). So `--from-branch` cannot actually test a PR branch unless the Runner worktree start point is also controllable. This plan therefore adds one minimal test-injection hook in `WorktreeManager.create()` (env fallback `FLYWHEEL_RUNNER_START_POINT`). Everything else stays in `scripts/` + config + docs.

**Synthetic mode is removed.** Going forward every slot is real. Rationale: (a) upcoming PRs largely touch Runner/spin, where synthetic coverage is a null test; (b) maintaining two modes doubles the branch matrix across argparse, preflight, clone, Bridge env, teardown, and docs for very little incremental signal; (c) a single real-only mental model keeps the framework coherent and removes the "did we wire the mode guard?" failure class. The sandbox preflight + rebuild are now unconditional.

Ship gates:

1. `test-deploy.sh --from-branch <br> <N>` (or `test-deploy.sh <N>` with default `main`) produces a per-slot sandbox clone in `${SLOT_DIR}/project`, starts Bridge with `FLYWHEEL_RUNNER_START_POINT=refs/remotes/origin/<br>`, writes file-backed `teamlead.db` + `bridge.log`.
2. `inject-linear-issue.sh <N> FLY-XXX` spawns a Runner that creates a FLY-95 worktree at `${SLOT_DIR}/project-<ISSUE>` whose HEAD matches sandbox `<br>` and that can `git push` / `gh pr create` against the sandbox.
3. `test-teardown.sh <N>` removes FLY-95 Runner worktrees, kills `runner-test-slot-<N>` tmux, deletes slot-local branches, prunes, then removes `SLOT_DIR` + CommDB.
4. FLY-108 Round 2 re-runnable on demand.

### 2.3 Scope justification: the WorktreeManager env hook

`Blueprint.ts:190` calls `worktreeManager.create({ mainRepoPath, projectName, issueId })` with no `startPoint`, so `WorktreeManager.create()` falls back to `origin/main`. For QA, this means the Runner always runs against sandbox/main instead of the PR branch under test — which defeats FLY-115's purpose.

The minimal, test-safe change is a 1-line env fallback inside `WorktreeManager.create()`:

```ts
// before: const startPoint = opts.startPoint ?? "origin/main";
// after:  const startPoint = opts.startPoint ?? process.env.FLYWHEEL_RUNNER_START_POINT ?? "origin/main";
```

Behavior in prod is identical (env unset → `origin/main`). `test-deploy.sh` sets the env to `refs/remotes/origin/<from-branch>` on the Bridge process only. This is the smallest hook that meets the AC and avoids touching `Blueprint` or `run-infra`.

### 4.2 Pre-flight (unconditional, immediately after §4.1)

```bash
# ── FLY-115: Pre-flight ───────────────────────────────
REBUILD_LOCK="/tmp/flywheel-qa-rebuild.lock"
SANDBOX_SLUG="xrliAnnie/flywheel-qa-sandbox"
SANDBOX_REMOTE_URL="${FLYWHEEL_SANDBOX_REMOTE_URL:-git@github.com:${SANDBOX_SLUG}.git}"

fail_preflight() {
  echo "ERROR [pre-flight]: $1" >&2
  echo "See doc/qa/framework/real-runner-e2e-guide.md." >&2
  exit 2
}

[[ -n "${LINEAR_API_KEY:-}" ]] || fail_preflight "LINEAR_API_KEY not set (required for /api/runs/start PreHydrator)"
gh auth status >/dev/null 2>&1 || fail_preflight "gh CLI not authenticated (required for Runner gh pr create)"
gh repo view "$SANDBOX_SLUG" >/dev/null 2>&1 \
  || fail_preflight "sandbox repo ${SANDBOX_SLUG} missing. Run: gh repo fork xrliAnnie/flywheel --fork-name flywheel-qa-sandbox --clone=false"

# Serialized preflight block — all the work that would collide across slots
# is done under one flock. Order: better-sqlite3 rebuild + require() probe,
# THEN TypeScript build of the dist artifacts Bridge imports.
log "Preflight under ${REBUILD_LOCK}: better-sqlite3 rebuild + flywheel-edge-worker build"
(
  flock 9
  cd "$REPO_ROOT"

  # 1. native addon rebuild
  pnpm -r --filter flywheel-comm --filter flywheel-inbox-mcp rebuild better-sqlite3
  ( cd "$REPO_ROOT/packages/flywheel-comm" && node -e "require('better-sqlite3')" ) \
    || exit 11
  ( cd "$REPO_ROOT/packages/inbox-mcp" && node -e "require('better-sqlite3')" ) \
    || exit 12

  # 2. Rebuild edge-worker dist so `scripts/run-bridge.ts` → `dist/WorktreeManager.js`
  #    picks up the FLYWHEEL_RUNNER_START_POINT env fallback from §3.1. Without
  #    this, /api/runs/start spawns Runners against stale `origin/main` dist.
  pnpm --filter flywheel-edge-worker build || exit 13

  # 3. Assert the env fallback actually landed in the built artifact. This is a
  #    cheap string check — cheaper than rerunning unit tests under flock, and
  #    it catches the case where someone forgets to rebuild after editing src.
  grep -q 'FLYWHEEL_RUNNER_START_POINT' \
    "$REPO_ROOT/packages/edge-worker/dist/WorktreeManager.js" || exit 14
) 9>"$REBUILD_LOCK" \
  || fail_preflight "preflight failed (better-sqlite3 rebuild, edge-worker build, or dist freshness check)"
```

## 5. Implementation detail — `scripts/test-teardown.sh`

Teardown must clean up **three layers** of state: FLY-95 Runner worktrees + branches, Runner tmux sessions, and the host slot clone.

### 5.1 New Step 1b — Kill Runner tmux sessions (before current Step 5 "Kill Bridge")

```bash
# ── Step 1b (FLY-115): Stop Runner tmux session if present ──
RUNNER_TMUX="runner-test-slot-${SLOT}"
if tmux has-session -t "$RUNNER_TMUX" 2>/dev/null; then
  log "Killing Runner tmux session: ${RUNNER_TMUX}"
  tmux kill-session -t "$RUNNER_TMUX" 2>/dev/null || true
fi
```

### 5.2 New Step 5b — FLY-95 Runner worktrees + slot-local branches (after Bridge kill, before `rm -rf SLOT_DIR`)

```bash
# ── Step 5b (FLY-115): Clean FLY-95 Runner worktrees + slot-local branches ──
HOST_REPO="${SLOT_DIR}/project"
if [[ -d "${HOST_REPO}/.git" ]]; then
  log "Removing FLY-95 Runner worktrees registered under ${SLOT_DIR}"
  # Every worktree path starting with SLOT_DIR is ours — Runner worktrees are
  # named like ${SLOT_DIR}/project-FLY-108 (see Blueprint/WorktreeManager).
  mapfile -t SLOT_WORKTREES < <(
    git -C "$HOST_REPO" worktree list --porcelain \
      | awk '/^worktree /{print $2}' \
      | grep -F "$SLOT_DIR/" || true
  )
  for wt in "${SLOT_WORKTREES[@]}"; do
    # Skip the host worktree itself — it is ${SLOT_DIR}/project and is torn
    # down when we rm -rf SLOT_DIR below.
    [[ "$wt" == "$HOST_REPO" ]] && continue
    log "  worktree remove --force ${wt}"
    git -C "$HOST_REPO" worktree remove --force "$wt" 2>/dev/null || true
    rm -rf "$wt"
  done

  # Delete slot-local branches: qa-slot-${SLOT}-* and FLY-95 Runner branches
  # (pattern: cyrus-${issueId}/... per WorktreeManager.branchName; we match any
  # local ref that was created inside this host repo). It is a per-slot clone,
  # so deletion has no cross-slot impact.
  log "Pruning local branches in ${HOST_REPO}"
  mapfile -t LOCAL_BRANCHES < <(git -C "$HOST_REPO" for-each-ref --format='%(refname:short)' refs/heads/)
  for b in "${LOCAL_BRANCHES[@]}"; do
    [[ "$b" == "main" || "$b" == "master" ]] && continue
    git -C "$HOST_REPO" branch -D "$b" 2>/dev/null || true
  done

  git -C "$HOST_REPO" worktree prune 2>/dev/null || true

  # Verification assertion — fail loud if anything survives inside SLOT_DIR
  REMAINING=$(git -C "$HOST_REPO" worktree list --porcelain \
    | awk '/^worktree /{print $2}' \
    | grep -F "$SLOT_DIR/" | grep -v "^${HOST_REPO}$" || true)
  if [[ -n "$REMAINING" ]]; then
    log "WARN: worktrees still registered in ${SLOT_DIR} after cleanup:"
    echo "$REMAINING" | sed 's/^/    /'
  fi
fi
```

### 5.3 Step 6 (existing) — `rm -rf $SLOT_DIR` stays

Covers the host clone itself + `teamlead.db` + `bridge.log`. Unchanged.

### 5.4 Ordering rationale

- Runner tmux killed **first** (Step 1b) so there is no live process writing to the worktree while we remove it.
- Bridge killed **before** git worktree remove (existing Step 5 → new Step 5b) so no new Runner can be dispatched into a half-cleaned state.
- Worktree remove + branch prune **before** `rm -rf SLOT_DIR` so git's worktree registry gets a chance to prune cleanly.
- Sandbox remote branches are **never** touched — any open PRs on sandbox are Annie's to close/retain.

## 6. `scripts/inject-linear-issue.sh` — full content

```bash
#!/usr/bin/env bash
# FLY-115: Inject a Linear-shaped run request into a test slot, bypassing
# Linear webhook → Bridge routing. Triggers /api/runs/start directly so a
# real Runner spawns against the slot's sandbox clone.
#
# Usage:
#   scripts/inject-linear-issue.sh <slot> <linear-issue-id> [--role main|qa]
#
# Example:
#   scripts/inject-linear-issue.sh 2 FLY-108
set -euo pipefail

SLOT="${1:?Usage: inject-linear-issue.sh <slot> <issue-id> [--role main|qa]}"
ISSUE_ID="${2:?issue-id required}"
shift 2

ROLE="main"
while [[ $# -gt 0 ]]; do
  case "$1" in
    --role) ROLE="${2:?--role requires value}"; shift 2 ;;
    *) echo "ERROR: unknown arg '$1'" >&2; exit 1 ;;
  esac
done

SCRIPT_DIR="$(cd "$(dirname "$0")" && pwd)"
REPO_ROOT="$(cd "$SCRIPT_DIR/.." && pwd)"
SLOTS_FILE="${HOME}/.flywheel/test-slots.json"
[[ -f "$SLOTS_FILE" ]] || { echo "ERROR: ${SLOTS_FILE} missing — deploy slot first" >&2; exit 1; }

SLOT_IDX=$((SLOT - 1))
PORT=$(jq -r ".slots[${SLOT_IDX}].bridgePort" "$SLOTS_FILE")
PROJECT_NAME="test-slot-${SLOT}"
[[ -n "$PORT" && "$PORT" != "null" ]] || { echo "ERROR: slot ${SLOT} not in ${SLOTS_FILE}" >&2; exit 1; }

BRIDGE_URL="http://localhost:${PORT}"
SLOT_DIR="/tmp/flywheel-test-slot-${SLOT}"
BRIDGE_LOG="${SLOT_DIR}/bridge.log"
curl -sf "${BRIDGE_URL}/health" >/dev/null \
  || { echo "ERROR: Bridge ${BRIDGE_URL} not healthy — tail ${BRIDGE_LOG} and redeploy" >&2; exit 1; }

echo "[inject] slot=${SLOT} issue=${ISSUE_ID} project=${PROJECT_NAME} role=${ROLE}" >&2

# Per-invocation temp file so parallel slot injects don't clobber each other
RESP_FILE="$(mktemp -t flywheel-inject.XXXXXX)"
trap 'rm -f "$RESP_FILE"' EXIT

# POST /api/runs/start — see packages/teamlead/src/bridge/runs-route.ts
HTTP_CODE=$(curl -s -o "$RESP_FILE" -w '%{http_code}' \
  -XPOST "${BRIDGE_URL}/api/runs/start" \
  -H 'content-type: application/json' \
  -d "{\"issueId\":\"${ISSUE_ID}\",\"projectName\":\"${PROJECT_NAME}\",\"sessionRole\":\"${ROLE}\"}" \
  || echo "000")

RESP_BODY="$(cat "$RESP_FILE" 2>/dev/null || echo '{}')"
echo "$RESP_BODY" | jq .

case "$HTTP_CODE" in
  200|201|202)
    echo "[inject] /api/runs/start accepted (HTTP ${HTTP_CODE})" >&2
    ;;
  409)
    echo "[inject] HTTP 409 — a run for ${PROJECT_NAME}/${ROLE} is already active (per FLY-59 dedup). This is usually fine for QA re-runs; stop the existing run first if not." >&2
    exit 3
    ;;
  502)
    echo "[inject] HTTP 502 — /api/runs/start PreHydrator could not verify the issue via Linear. Check: (a) LINEAR_API_KEY on the Bridge process, (b) ${ISSUE_ID} actually exists in Linear, (c) network reachable to linear.app. Tail ${BRIDGE_LOG}." >&2
    exit 4
    ;;
  503)
    echo "[inject] HTTP 503 — Bridge reports it cannot initialize the PreHydrator (LINEAR_API_KEY missing on the Bridge env). Re-deploy the slot and confirm the key." >&2
    exit 5
    ;;
  *)
    echo "[inject] unexpected HTTP ${HTTP_CODE}. Tail ${BRIDGE_LOG}." >&2
    exit 6
    ;;
esac
```

Rationale:
- Single deterministic HTTP path. Per Codex R1 #8, dropped the `--via-lead` alt path — it was redundant given direct `/api/runs/start` dispatch.
- Explicit status-code handling for 502 (Linear verify failure) + 503 (Bridge can't init PreHydrator) + 409 (dedup) covers the three known failure modes observed in `runs-route.ts`.
- `mktemp -t flywheel-inject.XXXXXX` + `trap rm` — multiple slot injects can run in parallel without clobbering a shared `/tmp/inject-resp.json`.
- No `flywheel-comm` invocation needed.

## 10. Risks / mitigations

| Risk | Mitigation |
|------|-----------|
| Env hook in `WorktreeManager` leaks into prod (someone sets `FLYWHEEL_RUNNER_START_POINT` unintentionally) | Env is opt-in, unset in all production launchers (`restart-services.sh`, Lead daemon scripts) — verified by grep in implementation step. Default fallback is unchanged (`origin/main`). |
| Sandbox branch diverges from flywheel main and breaks spin.md | `doc/qa/framework/sandbox-sync-guide.md` recipe plus §1 step 5 gate Annie's sync cadence. spin.md uses `origin`/`main` — with sandbox as origin, behavior is equivalent. |
| `pnpm rebuild` collides across slots | `flock /tmp/flywheel-qa-rebuild.lock` serializes rebuild + per-package `require('better-sqlite3')` verification gates success. |
| Stale `dist/WorktreeManager.js` silently ignores the new env fallback (Bridge imports `packages/edge-worker/dist/index.js` via `scripts/run-bridge.ts`) | §4.2 preflight runs `pnpm --filter flywheel-edge-worker build` under the same flock as the native rebuild, then `grep -q FLYWHEEL_RUNNER_START_POINT dist/WorktreeManager.js` fails fast if the build did not emit the expected code. §8 smoke #2 also asserts dist mtime. |
| Teardown leaves FLY-95 worktrees in host repo if git DB is corrupted | Best-effort `worktree remove --force`, then `rm -rf` the enclosing `SLOT_DIR` regardless. Post-teardown check logs WARN if any `${SLOT_DIR}` paths remain in worktree list. |
| Concurrent real-mode deploy exhausts disk (4× clones in `/tmp`) | Sandbox is a flywheel fork, ~50-80 MB per clone. 4× on `/tmp` is fine on dev macs. Documented in real-runner-e2e-guide. |
| Runner dedup (FLY-59) rejects re-injects | Documented in §6 inject-script (409 handling); Annie can teardown+redeploy or wait for session to end. |
| Runner worktree cleanup races Bridge still processing a run | Teardown kills Runner tmux (Step 1b) then Bridge (existing Step 5) before touching worktrees — enforced ordering. |
