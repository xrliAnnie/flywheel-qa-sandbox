# Plan: Discord Chat Thread Per Issue — Lead Thread Reply

**Version**: v1.22.0
**Issue**: FLY-91
**Date**: 2026-04-12
**Source**: `doc/architecture/product-experience-spec.md` §2.3, §7
**Status**: codex-approved
**Codex Review**: Round 1 CR → Round 2 CR → Round 3 **APPROVED**

---

## 1. Problem

Annie 在 chatChannel（主频道）跟 Lead 讨论多个 issue 时，所有对话混在一起。Lead 的回复（包括 Bridge 推送的通知）全部发到主频道，导致不同 issue 的对话串线。

Annie 期望：每个 issue 有独立的 Discord thread，Lead 在 thread 内回复。

## 2. Design Decision: Bridge 创建 (Recommended)

### 为什么不让 Lead 创建？

**Lead 创建 thread 在当前架构下不可行。** `ForumPostCreator.ts:1-4` 明确记录了这个约束：

> "Claude Lead can't create threads (Discord MCP plugin limitation), so Bridge does it automatically."

Discord MCP plugin 不暴露 thread 创建能力。即使未来 plugin 支持，仍存在：
- Lead 行为不确定（compact 后丢失 mapping）
- 每个 Lead 需要独立管理 thread 状态，不如 SQLite 可靠
- Annie 明确 concern："Lead 管 thread mapping 会不会太复杂"

### Bridge 创建的优势

- 复用 ForumPostCreator 的成熟模式（已在 prod 运行数月）
- Lead 逻辑最简单——收到 thread ID 就发
- Bridge 已有 Discord API 基础设施（bot token、validateThreadExists、error handling）
- Thread mapping 在 SQLite 中持久化，不受 Lead compact/restart 影响

### v1 Scope 说明

这是 **Runner-linked v1**：Chat thread 在 `session_started` 时 eager create。不覆盖 session 启动前的 issue 讨论场景。未来如需在 session 之外创建 chat thread，可以增加 lazy-ensure API（Bridge-owned），但 v1 不做。

## 3. Scope

**In scope**:
- **Feature flag**: `chatThreadsEnabled` boolean in BridgeConfig — 关闭时系统行为与当前完全一致
- Bridge 在 `session_started` 时为每个 issue 创建 chatChannel thread（await，非 fire-and-forget）
- StateStore 扩展: `chat_threads` 表 + CRUD 方法
- HookPayload 扩展: `chat_thread_id` 字段
- **所有** issue-scoped 通知路径填充 `chat_thread_id`（DirectEventSink、HeartbeatService、actions.ts、gate-poller.ts）
- CommDB instruction formatting: 包含 `Chat-Thread`（gate_question special format 也覆盖）
- Lead prompt rules + Lead 参考文档更新
- Bootstrap snapshot 包含 chat_thread_id

**Out of scope**:
- Forum thread 行为（不改）
- 强制所有对话进 thread（不强制）
- Thread 生命周期管理（auto-archive 由 Discord 管）
- Session 之外的 issue 讨论 threading（v2）
- Legacy `/events` HTTP 路径（已被 DirectEventSink 替代，不维护）

## 4. Implementation

### 4.1 StateStore — `chat_threads` 表

```sql
CREATE TABLE IF NOT EXISTS chat_threads (
    thread_id TEXT PRIMARY KEY,
    channel_id TEXT NOT NULL,
    issue_id TEXT,
    lead_id TEXT,
    created_at TEXT DEFAULT (datetime('now')),
    discord_missing_at TEXT
);
CREATE UNIQUE INDEX IF NOT EXISTS idx_chat_threads_issue_channel
    ON chat_threads(issue_id, channel_id);
```

**设计决策**: 用新表而非复用 `conversation_threads`。

**唯一键**: `(issue_id, channel_id)` 而非仅 `issue_id`。原因：
- Lead 是按 label 动态解析的，不同 Lead 有不同的 `chatChannel`
- 如果 issue label 变化导致切 Lead/channel，旧 thread 在旧 channel 里不应被复用
- 查询时必须同时匹配 `issue_id` + `channel_id`

新增方法：

```typescript
upsertChatThread(threadId: string, channelId: string, issueId: string, leadId?: string): void
getChatThreadByIssue(issueId: string, channelId: string): { thread_id: string; channel_id: string } | undefined
markChatThreadMissing(threadId: string): void
```

**`upsertChatThread()` SQL 语义**: 采用 delete-first 模式（同 `upsertThread()` 的 Forum 实现）：

```typescript
upsertChatThread(threadId: string, channelId: string, issueId: string, leadId?: string): void {
    // Delete any prior mapping for this issue+channel (unless same thread_id)
    this.db.run(
        "DELETE FROM chat_threads WHERE issue_id = ? AND channel_id = ? AND thread_id != ?",
        [issueId, channelId, threadId],
    );
    this.db.run(
        `INSERT INTO chat_threads (thread_id, channel_id, issue_id, lead_id)
         VALUES (?, ?, ?, ?)
         ON CONFLICT(thread_id) DO UPDATE SET
            channel_id = excluded.channel_id,
            issue_id = excluded.issue_id,
            lead_id = excluded.lead_id,
            discord_missing_at = NULL`,
        [threadId, channelId, issueId, leadId ?? null],
    );
    this.save();
}
```

这确保 404 → 重建场景下，旧 missing row 被正确替换，不会撞唯一约束。

### 4.2 ChatThreadCreator — 新模块

文件: `packages/teamlead/src/bridge/ChatThreadCreator.ts`

```typescript
import type { StateStore } from "../StateStore.js";
import { validateThreadExists } from "./thread-validator.js";

const DISCORD_API = "https://discord.com/api/v10";

export interface ChatThreadContext {
    chatChannelId: string;
    issueId: string;
    issueIdentifier?: string;
    issueTitle?: string;
    botToken: string;
    leadId?: string;
}

export interface ChatThreadResult {
    created: boolean;
    threadId?: string;
    error?: string;
}

export class ChatThreadCreator {
    constructor(private store: StateStore) {}

    async ensureChatThread(ctx: ChatThreadContext): Promise<ChatThreadResult> {
        // 1. Check existing (must match both issueId AND chatChannelId)
        const existing = this.store.getChatThreadByIssue(ctx.issueId, ctx.chatChannelId);
        if (existing) {
            const valid = await validateThreadExists(existing.thread_id, ctx.botToken, {
                markDiscordMissing: (id) => this.store.markChatThreadMissing(id),
            });
            if (valid) return { created: false, threadId: existing.thread_id };
        }

        // 2. Create PUBLIC_THREAD in chatChannel (with timeout)
        const threadName = ctx.issueIdentifier
            ? `[${ctx.issueIdentifier}] ${ctx.issueTitle ?? ctx.issueId}`
            : (ctx.issueTitle ?? ctx.issueId);

        const controller = new AbortController();
        const timeout = setTimeout(() => controller.abort(), 5000); // 5s fail-open

        let res: Response;
        try {
            res = await fetch(`${DISCORD_API}/channels/${ctx.chatChannelId}/threads`, {
                method: "POST",
                headers: {
                    Authorization: `Bot ${ctx.botToken}`,
                    "Content-Type": "application/json",
                },
                body: JSON.stringify({
                    name: threadName.slice(0, 100),
                    type: 11, // PUBLIC_THREAD
                    auto_archive_duration: 4320, // 3 days
                }),
                signal: controller.signal,
            });
        } catch (err) {
            clearTimeout(timeout);
            const msg = (err as Error).name === "AbortError" ? "timeout (5s)" : (err as Error).message;
            console.warn(`[ChatThreadCreator] Discord request failed: ${msg}`);
            return { created: false, error: msg };
        }
        clearTimeout(timeout);

        if (!res.ok) {
            const body = await res.text().catch(() => "");
            console.warn(`[ChatThreadCreator] Discord ${res.status}: ${body.slice(0, 200)}`);
            return { created: false, error: `Discord ${res.status}` };
        }

        const data = (await res.json()) as { id?: string };
        if (!data.id) return { created: false, error: "no thread ID in response" };

        this.store.upsertChatThread(data.id, ctx.chatChannelId, ctx.issueId, ctx.leadId);
        console.log(`[ChatThreadCreator] Created chat thread for ${ctx.issueIdentifier ?? ctx.issueId}: ${data.id}`);
        return { created: true, threadId: data.id };
    }
}
```

### 4.3 Shared Helper — `resolveChatThreadId()`

所有通知路径共用一个 helper 函数，避免重复查询逻辑：

```typescript
// packages/teamlead/src/bridge/chat-thread-utils.ts
export function resolveChatThreadId(
    store: StateStore,
    issueId: string,
    chatChannelId: string,
): string | undefined {
    const row = store.getChatThreadByIssue(issueId, chatChannelId);
    return row?.thread_id;
}
```

使用点：
- `DirectEventSink.pushNotification()` — 填充 `hookPayload.chat_thread_id`
- `HeartbeatService` — guardrail events (stuck/orphaned/stale)
- `actions.ts sendActionHook()` — action_executed
- `gate-poller.ts` — gate_question payload

### 4.5 DirectEventSink — 集成

**关键变更**: `emitStarted()` 中 ChatThread 创建改为 **await**（非 fire-and-forget）。

原因：`chat_thread_id` 不参与 `EventFilter` 分类逻辑（不同于 Forum `thread_id`），所以 await 不会破坏现有 `session_started → notify_agent` 语义。

```typescript
// emitStarted() — BEFORE pushNotification():
if (this.chatThreadCreator && lead.chatChannel) {
    const botToken = fpLead.botToken ?? this.config.discordBotToken;
    if (botToken) {
        try {
            const result = await this.chatThreadCreator.ensureChatThread({
                chatChannelId: lead.chatChannel,
                issueId: env.issueId,
                issueIdentifier: env.issueIdentifier,
                issueTitle: resolvedTitle,
                botToken,
                leadId: lead.agentId,
            });
            console.log(`[DirectEventSink] ensureChatThread: created=${result.created} threadId=${result.threadId ?? "none"}`);
        } catch (err) {
            console.warn(`[DirectEventSink] ensureChatThread failed:`, (err as Error).message);
            // Non-fatal: Lead falls back to main channel
        }
    }
}

// pushNotification() — after building hookPayload:
const chatThreadId = resolveChatThreadId(this.store, env.issueId, lead.chatChannel);
if (chatThreadId) {
    hookPayload.chat_thread_id = chatThreadId;
}
```

Forum Post 继续 fire-and-forget（不变）。

### 4.11 Lead Prompt Rules + Docs

## Chat Thread

Bridge 为每个 issue 在你的 chatChannel 里创建一个 Discord thread。

- `markChatThreadMissing()` — missing thread 不被查询返回

### Unit Tests — Notification Paths

- `DirectEventSink.pushNotification()` — payload 包含 `chat_thread_id`

- `actions.ts sendActionHook()` — payload 包含 `chat_thread_id`
- `GatePoller` payload — 包含 `chat_thread_id`

### Unit Tests — Edge Cases

- channel/lead 变化后不复用旧 channel 的 chat thread
- `emitStarted()` 第一条通知就带 `chat_thread_id`（await 验证）
- ChatThreadCreator 创建失败 → pushNotification 仍然执行（non-fatal）
- ChatThreadCreator Discord 超时 (5s) → 返回 error，emitStarted 继续执行
- Bootstrap snapshot 包含 `chatThreadId` 字段 + formatBootstrap 渲染 `[chat:...]` tag

### Integration Tests

- session_started → chat thread 创建 + 第一条通知包含 thread ID
- 同一 issue 重启 → 复用已有 chat thread
- Thread Discord 404 → 重新创建
- Bootstrap snapshot → 包含 chat thread info

### E2E Verification

- Lead 收到带 Chat-Thread 的通知 → 在 thread 里回复
- Annie 在 thread 里发消息 → Lead 在同一 thread 回复
- gate_question → Lead 在 chat thread 里转达给 Annie

## 7. Edge Cases

| 场景 | 处理 |
|------|------|
| Discord auto-archive (3 天不活跃) | 下次 API 调用时 Discord 自动 unarchive |
| Bot 无 CREATE_PUBLIC_THREADS 权限 | 创建失败 → error log → Lead 退回主频道 |
| chatChannel 未配置 | 跳过 chat thread 创建 |
| 同 issue 多个 session (retry) | 复用已有 chat thread（同 channel） |
| Issue label 变化 → 切 Lead/channel | 新 channel 创建新 thread（不复用旧 channel 的） |
| Lead compact/restart | Bootstrap snapshot 包含 chat_thread_id |
| ChatThreadCreator await 超时 | catch → log → 退回主频道 |

## 8. Risks

| Risk | Mitigation |
|------|-----------|
| `emitStarted()` await 增加延迟 | Discord API 通常 <500ms，且只在 session_started 时发生 |
| Discord API rate limit | 每个 session 只创建一次 thread |
| Lead compact 后忘记 thread | Bridge 每次通知都包含 chat_thread_id + bootstrap 恢复 |
| 多 Lead 同时处理同一 issue | `(issue_id, channel_id)` 唯一键避免重复 |

## 9. Known Limitations (v1)

- Chat thread 只在 `session_started` 时创建。如果 Annie 想在 session 启动前讨论某个 issue，thread 还不存在
- Legacy `/events` HTTP 路径不覆盖（已被 DirectEventSink 替代）
- Thread 不会在 issue Done 后自动归档（依赖 Discord 3 天 auto-archive）
