## Fix

### Change 1: Expand `terminate` to all started non-terminal states

**File**: `packages/core/src/workflow-fsm.ts`

**WORKFLOW_TRANSITIONS** â€” add `terminated` as a valid target from all non-terminal states:

```typescript
export const WORKFLOW_TRANSITIONS: Record<string, string[]> = {
  pending: ["running"],
  running: ["awaiting_review", "completed", "blocked", "failed", "terminated"],
  awaiting_review: ["approved_to_ship", "rejected", "deferred", "shelved", "terminated"],
  approved_to_ship: ["completed", "failed", "terminated"],
  blocked: ["deferred", "shelved", "terminated"],
  failed: ["shelved", "terminated"],
  rejected: ["shelved", "terminated"],
  deferred: ["shelved", "terminated"],
  // Terminal states (no outgoing transitions)
  approved: [],
  completed: [],
  shelved: [],
  terminated: [],
};
```

**ACTION_DEFINITIONS** â€” expand terminate's `fromStates`:

```typescript
{
  action: "terminate",
  fromStates: [
    "running",
    "awaiting_review",
    "approved_to_ship",
    "blocked",
    "failed",
    "rejected",
    "deferred",
  ],
  targetState: "terminated",
}
```

**Rationale**: `terminate` means "Annie explicitly orders this killed." It should work from
any started non-terminal state (i.e., the session has begun execution but hasn't reached a
terminal state). `handleTerminate()` in `actions.ts` already handles tmux kill + state
transition â€” no changes needed there.

**Note**: `pending` is intentionally excluded â€” a pending session has no tmux to kill and
hasn't started any work.

### Change 2: Fix `close-tmux` state validation

**File**: `packages/teamlead/src/bridge/plugin.ts`

Replace the hardcoded terminal-state whitelist with a "tmux protected states" model:

```typescript
// States where Runner still needs tmux (actively working or about to ship)
const tmuxProtectedStates = new Set(["running", "approved_to_ship"]);
if (tmuxProtectedStates.has(session.status)) {
  res.status(409).json({
    error: `Cannot close tmux for session in "${session.status}" state â€” Runner still needs tmux`,
  });
  return;
}
```

**Protected states**:
- `running` â€” Runner is actively coding
- `approved_to_ship` â€” Runner is executing ship stage (merge + CI); killing tmux would break this

**Not protected** (tmux can be closed):
- `awaiting_review`, `blocked`, `failed`, `rejected`, `deferred`, `shelved` â€” Runner is idle
- `completed`, `terminated`, `approved` â€” session is done
- `pending` â€” no tmux exists yet; endpoint returns "No tmux target found" naturally

**Rationale**: The semantic intent is "don't close tmux while Runner still needs it."
This replaces the misleading "terminal states" concept with explicit "tmux protected states."
The old whitelist incorrectly mixed terminal/non-terminal states and missed several valid
closeable states (`rejected`, `deferred`, `shelved`).

Remove the "Only works on running sessions" note from the terminate action documentation.

## Impact Analysis

### Backward Compatibility

- **No breaking changes**: existing API endpoints keep same signatures
- **Additive only**: new FSM transitions don't remove any existing ones
- **close-tmux**: becomes more permissive (allows more states), never less

### Risk Assessment

| Risk | Mitigation |
|------|-----------|
| Accidental terminate | terminate requires explicit CEO action via Lead â€” no auto-trigger path |
| close-tmux on active session | `running` and `approved_to_ship` protected; `pending` has no tmux target (returns "No tmux target found") |
| Existing tests break | Only additive changes â€” existing tests remain valid |
