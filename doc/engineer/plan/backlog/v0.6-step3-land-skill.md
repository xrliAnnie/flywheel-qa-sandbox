# v0.6 Step 3: Land Skill — 自动化 PR Merge

## Summary

添加自动化 PR landing 能力：PR 创建后 → 监控 CI → 处理 review 反馈 → 自动 merge。参考 OpenAI Symphony 的 `land` skill（`land_watch.py` 19KB 异步监控脚本）。

## Background

### 现状

Flywheel 的 session 生命周期在 PR 创建后就基本结束了：
- `run-issue.ts:332-338` 检测到 PR 创建或 push 后，10s grace period 后执行 `kill-pane`，主动结束 session
- 默认 session timeout 10 分钟（`scripts/run-issue.ts:358`，`scripts/lib/setup.ts:100`）
- `Blueprint` 的系统 prompt（`packages/edge-worker/src/Blueprint.ts:187-196`）步骤 6 是 "When all work is complete, stop and wait"
- PR 的后续流程（CI 通过、review 通过、merge）需要人工操作或项目自己的 skill（如 GeoForge3D 的 `/ship-pr`）

### 现有 auto-merge 路径

TeamLead bridge 已经有一条 auto-merge 路径：
- `Blueprint` 在 runner 结束后发送 `session_completed` event（`ExecutionEventEmitter.emitCompleted()`）
- `event-route.ts:175-224` 在 `decision.route === "auto_approve"` 时调用 `approveExecution()`
- `approveExecution()`（`actions.ts`）执行 `gh pr merge --squash`

**问题**：这条路径只处理 "Decision Layer 认为可以 auto-approve" 的情况，不处理 CI 失败修复、review 反馈、merge conflict 等场景。

### 目标

让 Flywheel 管理 PR 的完整生命周期：

```
PR created → CI running → CI passed → Review approved → Squash merge
                ↓ fail         ↓ fail        ↓ rejected
             Fix & push    Fix & push    Address feedback
```

### 参考: Symphony `land` Skill

Symphony 的 `land` skill 包含：

1. **`land_watch.py`**（19KB）: 异步监控脚本
   - 轮询 GitHub API 查 CI status + review status + branch protection
   - 检测 merge conflicts → auto-rebase
   - 检测 review comments → 总结回给 agent
   - CI 通过 + review approved → squash merge
   - 超时 / 持续失败 → 上报

2. **Codex Skill Prompt**: 指导 agent 如何调用 `land_watch.py` 和处理各种场景

### 我们的差异

- Symphony 用 `land_watch.py`（Python 脚本）+ Codex。我们用 Claude Code + `gh` CLI。
- Symphony 的 `land` 是给 Codex agent 调用的 skill。我们做成 **Claude Code skill**（在 session 内由 agent 调用）。
- GeoForge3D 已经有 `/ship-pr` skill，用 `:cool:` emoji 触发 deploy + merge。

## Design

### 方案选择

**方案 A: Flywheel 系统级** — session 创建 PR 后，Flywheel orchestrator 接管 landing 流程
- 优点：不依赖 Claude Code session 存活，更可靠
- 缺点：需要在 orchestrator 层新增 PR 监控 loop

**方案 B: Claude Code Skill** — session 在创建 PR 后继续运行，调用 `flywheel-land` skill 完成后续流程
- 优点：利用现有 session，逻辑在 skill 内，灵活
- 缺点：session 必须保持存活；需要修改 session 生命周期

**推荐：方案 B**（Claude Code Skill），理由：
1. 用户已经明确说"希望 session 把 end-to-end 的东西跑完"
2. GeoForge3D 已有类似的 `/ship-pr` skill，只是需要增强
3. 系统级方案会大幅增加 orchestrator 复杂度

**方案 B 需要的 session 生命周期改造**（纳入本 Step scope）：
1. **改造 session 终止检测**：不再在检测到 PR/push 后立即 kill-pane。新的终止信号下沉到 `TmuxRunner` 层（所有入口共享，包括 `run-issue.ts` 和 `DagDispatcher`），在 `waitForCompletion()` 的 pane_dead 轮询循环中增加 sentinel 文件检测：检查 `<cwd>/.flywheel/land-status.json` 是否存在且包含终态（`status === "merged"` 或 `status === "failed"`）。检测到终态后 kill-pane，触发 session 结束。理由：
   - `run-issue.ts` 不知道 worktree 路径（`cwd` 是 `Blueprint.runInner()` 内部创建的局部变量），把 sentinel 放在 `run-issue.ts` 层不可行
   - `DagDispatcher` / `run-project.ts` 路径也需要同样的终止逻辑
   - `TmuxRunner.waitForCompletion()` 已经有 5s 间隔的轮询循环，是最佳 owner
   - **注意**：sentinel 检测到终态后 **不立即 settle**。为避免 Blueprint 在 Claude session 仍存活时开始后处理（读 git、做 decision、可能 kill window），sentinel 触发后先进入 **grace period**（最多 30s），在此期间继续轮询 pane_dead。如果 30s 内 pane 退出 → `settle(false)`；如果超时 → 仍然 `settle(false)`（agent 写完 signal 后通常很快退出，30s 足够）。这保持了 `runner.run()` 返回时 session 已真正结束的语义，避免 Blueprint 对 live session 做 evidence collection / kill window
   - **不主动 kill-pane**。Window lifecycle 由 Blueprint 控制：`auto_approve` → kill window，`blocked/needs_review` → preserve window for inspection
2. 延长 session timeout 至 45 分钟（单一硬超时值，替代当前 10min）
3. 修改 `Blueprint` 系统 prompt 的完成条件：从 "创建 PR 后停止" 改为 "PR landed 后停止"
4. 统一 force-push policy：在 `flywheel-git-workflow` skill 中添加例外："仅在 rebase 解决 merge conflict 时允许 force push（single-author PR only）"

### Post-PR Lifecycle 的唯一 Owner

**关键设计决策**：land skill 负责 PR 的 CI/review/merge 流程，但 **不负责发送 `session_completed` event 或通知 Slack**。职责分离：

```
Land Skill (session 内)           Blueprint/TeamLead (session 后)
├── 监控 CI                       ├── 发送 session_completed event
├── 修复 CI 失败                   ├── Decision Layer 评估
├── 处理 merge conflicts           ├── 通知 Slack/OpenClaw
├── 直接 squash merge (不用 --auto) └── 更新 session 状态
└── 写 land-status.json（结果信号）
```

- Land skill 执行 `gh pr merge --squash`（立即 merge，不用 `--auto`）
- Land skill 完成后写 `.flywheel/runs/<executionId>/land-status.json`（机器可读的 landing 结果信号，见下方详细设计）。路径按 execution 隔离，避免 non-worktree 模式下跨 session 状态污染
- TmuxRunner sentinel 检测到 `land-status.json` 终态后 `settle(false)`（让 `waitForCompletion()` 返回），Blueprint 接管后续 window lifecycle
- `Blueprint` 的正常流程（evidence collection → Decision Layer → event emission）继续执行
- Evidence collection 阶段读取 `land-status.json`，并 **用 GitHub API 二次验证**：如果 signal 声称 `merged`，collector 执行 `gh pr view <prNumber> --json state,mergedAt` 确认 PR 确实已合并。验证失败则降级为 `{ status: "failed", failureReason: "verification_failed" }`。这防止 agent 过早/错误写入 signal 导致 false positive
- Decision Layer 根据 **验证后的** landing outcome 调整路由（landed → auto_approve without re-merge；land_failed → blocked）
- 现有的 `event-route.ts` auto_approve 路径将被修改：**如果 PR 已经被 merged（经验证），跳过 `approveExecution()`**

### Landing 结果信号

Land skill 在退出前写入 `.flywheel/runs/<executionId>/land-status.json`（在 worktree/repo 目录下）。路径按 execution 隔离，避免 non-worktree 模式下跨 session 串状态：

```json
{
  "status": "merged",
  "prNumber": 42,
  "mergedAt": "2026-03-08T12:34:56Z",
  "mergeCommitSha": "abc123"
}
```

失败时：

```json
{
  "status": "failed",
  "prNumber": 42,
  "failureReason": "ci_fix_exhausted",
  "failureDetail": "CI failed after 2 fix attempts: test suite timeout"
}
```

`failureReason` 枚举值：`ci_fix_exhausted` | `ci_timeout` | `review_timeout` | `merge_conflict` | `merge_failed` | `parse_error` | `signal_missing` | `verification_failed` | `unknown`

#### 自定义 `land_command` 与 signal 契约

**Signal 写入是所有 landing 路径的硬性契约**，无论是默认 `flywheel-land` 还是自定义 `land_command`：

- 默认 `flywheel-land` skill 在 prompt 中明确要求写 `land-status.json`
- 自定义 `land_command` 也必须写 `land-status.json`。Blueprint 系统 prompt 会在 `land_command` override 指令后附加："无论使用何种 landing 命令，完成后必须写 `.flywheel/runs/<executionId>/land-status.json`"
- 如果项目的自定义命令不写 signal（如老版本 `/ship-pr`），需要先升级该命令添加 signal 写入步骤，否则不要配置 `land_command`（让 Flywheel 使用默认的 `flywheel-land`）
- 理由：session 终止（TmuxRunner sentinel）、Decision Layer 路由、TeamLead skip-remerge guard 全部依赖此文件。如果 signal 不是硬性要求，override 路径的 session 会挂到 45min 硬超时，且 TeamLead 仍会尝试重复 merge

#### Signal 三态语义（显式，无隐式缺省）

`land-status.json` 的消费逻辑需要区分三种情况，避免语义歧义：

| 场景 | `land-status.json` 状态 | `landingStatus` 值 | Decision Layer 行为 |
|------|------------------------|--------------------|--------------------|
| 未尝试 landing（老 session、agent 未创建 PR） | 不存在 | `undefined` | 走现有完整流程（不变） |
| Landing 启动了但未完成（硬超时/crash） | `{ "status": "pending" }` | `{ status: "failed", failureReason: "signal_missing" }` | 路由 `blocked` |
| Landing 完成 — 成功 | `{ "status": "merged", ... }` | 原样传递 | Early return `auto_approve` |
| Landing 完成 — 失败 | `{ "status": "failed", ... }` | 原样传递 | HR-010 → `blocked` |
| Signal 文件存在但解析失败 | 存在但 malformed | `{ status: "failed", failureReason: "parse_error" }` | 路由 `blocked` |

**关键实现**：Evidence collector 需要知道 landing 是否被尝试过。判断方式：**运行时 marker 文件**，而非静态 skill 注入推断。

- Land skill（无论默认 `flywheel-land` 还是自定义 `land_command`）在启动时（创建 PR 后、开始轮询前）立即写入 `<cwd>/.flywheel/runs/<executionId>/land-status.json`：`{ "status": "pending" }`
- `executionId` 通过 **Blueprint 系统 prompt** 传递给 Claude（非 skill template）。不通过 SkillInjector 模板注入，因为 SkillInjector 写入的是共享的 `.claude/skills/flywheel-land/SKILL.md`，在 non-worktree 模式下多 session 会互相覆盖。系统 prompt 是 per-session 的，更安全：`"Landing signal path: .flywheel/runs/<executionId>/land-status.json"`
- Evidence collector 在 `collect()` 中读取该文件时：
  - 文件不存在 → `undefined`（landing 未尝试）
  - `status === "pending"` → `{ status: "failed", failureReason: "signal_missing" }`（landing 启动了但未完成——硬超时/crash）
  - `status === "merged"` 或 `status === "failed"` → 原样传递
  - 解析失败 → `{ status: "failed", failureReason: "parse_error" }`

这避免了基于 skill 注入静态推断的问题：SkillInjector 会无条件注入所有 `SKILL_TEMPLATES`，所以 "skill 文件存在" 不等于 "landing 被尝试过"。只有 agent 实际执行 land 流程并写入 `pending` marker 时，才算 landing attempted。

同时，`collect()` 方法不再需要 `landingExpected` 参数。三态语义完全由文件状态决定。

//   if (ctx.landingStatus?.status === "failed") → 直接返回 blocked

export const HR_010_LANDING_FAILED: HardRule = {
  id: "HR-010",
  description: "PR landing failed — block for review",
  priority: 0,  // 高于 HR-007 timeout (priority 1)，确保 landing 状态优先
  evaluate: (ctx) =>

**6. event-route guard**（`packages/teamlead/src/bridge/event-route.ts`）：

在 `auto_approve` 路径中，检查 evidence payload 中的 `landingStatus`：
- `landingStatus.status === "merged"` → 跳过 `approveExecution()`（PR 已 merged），直接写 `approved` 状态
- 否则 → 执行现有 `approveExecution()` 逻辑

**7. 向后兼容**：

- `landingStatus` 在所有接口中都是 `optional`
- 没有 `land-status.json`（老 session、或 agent 未创建 PR / 未执行 land 流程）→ `landingStatus` 为 `undefined` → 走完整 Decision Layer 流程（与现有行为完全一致）
- `land-status.json` 存在但 `status === "pending"`（land 启动了但未完成——硬超时/crash）→ `landingStatus = { status: "failed", failureReason: "signal_missing" }` → 路由 `blocked`
- `land-status.json` 存在但解析失败 → `landingStatus` 设为 `{ status: "failed", failureReason: "parse_error" }` → 路由 `blocked`

### Land Skill 流程

```
flywheel-land (在 PR 创建后由 agent 自动执行)
  ↓
1. 获取 PR 信息
   - `gh pr view --json number,headRefName,baseRefName,statusCheckRollup,reviewDecision,mergeStateStatus,mergeable,state`
   - 保存 baseRefName 供后续 rebase 使用
  ↓
2. 轮询等待 CI + Review:
   - 轮询间隔: 30s
   - 每次轮询: `gh pr view <PR> --json statusCheckRollup,reviewDecision,mergeable,state,mergedAt`
   - CI pending → 继续等待
   - CI 失败 → 分析失败原因 → 尝试修复（最多 2 次）→ push → 重新轮询
   - CI 超时（30min 无进展）→ 写 land-status.json (failed, ci_timeout)，escalate，停止
   - Review changes_requested → 读取 review comments → 尝试修改 → push → 重新轮询
   - Review 未提交且项目需要 required reviews → 继续等待（最多 30min），超时则写 land-status.json (failed, review_timeout)，escalate
  ↓
3. 检查 merge conflicts:
   - mergeable === "CONFLICTING" → 先检查 single-author（`git log --format='%ae' origin/<baseRefName>..HEAD | sort -u`，>1 则 block）→ `git fetch origin <baseRefName> && git rebase origin/<baseRefName>` → force push → 回到轮询
   - mergeable === "UNKNOWN" → 等待 30s 后重试（GitHub 刚 push 后经常先返回 UNKNOWN）
   - rebase 失败超过 2 次 → 写 land-status.json (failed, merge_conflict)，escalate
  ↓
4. Merge:
   - 所有 checks 通过 + review 通过（或无 required reviews）+ mergeable
   - 执行: `gh pr merge <PR> --squash`（立即 merge，不使用 --auto）
   - **Post-merge 验证**：`gh pr view <PR> --json state,mergedAt` 确认 `mergedAt` 存在
   - 如果 `state !== "MERGED"`：等待 30s 后重试 `gh pr view`。仍未 merged → **post-merge queue 检测**：这可能是 merge queue repo（`gh pr merge` 只入队不立即 merge）。写 `land-status.json (failed, merge_queue_unsupported)`，escalate，停止
   - 本 step 不支持 merge queue repo。检测方式是 post-merge 而非 pre-detect，避免 `mergeStateStatus === "BEHIND"` 等状态的误判（`BEHIND` 也出现在普通需要 rebase 的 PR 上）
  ↓
5. Post-merge cleanup（best-effort，在写 terminal signal 之前尝试）:
   - 删除远程分支: `git push origin --delete <branch>`（best-effort：如果 GitHub 已自动删除分支或 ref 不存在，视为成功，不阻塞后续步骤）
  ↓
6. 写 terminal signal（最后一步，紧贴 session 退出）:
   - 无论 cleanup 成功与否，都必须写 terminal signal
   - 写 land-status.json (merged, prNumber, mergedAt, mergeCommitSha)
   - **时序关键**：terminal signal 必须在 cleanup 尝试之后写入。TmuxRunner sentinel 一旦检测到终态就 `settle(false)` 交还控制权给 Blueprint，Blueprint 可能 kill window。如果 signal 在 cleanup 之前写，branch deletion 等操作会被打断
   - Session 正常退出，Blueprint 后续流程接管通知
```

### 与项目 Skill 的关系

Land skill 是 **Flywheel 提供的通用 skill**，但项目可以 override：

- 如果项目 repo 已有自定义 landing 流程（如 GeoForge3D 的 `/ship-pr` slash command），在 `.flywheel/config.yaml` 的 `skills` section 中配置 `land_command` 即可
- **`land_command` 的语义**：它是一个**原样插入 prompt 的调用指令字符串**。例如 `land_command: /ship-pr` 会让 Blueprint 系统 prompt 说 "使用 `/ship-pr` 替代 `flywheel-land` skill 来 land PR"。无论是 Claude Code skill、slash command、还是 MCP tool 调用，都可以作为 `land_command` 的值——只要 Claude Code session 能执行即可
- **不需要修改 SkillInjector 来发现/复制项目 skill** — Claude Code 本身会自动发现 working directory 下的 `.claude/skills/`，repo-native skills 和 slash commands 天然可用
- Flywheel 只负责注入自己的默认 `flywheel-land` skill，项目 override 通过 prompt 指令 + config 声明实现

### Skill 命名

统一使用 `flywheel-land` 作为 canonical skill identifier，与现有 `flywheel-context`、`flywheel-tdd` 等命名保持一致：
- 模板文件：`packages/edge-worker/src/skill-templates/flywheel-land.ts`
- `SKILL_TEMPLATES` key：`"flywheel-land"`
- 注入目录：`.claude/skills/flywheel-land/SKILL.md`
- SKILL.md frontmatter `name:`：`flywheel-land`
- Blueprint 系统 prompt：引用 `flywheel-land` skill
- `land_command` 配置示例：自定义命令名（如 `/ship-pr`），与默认 `flywheel-land` 不冲突

# flywheel-land — Automated PR Landing

## Trigger
After creating a PR, follow this skill to automatically land it.

## Steps
0. Write pending marker immediately. The system prompt provides the exact landing signal path (e.g., `.flywheel/runs/<executionId>/land-status.json`). Create the directory and write the pending marker:
   ```
   mkdir -p .flywheel/runs/<executionId>
   echo '{"status":"pending"}' > .flywheel/runs/<executionId>/land-status.json
   ```

1. Get PR info including base branch:
   `gh pr view --json number,headRefName,baseRefName,statusCheckRollup,reviewDecision,mergeStateStatus,mergeable,state`
2. Poll every 30s until CI + review ready:
   - `gh pr view <PR_NUMBER> --json statusCheckRollup,reviewDecision,mergeable,state,mergedAt`
   - If CI fails: analyze failure, fix code, push, continue polling (max 2 fix attempts)
   - If CI times out (30min no progress): write land-status.json (failed), escalate, stop
   - If review has change requests: read comments, address feedback, push, continue polling
   - If mergeable is CONFLICTING:
     1. Verify single-author: `git log --format='%ae' origin/<baseRefName>..HEAD | sort -u | wc -l` — if > 1, write land-status.json (failed, merge_conflict), escalate, stop (do NOT force push multi-author PRs)
     2. If single-author: `git fetch origin <baseRefName> && git rebase origin/<baseRefName> && git push -f`
   - If mergeable is UNKNOWN: wait 30s, re-poll
3. When all checks pass + review approved (or no required reviews) + mergeable:
   - `gh pr merge <PR_NUMBER> --squash` (immediate merge, NOT --auto)
   - Verify: `gh pr view <PR_NUMBER> --json state,mergedAt` — confirm mergedAt exists
   - If not merged after 30s retry: likely merge queue repo. Write land-status.json (failed, merge_queue_unsupported), escalate, stop
4. Post-merge cleanup (best-effort, BEFORE writing terminal signal):
   - Delete remote branch: `git push origin --delete <branch>` — if this fails (branch already deleted by GitHub auto-delete or ref not found), treat as success and continue
5. Write terminal signal (LAST step before session exit — write regardless of cleanup outcome):
   - On success: `echo '{"status":"merged","prNumber":<N>,"mergedAt":"<ts>"}' > .flywheel/runs/<executionId>/land-status.json`
   - On failure: `echo '{"status":"failed","prNumber":<N>,"failureReason":"<reason>"}' > .flywheel/runs/<executionId>/land-status.json`
   - (Use the same landing signal path from the system prompt as in step 0)
   - CRITICAL: Write signal AFTER cleanup attempt. TmuxRunner sentinel triggers Blueprint immediately on terminal signal — any pending operations will be interrupted.
6. Session exits normally — Blueprint handles notifications

## Failure Modes
- CI fix attempts exhausted (2x): write land-status.json (failed, ci_fix_exhausted), escalate, stop
- CI timeout (30min no progress): write land-status.json (failed, ci_timeout), escalate, stop
- Review wait timeout (30min): write land-status.json (failed, review_timeout), escalate, stop
- Merge conflict after 2 rebase attempts: write land-status.json (failed, merge_conflict), escalate, stop
- gh pr merge fails: write land-status.json (failed, merge_failed), escalate, stop
- Merge queue repo detected: write land-status.json (failed, merge_queue_unsupported), escalate, stop

## CRITICAL: Always write land-status.json
Whether landing succeeds or fails, you MUST write the landing signal file (path provided in system prompt) before stopping.
This file is the machine-readable signal that Blueprint uses to determine session outcome.
Branch cleanup failures must NOT prevent signal writing — always write the signal regardless of cleanup outcome.

## Do NOT
- Use `gh pr merge --auto` (this only enables auto-merge, doesn't actually merge)
- Force push on PRs with other authors' commits
- Send session_completed events yourself — Blueprint handles this
- Keep retrying indefinitely — max 2 CI fix attempts, 30min wait timeouts
- Update Linear issue status — handled by a separate mechanism (out of scope for this skill)

### Scope 限制 — Multi-Repo

**默认 `flywheel-land` 仅支持 single-repo 或 worktree-enabled 路径**。Multi-repo 项目（如 umbrella repo 包含 sub-repos，`run-issue.ts` 以 `enableWorktree: false` 启动）不兼容默认 skill，因为：
- `gh pr view` / `gh pr merge` 在 root repo 上执行，但 PR 可能在 sub-repo
- `git rebase` / `git push` 也会在错误的 repo 上执行

Multi-repo 项目 **完全不支持自动 landing**（包括 `land_command` override）。原因：即使 repo-aware 的自定义命令能 merge sub-repo PR，Blueprint 的 evidence collection、`GitResultChecker`、`assertCleanTree` 等都只对单个 `cwd`（umbrella root）操作，landing outcome 的审计数据会错。Blueprint 在 non-worktree 模式下（`!this.worktreeManager`）跳过 landing，系统 prompt 保持 legacy 行为（"stop and wait after PR"）。

### Out of Scope
- 项目特定的 deploy 逻辑（`:cool:` emoji 等——由项目的 skill 处理）
- Rollback 机制（merge 后发现问题→revert）
- PR 多人 review 协调（单人 review 或 auto-approve 足够）
- Branch protection bypass（不应该绕过保护规则）
- 可执行的 land helper 脚本（本 step 是 prompt-only skill；如果需要可执行脚本，在后续 step 中根据实际运行效果决定）
- SkillInjector 的项目 skill 发现机制（Claude Code 自动发现 repo-native skills）
- **Merge queue 支持**：启用 merge queue 的 repo 不在本 step 范围内。检测到 merge queue → `blocked`，后续 step 按需支持
- **Multi-repo landing**：umbrella / multi-repo 项目完全不支持自动 landing（包括 `land_command` override），因为 Blueprint evidence/git 采集只对单个 `cwd` 有效
- **Linear issue 状态更新**（`gh` 不支持 Linear API；需要单独的 Linear mutation 机制，如专用 skill 或 MCP。推迟到后续 step，或由 TeamLead 在收到 `session_completed` + `landingStatus: merged` 后统一处理）

## Risks

1. **CI 修复循环**: Agent 可能在 CI 失败时不断尝试修复但修不好。限制最多 2 次 fix attempt，超过则 escalate（使用 `flywheel-escalation` skill）。
2. **Force push 风险**: Rebase + force push 可能丢失他人的 review comments。只在 single-author PR 上允许 force push，land skill 明确指示这一点。
3. **与项目 skill 冲突**: 如果项目已有 `/ship-pr` 且行为不同（如先 deploy 再 merge），通过 `land_command` 配置 override 确保优先使用项目的命令。
4. **Session timeout**: Land 流程可能耗时超过 30min（CI 慢 + review 等待）。超时策略：land 内部 30min 无进展则 escalate，session 45min 硬超时作为 safety net。
5. **`mergeable: UNKNOWN`**: GitHub 在 push 后不会立即计算 mergeable status。Land skill 需要处理这个状态，等待后重试。
9. **Rebase 污染 evidence 模型**: Landing 过程中的 `rebase + force-push` 会改变 `baseSha..HEAD` 范围，可能将 upstream commits 算入 agent 工作。这是现有 evidence 模型的已知限制（手动 rebase 也会触发）。本 step 不修改 evidence 管线——对于 landed PR，Decision Layer 已 early return `auto_approve`，evidence 中的 commit count/diff 仅用于审计记录，不影响路由决策。如果未来需要精确的 agent-only diff，可在 evidence collector 中改用 GitHub PR API（`GET /repos/:owner/:repo/pulls/:number/files`）替代本地 git range。
6. **重复 merge**: Land skill 已 merge 但 TeamLead auto_approve 再尝试 merge。通过 Task 5 的 landing status signal + event-route guard 解决。
7. **land-status.json 未写入**: 如果 session 被硬超时终止，land skill 可能来不及写终态。此时 `land-status.json` 仍为 `pending` → evidence collector 转换为 `{ status: "failed", failureReason: "signal_missing" }` → 路由 `blocked`，需人工审核。
8. **Merge queue repo**: `gh pr merge --squash` 在 merge queue repo 上只会入队不会立即 merge。本 step 检测到 queue 状态后直接 `blocked`，不做队列等待。
