# v1.0 Phase 2 — Disable Auto-Approve + Always Notify

> Status: codex-approved
> Author: Annie + Claude
> Date: 2026-03-10
> Prerequisite: v1.0 Phase 1 (merged)

## Motivation

CEO 反馈（2026-03-10）：

1. **Always notify**: 每个 session 开始跑起来后，不管结果如何，都要发 Slack 通知。不能 miss 任何东西。
2. **No auto-approve**: 所有 PR 都需要 CEO approve 才能 merge。Claude Code 不应该自己 merge PR。
3. 以后运行成熟后，再逐步过渡到 auto-approve。

> "目前所有东西都需要我这边 Approve 才可以。不然的话，我有时候都不知道你们那边在发生什么。"

## Notification Path Ownership

**Authoritative path: TeamLead/OpenClaw (Bridge)**

- `TEAMLEAD_OWNS_SLACK=true` 已设置在 `~/.zshrc`
- Bridge `event-route.ts` 对每个 event 无条件发 hook 到 OpenClaw → product-lead → Slack
- 这是生产路径，本 plan 的核心变更集中在此

**Legacy path: `scripts/run-issue.ts` + `scripts/run-project.ts`**

- 仅用于手动测试 / standalone 开发
- 本 plan 同步修复 legacy path 的 route filter，保持一致性
- 当 `TEAMLEAD_OWNS_SLACK=true` 时，legacy `SlackNotifier` 被禁用

## Design Decision

**Quick fix 方案**（不做 hold-and-wait，session 仍然正常退出）：

```
Claude Code → 创建 PR → 监控 CI → CI pass → 写 "ready_to_merge" 信号 → session 退出
                                                        ↓
                                         Decision Layer → needs_review（强制）
                                                        ↓
                              ┌─ Bridge path: hook → product-lead → Slack → CEO
                              │                              ↓
                              │               CEO approve → Bridge ApproveHandler → gh pr merge
                              │
                              └─ Legacy path: SlackNotifier → Slack → waitForAction → ApproveHandler
```

核心变更：`flywheel-land` 不再 merge PR，改为报告 CI 状态后退出。CEO 通过 Slack approve 后，Bridge 的 `ApproveHandler` 执行 `gh pr merge --squash`。

**Hold-and-wait（session 暂停等审批）是后续 v1.0 Phase 3 的工作**，本 plan 不涉及。

#### 2.5 `packages/edge-worker/src/SlackNotifier.ts`

> Codex Round 3 Issue #3: normalization 不完整。
> Codex Round 4 Issue #1: 过度归一化 — `merged` backward-compat 的 `auto_approve` 被误当成待审批。

**Legacy path `merged` backward-compat 限制声明**：
- Legacy Slack path（`SlackNotifier`）不承担 `merged` backward-compat 语义
- 生产路径（Bridge/OpenClaw）正确处理 `merged` → `approved`（Wave 3.3）
- Legacy path 仅在 `TEAMLEAD_OWNS_SLACK` 未设置时使用（手动测试）
- 新部署后不会有新的 `merged` signal（`flywheel-land` 改为写 `ready_to_merge`）
- 此限制记录在 Scope/Risk 中

修改方案 — **`normalizedDecision` 单点真值 + `merged` 场景不发通知**：
```typescript
async notify(ctx, decision, extra?): Promise<{ sent: boolean }> {
    // Backward compat: HR-LANDED (already merged) — skip legacy notification
    // Production path (Bridge/OpenClaw) handles merged sessions correctly
    if (decision.route === "auto_approve" && decision.hardRuleId === "HR-LANDED") {
        return { sent: false };
    }

    // Normalize remaining auto_approve (shouldn't happen after final guard, defense-in-depth)
    const normalizedDecision: DecisionResult =
        decision.route === "auto_approve"
            ? { ...decision, route: "needs_review" as DecisionRoute }
            : decision;

    const blocks = normalizedDecision.route === "needs_review"
        ? this.buildNeedsReviewBlocks(ctx, normalizedDecision)
        : this.buildBlockedBlocks(ctx, normalizedDecision, extra);

    const fallbackText = normalizedDecision.route === "needs_review"
        ? `Review Required: ${ctx.issueIdentifier} — ${ctx.issueTitle}`
        : `Blocked: ${ctx.issueIdentifier} — ${ctx.issueTitle}`;

    await this.messageService.postMessage({
        token: this.config.botToken,
        channel: this.config.channelId,
        text: fallbackText,
        blocks,
    });
    return { sent: true };
}
```

这样 `HR-LANDED` 的 "already done" 语义在 legacy path 也不会产生错误的 "Review Required" 通知。

## Risk

| Risk | Impact | Mitigation |
|------|--------|------------|
| CEO 没有及时 approve → PR 积压 | Medium | Stuck watcher 会提醒 |
| ApproveHandler merge 失败（CI 过期） | Medium | 失败消息回显到 Slack，CEO 可以 retry |
| flywheel-land 改动后 session 行为变化 | Low | skill template 只改指令，不影响 runner |
| 部署窗口中旧 `merged` signal | Low | `merged` 保留 "already done" 语义 |
| Legacy + Bridge 重复通知 | Low | TEAMLEAD_OWNS_SLACK=true 禁用 legacy |
| `land_command` 绕过 policy | Low | 本 plan 显式覆盖两条 prompt 分支 |
| Legacy path 对 `merged` 不发通知 | Very Low | 生产用 Bridge path；legacy 仅测试用；新部署无新 `merged` signal |
