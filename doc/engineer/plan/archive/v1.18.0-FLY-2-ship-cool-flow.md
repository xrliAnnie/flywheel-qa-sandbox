# Plan: Flywheel Ship :cool: Flow — CI Green Gate + Fix Loop

**Version**: v1.18.0
**Issue**: FLY-2
**Date**: 2026-03-30
**Source**: `doc/engineer/exploration/new/FLY-2-ship-cool-flow.md`, `doc/engineer/research/new/FLY-2-ship-pr-skill.md`
**Status**: codex-approved (4 rounds Codex + self-review Round 5, Codex rate-limited)

## 概述

为 Flywheel 创建项目级 `/ship-pr` skill，实现 CI green gate + fix loop + pre-merge archive + merge。同时更新 orchestrator.md 和 spin.md，统一 ship 入口。

## 约束

- GitHub Free plan → 无 branch protection
- Flywheel 无 deploy → 无需 `:cool:` Action
- 需要覆盖两种场景：orchestrator worker + 手动使用
- **"Never push main" 规则**：`/ship-pr` 不触碰 main，所有变更在 PR 分支内完成

## 设计决策

### D1: 职责边界 — `/ship-pr` 只管 CI + merge，不管 post-merge bookkeeping

**问题**: Post-merge 更新（MEMORY.md, CLAUDE.md milestone, VERSION bump, Linear → Done）需要在 merge 后执行，但直推 main 违反项目规则。

**决策**: `/ship-pr` 的职责止于 merge。Post-merge bookkeeping 是**调用方**的责任：
- **Orchestrator**: Ship+Cleanup 步骤 inline 处理（已有此模式）
- **手动**: 用户自行处理或 `/spin` Archive 段引导
- **MEMORY.md**: 是本地文件（`~/.claude/projects/.../memory/`），不在 git repo 中，不涉及 git push

**理由**: 保持 `/ship-pr` 单一职责，避免 push-main 争议，符合现有 orchestrator 实践。

### D4: Post-merge docs-to-main — 已知限制

**现状**: Orchestrator post-merge bookkeeping（CLAUDE.md milestone + VERSION bump）需要 commit + push main。这与 "never push main" 规则存在张力。

**当前处理**: 沿用 orchestrator.md 既有模式（docs-only direct commit）。这是 FLY-2 scope **不改变**的现有行为。

**未来改进**（out of scope）: 可以引入 `docs/bookkeeping` PR 自动化，让 post-merge 更新也走 PR。作为独立 issue 跟踪。

### D2: 接口设计

**用法**: `/ship-pr [<pr-number>] [--yes]`
- 无参数: 推断 PR + 交互确认
- `<pr-number>`: 指定 PR，仍需确认（除非 `--yes`）
- `--yes`: 跳过确认，直接执行（orchestrator worker 模式）

### D3: 失败上报

`/ship-pr` 不承担 escalation routing。CI 修不好时：
- 输出详细失败信息（哪个 step、什么错误、已尝试什么）
- **退出**，让调用方决定如何处理（orchestrator worker 可 SendMessage 回 orchestrator lead，手动模式由用户决定）

### Step 1: 创建 `.claude/commands/ship-pr.md`

项目级 skill，覆盖全局 GeoForge3D 版本。

allowed-tools:

  - Bash(git -C:*)

```

注意：不含 Linear MCP 或 SendMessage — 这些不属于 `/ship-pr` 的职责。

#### Phase 0: Identify the PR

**输入**: `/ship-pr [<pr-number>] [--yes]`

1. 如果提供了 `<pr-number>`，直接使用
2. 否则推断：当前 branch 关联的 open PR → session context → `gh pr list`
3. 如果没有 `--yes`，展示 PR 信息并确认；有 `--yes` 则跳过

**Metadata hydration** — 获取后续阶段需要的变量：
```bash
PR_NUMBER={from step 1-2}
HEAD_BRANCH=$(gh pr view {PR_NUMBER} --json headRefName -q '.headRefName')
ISSUE_ID=$(echo "$HEAD_BRANCH" | grep -oE '(GEO|FLY)-[0-9]+' | head -1)
# 如果 branch 名不含 issue ID，从 PR title 或 body 中提取
```
后续 Phase 1-2 使用 `{PR_NUMBER}`, `{HEAD_BRANCH}`, `{ISSUE_ID}` 均来自此步骤。

#### Phase 1: CI Green Gate

```
1a. 等待 CI 完成：
    gh pr checks {PR_NUMBER} --watch
    # exit code: 0 = all pass, non-zero = has failures

1b. 如果失败，查询详情：
    gh pr checks {PR_NUMBER} --json name,bucket,link,workflow,state

    从失败 check 的 link 字段提取 run ID：
    # link 格式: https://github.com/{owner}/{repo}/actions/runs/{RUN_ID}/job/{JOB_ID}
    # 用 gh api 或 gh run list 获取最新 failed run：
    gh run list --branch {HEAD_BRANCH} --workflow CI --status failure --limit 1 --json databaseId -q '.[0].databaseId'

1c. 读失败日志：
    gh run view {RUN_ID} --log-failed

1d. 分类并修复：

    | 信号 | 分类 | 自动修复 |
    |------|------|----------|
    | biome/lint error | lint | pnpm format && git add -A && git commit -m "fix: auto-format" && git push |
    | TypeScript compile error | build | 读错误 → 修代码 → git add → commit → push |
    | Type error | typecheck | 读错误 → 修类型 → git add → commit → push |
    | Test assertion failure | test | 读断言 → 修代码 → git add → commit → push |
    | Timeout / network / 5xx | flaky | gh run rerun {RUN_ID} --failed |
    | Missing secret / permission | config | 输出错误详情 → 退出 |

1e. 修复后重新等待 CI：
    gh pr checks {PR_NUMBER} --watch

1f. 循环最多 3 次。3 次后：
    输出: "CI failed after 3 fix attempts. Last failure: {step}: {error summary}"
    退出，让调用方决定下一步。
```

#### Phase 2: Pre-merge Archive + Merge

```
2a. Archive pipeline docs（如果存在）：
    # 先检测哪些文件存在，逐个 mv
    ISSUE_ID="{ISSUE_ID}"  # e.g., FLY-2 or GEO-145

    for dir_pair in "doc/engineer/plan/inprogress:doc/engineer/plan/archive" "doc/engineer/research/new:doc/engineer/research/archive" "doc/engineer/exploration/new:doc/engineer/exploration/archive"; do
      src="${dir_pair%%:*}"
      dst="${dir_pair##*:}"
      files=$(find "$src" -name "*${ISSUE_ID}*" -type f 2>/dev/null)
      if [ -n "$files" ]; then
        for f in $files; do
          git mv "$f" "$dst/"
        done
      fi
    done

    # 如果有 archive 变更，commit + push
    if ! git diff --cached --quiet; then
      git commit -m "docs: archive ${ISSUE_ID} docs before merge"
      git push
      # CI 会重跑，等待
      gh pr checks {PR_NUMBER} --watch
    fi

2b. Merge（仅在 CI 绿时）：
    gh pr merge {PR_NUMBER} --squash --delete-branch

2c. 确认 merge 成功：
    gh pr view {PR_NUMBER} --json state,mergedAt

    输出: "PR #{PR_NUMBER} merged at {mergedAt}. Branch deleted."
```

### Step 2: 更新 orchestrator.md Ship Gate

修改 Section 7 "Ship + Cleanup (Teammate Executes)" 的 Step 2。

**替换当前 A-G 步骤为**：

```markdown
2. Teammate executes **ALL** of the following (do not skip any):

   **A. Ship via /ship-pr**
   `/ship-pr {PR_NUMBER} --yes`
   Handles: CI green gate → fix loop (max 3 attempts) → archive docs → squash merge.
   If CI stuck after 3 attempts, report failure to orchestrator lead and stop.

   **B. Post-merge bookkeeping** (all commands anchored to main repo via `git -C`)

   Switch to main and pull:
   ```bash
   cd ~/Dev/flywheel
   git checkout main && git pull origin main
   ```

   VERSION bump (if first ship of sprint) — run from main repo dir:
   ```bash
   cd ~/Dev/flywheel && bash -c 'source .claude/orchestrator/config.sh && current=$(get_feature_version) && if [ "$current" != "{SPRINT_VERSION}" ]; then bump_feature_version minor; fi'
   ```

   Update CLAUDE.md milestone table + commit (all from main repo):
   ```bash
   cd ~/Dev/flywheel
   # Edit CLAUDE.md to add milestone row
   git add CLAUDE.md doc/VERSION
   git commit -m "docs: update CLAUDE.md + VERSION after {ISSUE_ID} merge (PR #{N})"
   git push origin main
   ```

   Update MEMORY.md (local file at `~/.claude/projects/...`, not git tracked):
   - Mark issue as ✅ Done in Next Steps table
   - Add one-line summary with PR number

   **C. Update Linear**
   Use the cached Linear MCP handle (resolved at orchestrator startup per Section "Startup" step 2) to mark the issue as Done. Example (namespace varies by environment):
   `{linear_mcp_handle}(id="{ISSUE_ID}", state="Done")`

   **D. Clean up worktree** (from main repo, after leaving worktree)
   ```bash
   cd ~/Dev/flywheel
   git worktree remove {worktree_path} 2>/dev/null
   git branch -D {branch} 2>/dev/null  # if not already deleted by --delete-branch
   ```

   **E. Report completion**
   Mark task as completed. Report merge + cleanup summary.
```

**关键变化**:
- Step A 从 inline CI check 变为调用 `/ship-pr`
- Step B 保留 post-merge bookkeeping inline（`/ship-pr` 不负责这部分）
- VERSION bump 函数名修正：`get_feature_version`（不是 `get_current_version`）
- 所有 main repo 操作用 `git -C ~/Dev/flywheel`（非交互，不依赖 cd）

### Step 3: 更新 spin.md Archive 段

修改 spin.md "Stage: Archive (after PR merges)" 段（约 line 154-180）。

**替换为**：
```markdown
### Stage: Archive (after PR merges)

After implementation is shipped (PR merged to main):

**If `/ship-pr` was used**: docs are already archived (Phase 2 handles it). Skip to post-merge bookkeeping.

**If merged without `/ship-pr`**: archive docs manually from main repo:
```bash
cd ~/Dev/flywheel && git pull origin main
ISSUE_ID="{ISSUE_ID}"
for dir_pair in "doc/engineer/plan/inprogress:doc/engineer/plan/archive" "doc/engineer/research/new:doc/engineer/research/archive" "doc/engineer/exploration/new:doc/engineer/exploration/archive"; do
  src="${dir_pair%%:*}"; dst="${dir_pair##*:}"
  for f in $(find "$src" -name "*${ISSUE_ID}*" -type f 2>/dev/null); do
    git mv "$f" "$dst/"
  done
done
git diff --cached --quiet || git commit -m "docs: archive ${ISSUE_ID} docs after merge"
```

**Post-merge bookkeeping** (always required):
1. Update CLAUDE.md: add milestone to table
2. Update MEMORY.md (local file): move to archived, mark Done
3. Update Linear issue status to Done
4. Clean up worktree: `cd ~/Dev/flywheel && git worktree remove ../flywheel-{slug}`
5. Commit + push docs changes
```
