## Step 1: Bridge — `GET /api/linear/issues` 端点

### 1.1 端点规格

**Path**: `GET /api/linear/issues`
**Auth**: `tokenAuthMiddleware(config.apiToken)`（与 create/update 一致）

**Query Parameters**:

| 参数 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `project` | string | 否 | — | Project 名称 (e.g., "Flywheel") |
| `state` | string | 否 | — | 状态类型，逗号分隔: `backlog,started` |
| `labels` | string | 否 | — | Label 名称，逗号分隔: `Product,Operations` |
| `limit` | number | 否 | 50 | 返回数量上限 (max 250) |

**Response** (`200`):
```json
{
  "issues": [
    {
      "id": "uuid",
      "identifier": "GEO-280",
      "title": "Sprint 收尾流程",
      "description": "## Background\n...",
      "priority": 2,
      "priorityLabel": "High",
      "state": "Backlog",
      "stateType": "backlog",
      "labels": ["Product"],
      "assignee": null,
      "url": "https://linear.app/geoforge3d/issue/GEO-280/...",
      "createdAt": "2026-03-20T00:00:00.000Z",
      "updatedAt": "2026-03-28T00:00:00.000Z"
    }
  ],
  "count": 15,
  "truncated": false
}
```

**Response** 额外字段:
- `count`: 本次返回的 issue 数量
- `truncated`: boolean — 是否有更多 issue 未返回（`hasNextPage`）

**Error Responses**:
- `501` — LINEAR_API_KEY not configured
- `502` — Linear API error

### 1.2 实现

**文件**: `packages/teamlead/src/bridge/plugin.ts`

在现有 `app.patch("/api/linear/update-issue", ...)` 之后添加。

**用 SDK `rawRequest` 做 GraphQL 查询**，避免 N+1 async 属性问题，同时保持与现有 Linear 接入模式一致：

```typescript
app.get(
  "/api/linear/issues",
  tokenAuthMiddleware(config.apiToken),
  async (req, res) => {
    if (!config.linearApiKey) {
      res.status(501).json({ error: "LINEAR_API_KEY not configured" });
      return;
    }

    const project = req.query.project as string | undefined;
    const stateParam = req.query.state as string | undefined;
    const labelsParam = req.query.labels as string | undefined;
    const limitRaw = req.query.limit !== undefined
      ? parseInt(String(req.query.limit), 10) : 50;
    // NaN (non-numeric input) → default 50; valid int → clamp to 1-250
    const limit = Number.isNaN(limitRaw)
      ? 50
      : Math.min(Math.max(1, limitRaw), 250);

    // Build Linear GraphQL filter
    const filter: Record<string, unknown> = {};
    if (project) {
      filter.project = { name: { eq: project } };
    }
    if (stateParam) {
      const states = stateParam.split(",").map((s) => s.trim());
      if (states.length === 1) {
        filter.state = { type: { eq: states[0] } };
      } else {
        filter.state = { type: { in: states } };
      }
    }
    if (labelsParam) {
      const labels = labelsParam.split(",").map((l) => l.trim());
      if (labels.length === 1) {
        filter.labels = { name: { eq: labels[0] } };
      } else {
        filter.or = labels.map((name) => ({
          labels: { name: { eq: name } },
        }));
      }
    }

    const query = `
      query ListIssues($filter: IssueFilter, $first: Int) {
        issues(filter: $filter, first: $first, orderBy: updatedAt) {
          nodes {
            id
            identifier
            title
            description
            priority
            priorityLabel
            url
            createdAt
            updatedAt
            state { name type }
            labels { nodes { name } }
            assignee { name }
          }
          pageInfo { hasNextPage endCursor }
        }
      }
    `;

    try {
      const { LinearClient } = await import("@linear/sdk");
      const client = new LinearClient({ apiKey: config.linearApiKey });
      const result = await client.client.rawRequest(query, {
        filter,
        first: limit,
      });

      const data = result.data as {
        issues: {
          nodes: Array<{
            id: string;
            identifier: string;
            title: string;
            description: string | null;
            priority: number;
            priorityLabel: string;
            url: string;
            createdAt: string;
            updatedAt: string;
            state: { name: string; type: string };
            labels: { nodes: Array<{ name: string }> };
            assignee: { name: string } | null;
          }>;
          pageInfo: { hasNextPage: boolean; endCursor: string | null };
        };
      };

      const nodes = data.issues.nodes;
      const issues = nodes.map((n) => ({
        id: n.id,
        identifier: n.identifier,
        title: n.title,
        description: n.description,
        priority: n.priority,
        priorityLabel: n.priorityLabel,
        state: n.state.name,
        stateType: n.state.type,
        labels: n.labels.nodes.map((l) => l.name),
        assignee: n.assignee?.name ?? null,
        url: n.url,
        createdAt: n.createdAt,
        updatedAt: n.updatedAt,
      }));

      res.json({
        issues,
        count: issues.length,
        truncated: data.issues.pageInfo.hasNextPage,
      });
    } catch (err) {
      console.error(
        "[linear-proxy] list-issues failed:",
        (err as Error).message,
      );
      res.status(502).json({ error: "Linear API error" });
    }
  },
);
```

**设计决策**:
- 用 `@linear/sdk` 的 `client.client.rawRequest()` — 与现有 Bridge Linear 接入一致（都通过 `await import("@linear/sdk")` + `new LinearClient()`），同时避免 N+1 async 属性问题
- `limit` 策略：NaN → 默认 50，有效数字 → clamp 到 1-250。不返回 400（简单 clamp 更宽容）
- 返回 `count`（本次返回数量）和 `truncated`（是否有更多），不返回不精确的 `total`
- `description` 原样返回（Simba 用 Claude 理解，不需要 Bridge truncate）
- `limit` 用 `parseInt()` 而非 `Number()` — 避免浮点数穿透到 GraphQL `Int` 变量
- Linear project 名称是 "Flywheel"（与 Linear API 返回一致，projects.json projectName 也是 "geoforge3d" 但 Linear project 是 "Flywheel"）

## Step 2: GeoForge3D — Simba Agent.md Triage 行为

### 2.1 新增 Section: Triage

在 `cos-lead/agent.md` 的 "事件处理" section 之后，添加 "Triage" section：

```markdown
## Triage — LNO/ICE 优先级评估

CEO 说 "triage" / "看看 backlog" / "有什么要做的" 时，执行以下流程：

### Step 1: 数据收集

执行三个查询：

```bash
# 1. 查询 Linear 未完成 issues（Backlog + Todo + In Progress）
curl -s -H "Authorization: Bearer $TEAMLEAD_API_TOKEN" \
  "$BRIDGE_URL/api/linear/issues?project=GeoForge3D&state=backlog,unstarted,started&limit=100"

# 2. 查询当前活跃 sessions（正在跑的 Runner）
curl -s -H "Authorization: Bearer $TEAMLEAD_API_TOKEN" \
  "$BRIDGE_URL/api/sessions"

# 3. 查询 Runner 并发容量
curl -s -H "Authorization: Bearer $TEAMLEAD_API_TOKEN" \
  "$BRIDGE_URL/api/runs/active"
```

**注意**: 不要用 `mode=recent`（那是历史记录），用默认的 `/api/sessions` 获取真正的 active sessions。

### Step 2: LNO 分类

根据 North Star 对每个 issue 分类：

**North Star**: 完成 Flywheel 自动化系统，让 CEO 全程通过 Discord 操作，不需要打开终端。

| 分类 | 标准 | 例子 |
|------|------|------|
| **L (Leverage)** | 直接推进自动化、提升 Lead/Runner 能力、CEO workflow 改善。10x return | 新的 Lead 能力、关键 bug 修复、自动化流程 |
| **N (Neutral)** | 有价值但非关键。1:1 return | 代码质量、非关键 bug、nice-to-have |
| **O (Overhead)** | 当前阶段不需要。负 ROI | 过度工程化、未来才需要的功能 |

### Step 3: ICE 打分（仅 Leverage 项）

对每个 Leverage issue 打分 (1-10):
- **I (Impact)**: 对 North Star 的影响多大
- **C (Confidence)**: 有多确定能做好
- **E (Ease)**: 实现难度（10=最容易）
- **ICE = (I + C + E) / 3**，保留一位小数

### Step 4: 生成报告

格式（注意 Discord 2000 字符限制，必要时拆分多条）：

```
📋 Triage 报告 — {日期}
Triage candidates: {N} 个可分配 issue (Product {X} / Operations {Y} / 需手动路由 {Z})
Runner 容量: {running}/{max} 个在跑，还能开 {available} 个

🔴 Leverage (可分配)
1. {identifier} {title} [{label}] ICE {score}
2. ...

🟡 Neutral (可分配)
3. {identifier} {title} [{label}]
4. ...

⚪ Overhead (当前不做)
5. {identifier} {title} — {原因}

📌 已在进行中 (仅展示)
- {identifier} {title} [{label}] [运行中/待审核/进行中]

建议: {简短建议}
{如果 truncated: "⚠️ 还有更多 issue 未显示，需要我继续看吗？"}
需要你确认或调整。@Peter @Oliver 有意见吗？
```

### Step 5: 分类 issue 状态（展示 vs 可分配）

从 `/api/sessions` 活跃 sessions 和 issue stateType 确定每个 issue 的处理方式：

**展示分类**:
- 有 active session 且 `status === "running"` → `[运行中]` — 展示但不可分配
- 有 active session 且 `status === "awaiting_review"` → `[待审核]` — 展示但不可分配
- `stateType === "started"` 且无 active session → `[进行中]` — 展示但不可分配（可能有人手工推进或有 branch）
- `stateType === "backlog"` 或 `"unstarted"` 且无 active session → **可分配**

**报告结构**:
- 主体：可分配的 issue（backlog/unstarted，无 active session）→ LNO 分组 + ICE 打分
- 附录："已在进行中" 区块 → 列出 `[运行中]`、`[待审核]`、`[进行中]` 的 issue（仅展示，不参与分配）
- 如果 `truncated=true`，末尾注明"还有更多 issue 未显示"

**报告首行**: 用 "Triage candidates" 而非 "Backlog"（因为候选集包含多种状态）

### Step 6: CEO 确认后分配

**确认协议**（避免误分配）：
- Simba 发完 triage 报告后，进入 **待确认状态**
- 只有当 CEO **直接回复 triage 报告**或**明确提到 triage/分配**时才执行分配
- 模糊的 "OK" 如果不是紧跟 triage 报告上下文，Simba 应追问："你是在确认刚才的 triage 报告吗？"
- CEO 可以说 "Simba，按 triage 分配" 或 "分配 GEO-280 GEO-285" 来明确触发
- 如果已经生成新的 triage 报告，旧报告的确认自动失效
- 显式指定的 GEO-XX 必须在最近一份报告的可分配列表中，否则 Simba 拒绝并解释原因

**设计选择**: 不用正式 state machine（如 reportMessageId/expiresAt）。Simba 是 Claude Code agent，天然维护 session context，能判断"最近的 triage 报告"是哪一份。Phase 1 用 prompt-level context 足够；如果实际运行中出现误触发，Phase 2 (GEO-288) 再引入正式 state machine。

**Label 路由规则**（基于 `labels: string[]`）:

| 条件 | 路由 |
|------|------|
| 恰好命中一个已配置 Lead label (Product/Operations) | 自动路由到对应 Lead |
| 零个已配置 Lead label (e.g., 只有 Marketing) | → `需要手动路由` 区块，告知 CEO |
| 多个已配置 Lead label (e.g., Product + Operations) | → `需要手动路由` 区块，告知 CEO |

确认后：

1. 按上述规则分组，只自动分配"恰好命中一个 Lead"的 issue
2. `需要手动路由` 的 issue 在 core channel 告知 CEO："以下 issue 需要你指定分配：GEO-XX (Product+Operations), GEO-YY (Marketing)"
3. 在各 Lead 的 chat channel 发送分配消息：

```
CEO 确认了 triage，分配给你：

1. GEO-280 Sprint 收尾 — Runner post-merge 自动关闭
2. GEO-285 Context Window 管理

请按优先级顺序启动 Runner。
```

Channel IDs: Peter `1485787822894878955`, Oliver `1485789342541680661`

**关于 Lead 接收分配**: Peter/Oliver 已有 "Bubble DOWN" 行为（接收 chat channel 指令并执行，包括启动 Runner via `POST /api/runs/start`）。Simba 的分配消息标明 "来源：CEO 确认的 triage"，Lead 按现有 Bubble DOWN 流程处理。这已在 GEO-274 (Lead Start Ability) 中验证过。E2E 需验证完整链路。
```

**Triage Happy Path**:
1. 在 #geoforge3d-core 发 "Simba, triage"
2. 验证 Simba 发出 triage 报告（LNO 分组、ICE 打分、Runner 容量）
3. **直接回复** triage 报告说 "按 triage 分配"

7. 先生成 triage A，再触发 triage B（"Simba, triage again"），然后回复 triage A 说 "OK" → Simba 要求基于最新报告 B 重试
8. CEO 说 "分配 GEO-XX"，但该 issue 不在最近 triage 的 assignable set（如已标记 [进行中]）→ Simba 拒绝并解释原因

**状态分类验证（Required Preconditions — 缺任意一个则验证不完整）**:

验证前确保 Linear 中存在以下样本数据：
- P1: 至少 1 个 `stateType=started` (In Progress) 且无 active session 的 issue
- P2: 至少 1 个有 active running session 的 issue（通过 Bridge 启动）
- P3: 至少 1 个只有非 Product/Operations label 的 issue（如 Marketing 或无 label）
- P4: 1 个同时带 `Product + Operations` label 的 issue

如果缺少 P3/P4，在 Linear 中临时创建测试 issue，验证后关闭。P4 **必测**——multi-match routing 是最容易出双分配 bug 的分支。

验证项：
8. P1 的 issue 出现在 `已在进行中` 区块，**不出现**在 LNO 分组中
9. P2 的 issue 标记为 `[运行中]`，**不被分配**
10. P3 的 issue 出现在 `需要手动路由` 区块，**不向任何 Lead 发消息**
11. P4 出现在 `需要手动路由` 区块（多标签不自动分配），**不向任何 Lead 发消息**

## 风险

| 风险 | 影响 | 缓解 |
|------|------|------|
| Linear GraphQL schema 变更 | 端点返回 502 | GraphQL 字段都是 stable API，变更风险低 |
| Discord 2000 字符溢出 | 报告被截断 | Agent.md 指示 Simba 拆分多条消息 |
| LNO 判断不准 | CEO 需要频繁调整 | 报告格式支持 CEO 修改，这是 feature not bug |
| Backlog 超过 limit | 报告不完整 | `truncated` 标志 + agent 提示"还有更多" |
| 误分配（CEO 说 OK 但不是指 triage）| 错误启动 Runner | 确认协议要求明确 triage 上下文 |
