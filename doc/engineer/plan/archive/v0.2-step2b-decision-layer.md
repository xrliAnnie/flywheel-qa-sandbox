# Plan: v0.2 Step 2b — Decision Layer

## Context

Step 2a (PR #6, merged) wired the core integration pipeline:
- TmuxRunner HTTP callback + pane_dead dual-path completion
- Blueprint worktree → skill injection → Claude Code → evidence collection
- ExecutionEvidenceCollector gathering rich git evidence
- Composition root in `scripts/run-issue.ts`

Step 2b plugs in the **Decision Layer** — after evidence collection, automatically decide how to handle the result (auto-approve, notify CEO, or shelve).

**Architecture doc**: `doc/architecture/v0.2-architecture.md` §2.2
**Design doc**: `doc/engineer/exploration/new/v0.2-decision-layer.md`
**External survey**: `doc/engineer/exploration/new/v0.2-external-repo-survey.md` (Conductor retry policy, Maestro typed errors)

---

## Step 2b Goal

> Evidence → Hard Rules (deterministic) → Haiku Triage (LLM) → Haiku Verify (optional) → DecisionResult → AuditLog

Step 2b does NOT add Slack notifications or CEO button callbacks — those are Step 2c. Decision Layer outputs a `DecisionResult` that Blueprint logs and returns. The actual notification/reaction execution happens in Step 2c.

**Scope: Decision logic only.** The `DecisionResult.route` controls Blueprint's return value but doesn't execute side effects (no Slack, no merge, no shelve).

---

## Step 2b Completion Criteria

1. `DecisionLayer.decide(ctx, cwd)` returns a typed `DecisionResult`
2. HardRuleEngine evaluates 8 deterministic rules (short-circuit on first match)
3. HaikuTriageAgent classifies into `auto_approve` / `needs_review` / `blocked`
4. HaikuVerifier double-checks `auto_approve` decisions with full diff
5. FallbackHeuristic handles LLM failure (never auto-approves)
6. AuditLogger persists every decision to SQLite
7. Blueprint.run() integrates DecisionLayer after evidence collection
8. FlywheelError typed error hierarchy introduced (types only — existing string errors NOT migrated in Step 2b)
9. RetryPolicy types defined (execution deferred to Step 3 DagDispatcher)
10. PreHydrator enriched with labels, projectId, issueIdentifier
11. All existing tests pass; new unit + integration tests verify decision logic
12. `pnpm build && pnpm test` green

## Task 1: FlywheelError + RetryPolicy Types

**Create**: `packages/core/src/flywheel-error-types.ts`
**Modify**: `packages/core/src/index.ts` — add exports

### Interface

```typescript
/**
 * Typed error hierarchy — introduces typed errors alongside existing string errors.
 * Migration of existing string error paths deferred to Step 2c/Step 3.
 * From Maestro sealed exception pattern (external-repo-survey.md §3.3).
 * Distinguishes startup vs runtime vs decision errors for retry policy.
 */
export type FlywheelError =
  | { type: 'runner_timeout'; sessionId?: string; elapsed: number }
  | { type: 'runner_startup_failure'; reason: string }
  | { type: 'git_conflict'; worktree: string; files: string[] }
  | { type: 'git_infra_error'; command: string; reason: string }
  | { type: 'hook_callback_timeout'; token: string }
  | { type: 'worktree_create_failure'; reason: string }
  | { type: 'decision_escalation'; reason: string }
  | { type: 'terminal_error'; reason: string };

/** Whether an error is retryable */
export function isRetryable(err: FlywheelError): boolean {
  return err.type === 'runner_timeout'
    || err.type === 'runner_startup_failure'
    || err.type === 'hook_callback_timeout';
}

/**
 * Retry policy — from Conductor retry pattern (external-repo-survey.md §1.2).
 * Claude sessions are expensive, so defaults are conservative.
 * NOTE: Step 2b defines types only. Actual retry loop is Step 3 (DagDispatcher).
 */
export interface RetryPolicy {
  maxRetries: number;        // default: 1
  delaySeconds: number;      // default: 30
  backoff: 'fixed' | 'linear' | 'exponential';
  backoffRate?: number;      // multiplier for linear/exponential
}

export const DEFAULT_RETRY_POLICY: RetryPolicy = {
  maxRetries: 1,
  delaySeconds: 30,
  backoff: 'fixed',
};

/** Calculate delay for a given attempt */
export function retryDelay(policy: RetryPolicy, attempt: number): number {
  const base = policy.delaySeconds * 1000;
  switch (policy.backoff) {
    case 'fixed': return base;
    case 'linear': return base * attempt * (policy.backoffRate ?? 1);
    case 'exponential': return base * Math.pow(policy.backoffRate ?? 2, attempt - 1);
  }
}
```

### Interface

```typescript
import Anthropic from "@anthropic-ai/sdk";
import type { LLMClient } from "flywheel-core";

export class AnthropicLLMClient implements LLMClient {
  private client: Anthropic;

  constructor(apiKey?: string) {
    this.client = new Anthropic({ apiKey });
  }

  async chat(params: {
    model: string;
    messages: Array<{ role: string; content: string }>;
    max_tokens: number;
  }): Promise<{ content: string }> {
    const response = await this.client.messages.create({
      model: params.model,
      max_tokens: params.max_tokens,
      messages: params.messages.map(m => ({
        role: m.role as "user" | "assistant",
        content: m.content,
      })),
    });
    const textBlock = response.content.find(b => b.type === "text");
    return { content: textBlock?.text ?? "" };
  }
}
```

**Create**: `packages/edge-worker/src/decision/HardRuleEngine.ts`

### Interface

```typescript
import type { ExecutionContext, HardRuleResult } from "flywheel-core";

export interface HardRule {
  id: string;
  description: string;
  priority: number;
  evaluate: (ctx: ExecutionContext) => HardRuleResult;
}

export class HardRuleEngine {
  constructor(rules?: HardRule[])
  registerRule(rule: HardRule): void
  /** Short-circuit — first triggered rule wins (sorted by priority) */
  evaluate(ctx: ExecutionContext): HardRuleResult | null
}
```

### 8 Rules (7 from architecture doc §2.2 + 1 new from Codex review)

The engine evaluates in two passes: **block rules first** (sorted by priority), then **escalate rules** (sorted by priority). First triggered rule in each pass wins. This ensures `block` actions (which map to `route: 'blocked'` → DagDispatcher shelve+halt) are never masked by earlier `escalate` rules.

| ID | Priority | Trigger | Action |
|----|----------|---------|--------|
| HR-007 | 1 | exitReason === 'timeout' | **block** |
| HR-008 | 2 | partial === true | escalate |
| HR-001 | 3 | Labels contain security/auth/billing | escalate |
| HR-002 | 4 | consecutiveFailures >= 3 | escalate |
| HR-003 | 5 | Changed files match *.env* / *secret* / *credential* | escalate |
| HR-004 | 6 | linesAdded > 500 | escalate |
| HR-005 | 7 | Labels contain breaking-change | escalate |
| HR-006 | 8 | (reserved for trust score < 300, Phase 5) | escalate |

**Evaluation order**: Block rules (HR-007) are evaluated first. If no block triggers, escalate rules (HR-008..HR-006) are evaluated in priority order. This prevents HR-008 (partial evidence) from masking HR-007 (timeout block).

**HR-008 (NEW)**: When evidence is partial (some git commands failed), force `escalate` to prevent auto-approving based on incomplete data. From Codex review round 1 feedback.

**Note**: HR-006 is registered but always returns `{ triggered: false }` in Step 2b (trust score not yet implemented). This preserves ordering for Phase 5.

### Interface

```typescript
import type { ExecutionContext, DecisionResult } from "flywheel-core";

export interface IDecisionLayer {
  decide(ctx: ExecutionContext, cwd: string): Promise<DecisionResult>;
}

export interface FullDiffProvider {
  getFullDiff(cwd: string, baseSha: string): Promise<string>;
}

export class DecisionLayer implements IDecisionLayer {
  constructor(
    private hardRules: HardRuleEngine,
    private triage: HaikuTriageAgent,
    private verifier: HaikuVerifier,
    private fallback: FallbackHeuristic,
    private auditLogger: AuditLogger,
    private diffProvider: FullDiffProvider,
    // NOTE: NO fixed cwd — each decide() call receives cwd as parameter.
    // This is required because Blueprint is reused across nodes with different worktrees.
  )

  async decide(ctx: ExecutionContext, cwd: string): Promise<DecisionResult>
}
```

### Flow

```
1. hardRules.evaluate(ctx) → triggered?
   - action === 'block'    → return { route: 'blocked', source: 'hard_rule' }
   - action === 'escalate' → return { route: 'needs_review', source: 'hard_rule' }
2. try triage.triage(ctx) catch → fallback.evaluate(ctx, error)
3. if route === 'auto_approve' → fullDiff = diffProvider.getFullDiff(cwd, ctx.baseSha)
4. try verifier.verify(ctx, fullDiff) catch → route = 'needs_review'
5. if verifier.approved → return auto_approve
6. if verifier.concerns → return needs_review
7. auditLogger.log(ctx, result) at every exit path (best-effort — catch errors)
```

**IMPORTANT**: HardRule `action: 'block'` maps to `route: 'blocked'`, NOT `needs_review`. This ensures HR-007 (timeout) correctly triggers DagDispatcher shelve+halt.

| 8 | AuditLogger.log called for every decision | Audit completeness |

## Implementation Notes

1. **ESM imports**: use `.js` extension (`import { X } from "./X.js"`)
2. **LLMClient interface**: defined in `packages/core` (abstract). Concrete `AnthropicLLMClient` in `packages/claude-runner` (where `@anthropic-ai/sdk` already exists). This preserves the architecture doc's model-agnostic boundary — `edge-worker` never imports an LLM SDK directly.
3. **better-sqlite3**: **NEW dependency** — must be explicitly added to `packages/edge-worker/package.json`. Not currently in the monorepo.
4. **HR-006 stub**: registered but always returns false. Preserves priority ordering for Phase 5 trust score.
5. **HR-008 partial evidence**: Highest priority in the escalate group (priority 2, after block-group HR-007). Prevents auto-approve on incomplete evidence. Added per Codex review round 1 feedback. Under two-pass evaluation, block rules (HR-007) always run before escalate rules (HR-008+), so timeout block is never masked.
6. **DecisionLayer failure**: ALWAYS degrades to `{ route: 'needs_review', decisionSource: 'decision_error_fallback' }`, never throws. Conservative: pipeline continues, CEO gets notified in Step 2c.
7. **Backward compat**: all new Blueprint/PreHydrator params are optional with safe defaults. Without DecisionLayer, Blueprint uses v0.1.1 logic (`commitCount > 0 && !timedOut`).
8. **Trust prompt detection**: conservative — case-insensitive known strings, no regex wildcards. False negatives are safer than false positives. `dismiss()` is **synchronous** matching TmuxRunner's `ExecFileFn` convention.
9. **RetryPolicy execution**: Step 2b defines types only (`RetryPolicy`, `isRetryable()`, `retryDelay()`). **No retry loop in Blueprint.** Actual retry wraps the entire `dispatchOne()` call in Step 3 (DagDispatcher parallel rewrite).
10. **fullDiff for Verifier**: `ExecutionEvidenceCollector.getFullDiff()` — new method, called lazily by DecisionLayer only on `auto_approve` path. Not stored in `ExecutionEvidence` by default (can be very large).
11. **Prompt templates**: Hardcoded in agent files (not external templates). Simple enough for Step 2b; Step 2c may extract to config if CEO wants to customize.
12. **DagDispatcher mapping**: `auto_approve` / `needs_review` → `success=true`, `blocked` → `success=false`. Step 2c adds `onNodeComplete` callback for `needs_review` notifications.
13. **consecutiveFailures**: Tracked externally by the caller (DagDispatcher or run-issue.ts script). Passed to Blueprint via `BlueprintContext.consecutiveFailures`. Step 2b passes 0 by default — Step 3 implements real tracking.
14. **Composition root**: `scripts/run-issue.ts` always enables HardRuleEngine + FallbackHeuristic + AuditLogger. Only LLM components (triage/verifier) are conditional on `ANTHROPIC_API_KEY`. Without API key, triage throws → fallback handles → never auto-approves. Hard rules (security/auth/billing/timeout/partial) are ALWAYS active.
15. **HardRule block mapping**: HardRule `action: 'block'` maps to `route: 'blocked'` (not `needs_review`). This ensures HR-007 (timeout) correctly triggers DagDispatcher shelve+halt.
16. **DecisionLayer cwd**: `cwd` is passed as parameter to `decide(ctx, cwd)`, NOT stored in constructor. This is because Blueprint is reused across nodes with different worktrees in DagDispatcher.
17. **Audit DB directory**: `mkdirSync(~/.flywheel, { recursive: true })` before `AuditLogger` init. Prevents first-run failure.
18. **FlywheelError scope**: Step 2b **introduces** FlywheelError types but does NOT migrate existing string error paths. Migration is a follow-up (Step 2c or Step 3). Existing `BlueprintResult.error: string` and `FlywheelRunResult` remain unchanged.
