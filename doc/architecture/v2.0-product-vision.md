# Flywheel v2.0 — Product Vision & Target Architecture

**Date**: 2026-03-22
**Status**: Approved (CEO confirmed direction)
**Predecessor**: `doc/architecture/v0.2-architecture.md`

---

## 一句话定义

Flywheel 是一个 **自主开发团队**：多个 AI Lead（部门负责人）各自管理 AI Runner（工程师），通过 Discord 协作，CEO 只在需要时参与。

---

## 核心理念

### CEO 的时间是最稀缺的资源

整个系统的设计目标是 **最小化 CEO 的注意力消耗**：

- Lead 能自己解决的问题，不要问 CEO
- Runner 能自己解决的问题，不要问 Lead
- 只有真正需要人类判断的决策才 escalate
- Lead 的自主程度 **随时间增长**（通过 CIPHER 等学习机制）

### 管理层级

```
CEO (Annie)
  └── Lead (部门负责人, Claude Code session)
        └── Runner (工程师, Claude Code session in tmux)
```

- **CEO** — 定方向、做重大决策、参与站会
- **Lead** — 管理者。监督 Runner、回答 Runner 问题、向 CEO 汇报、跨部门协作
- **Runner** — 执行者。写代码、跑测试、提 PR。遇到问题可以问 Lead

## 日常运行模型

### 晚间（自主运行）

```
Lead 扫 Linear backlog → 发现可做的 issue
  → 检查系统资源 (CPU/内存) → 决定并发数量
  → 启动 Runner (tmux) → 监督执行
  → Runner 遇到问题 → Lead 自主判断回答
  → Runner 完成 → Lead review 结果
  → 记录进展，准备日报
```

Lead 晚间完全自主：发现任务、调度执行、解决问题、记录结果。只有真正无法判断的问题才暂存等 CEO 回复。

### 早上（站会 + 交互）

```
09:00  各 Lead 自动 post 日报到 Project-wide Channel
       内容：昨晚完成了什么、遇到了什么问题、今天打算做什么

~10:00 CEO 进入 Project-wide Channel
       - 看各 Lead 日报
       - 回复积压的问题
       - 讨论接下来的方向和优先级
       - 给出新的指令
```

### 日间（按需交互）

- Lead 有问题随时通过 Discord 找 CEO
- CEO 可能几分钟内回复，也可能几小时后才回
- Runner 遇到问题 → 问 Lead → Lead 判断：自己答 or 等 CEO
- Runner 等待期间 **暂停但不丢失进度**

## Discord 频道架构

```
Discord Server
├── #project-standup (Project-wide Channel)
│   ├── 所有 Lead + CEO
│   ├── Daily Stand-up (异步)
│   └── 跨部门协作请求
│
├── #product-team (Product 部门)
│   ├── Product Lead + CEO
│   ├── Runner 执行状态 (Forum)
│   └── CEO ↔ Product Lead 对话 (Chat)
│
├── #ops-team (Operations 部门)
│   ├── Ops Lead + CEO
│   └── ...
│
└── #marketing-team (Marketing 部门)
    ├── Marketing Lead + CEO
    └── ...
```

### 频道职责

| 频道 | 谁在 | 用途 |
|------|------|------|
| Project-wide | 所有 Lead + CEO | 站会、跨部门协作、全局方向 |
| Team Forum | Lead + CEO | 自动化状态追踪（ForumPost + Tag） |
| Team Chat | Lead + CEO | Lead ↔ CEO 对话、指令下达 |

### 通信路径

| 场景 | 路径 | 通过 |
|------|------|------|
| CEO ↔ Lead | Team Chat Channel | Discord |
| Lead ↔ Lead | Project-wide Channel | Discord |
| CEO ↔ 所有 Lead | Project-wide Channel (站会) | Discord |
| Lead → Runner (指令) | 文件 inbox / tmux | 本地 |
| Runner → Lead (提问) | MCP tool → 文件 inbox | 本地 |
| Lead → Runner (可见性) | `tmux capture-pane` | 本地 |

**关键原则**: Discord 是 Lead 层的通信总线。Lead ↔ Runner 通信走本地机制（不经过 Discord）。

## Lead 行为模型

### 角色定义

每个 Lead 是一个 **部门负责人**：

- **管理 Runner** — 启动、监督、指导、终止
- **回答 Runner 问题** — 简单的自己答，复杂的问 CEO
- **向 CEO 汇报** — 日报、问题 escalation、进展通知
- **跨部门协作** — 在 project-wide channel 与其他 Lead 沟通
- **自主发现工作** — 扫 Linear backlog，主动领取任务
- **渐进式自主** — 初期多问 CEO，随着学习逐渐自主决策

### 自主程度演进

```
阶段 1 (初期): 大多数问题问 CEO，只有最简单的自己判断
阶段 2 (成长): 常见问题自己判断，只有新类型问题问 CEO
阶段 3 (成熟): 绝大多数问题自主处理，只有重大决策 escalate
```

通过 CIPHER 决策记忆 + agent persistent memory 实现渐进式学习。

### Lead 的工具能力

| 能力 | 实现方式 |
|------|----------|
| 看 Runner 在做什么 | `tmux capture-pane -t flywheel:{window}` |
| 给 Runner 发指令 | 写文件到 Runner inbox |
| 回答 Runner 问题 | 读 Lead inbox → 写回 Runner inbox |
| 启动 Runner | `tmux new-window` + Claude Code CLI |
| 终止 Runner | `tmux kill-window` |
| 查系统资源 | `top`, `free`, `ps` 等系统命令 |
| 查 Linear | `curl` Linear API 或 Bridge proxy |
| 执行 CEO 指令 | Bridge API (`/api/actions/*`) |

## Runner 行为模型

Runner 是 **headless 工程师** — 在 tmux window 中运行的 Claude Code session。

### Runner 的能力

- 正常工作：读代码、写代码、跑测试、提 PR
- 遇到问题：调用 `ask_lead(question)` MCP 工具 → 暂停等待 Lead 回复
- 收到指令：通过 `check_inbox()` MCP 工具读取 Lead 发来的指导
- 汇报进展：通过 `report_progress(status)` MCP 工具异步通知 Lead

### Runner 不能做的

- 不能直接跟 CEO 对话（通过 Lead 中转）
- 不能跟其他 Runner 对话
- 不能自己决定做什么 issue（由 Lead 分配）
- 不知道其他 Runner 的存在

## 资源管理

### 并发控制

- 同时运行的 Runner 数量受 **硬件资源** 限制（CPU、内存、GPU）
- Lead 启动新 Runner 前应检查系统资源
- 如果资源不足，暂缓启动，等有空间再继续
- 需要一个 **Resource Monitor** 组件提供资源使用情况

### 资源监控指标

| 指标 | 阈值 | 行为 |
|------|------|------|
| 内存使用 | > 85% | 暂停启动新 Runner |
| CPU 使用 | > 90% (持续) | 暂停启动新 Runner |
| 活跃 Runner 数 | 可配置上限 | 硬性限制 |

## 技术架构选型

### Lead Runtime: Claude Code CLI

| 决策 | 选择 | 理由 |
|------|------|------|
| Lead 运行环境 | Claude Code CLI (`--agent` + `--channels discord`) | 开箱即用的 CLAUDE.md、Hooks、MCP、Memory、tmux access |
| Lead identity | `--agent product-lead` (GEO-205) | 原生 agent persona + tool restriction + persistent memory |
| Lead ↔ CEO | Discord `--channels` plugin | 已验证可用 |
| Lead ↔ Lead | Discord (project-wide channel) | 无需自建协议 |
| Lead ↔ Runner | 自建文件 inbox + MCP tool | 受 Agent Teams 启发，但自主控制 |
| Lead 持续运行 | tmux session + supervisor script | 简单可靠 |

### 不使用 Agent Teams

Agent Teams 是 Claude Code 的原生多 agent 功能，但不适合 Flywheel：
- Lead 之间已通过 Discord 通信，不需要 file-based inbox
- Experimental 限制太多（不能 resume、一次一个 team、Lead crash 全丢）
- Agent Teams 的 "team lead" 是 task coordinator，Flywheel 的 Lead 是 department manager

### 不使用 OpenClaw

OpenClaw 之前作为 Lead runtime 使用，但 Claude Code CLI 在以下方面有本质优势：
- 本地 tmux 访问（看 Runner、管 Runner）
- 原生 CLAUDE.md、Hooks、MCP、Memory
- `--agent` identity system
- Session resume (`--resume`)

OpenClaw 的持续运行优势（launchd service）可通过 tmux + supervisor script 替代。

### Blueprint/DAG 演进

Blueprint（DAG 执行引擎）的调度和决策角色将被 Lead 吸收：
- **之前**: Linear → DAG Resolver → Blueprint → Runner
- **之后**: Linear → Lead 自主判断优先级和依赖 → Lead 直接 spawn Runner

Bridge 保留基础设施角色：
- StateStore（状态持久化）
- Forum 管理（ForumPostCreator、ForumTagUpdater）
- 审计日志（Event Journal）
- Dashboard
- CEO 查询 API

TeamLead Daemon 保留 scheduler 角色：
- Linear polling → 通知 Lead 有新 issue
- Stand-up 触发（定时提醒 Lead 发日报）
- Resource Monitor（系统资源监控）

## 部署模型

### 当前

- **单机**: Lead + Runner 都在同一台 Mac 上
- **单项目**: GeoForge3D
- **多 Lead**: Product Lead + (future) Ops Lead + Marketing Lead

### 未来扩展

- 多项目时，每个项目可独立一台机器
- 每台机器上的 Lead 和 Runner 共存（同机优先）
- 跨机器通信在当前阶段不考虑

## 与现有系统的关系

| 组件 | 现状 | v2.0 角色 |
|------|------|-----------|
| Blueprint + DAG | 执行调度引擎 | Lead 吸收调度角色，逐步淘汰 |
| Bridge API | 通信中转 + 基础设施 | 保留基础设施，不再做通信中转 |
| Decision Layer | Hard Rules + Haiku Triage | 可被 Lead 自主判断替代（渐进） |
| EventFilter | 事件分类路由 | 保留，但路由目标从 webhook 变为本地通知 |
| RuntimeRegistry | Lead runtime 管理 | 简化（只有 Claude Code runtime） |
| ForumPostCreator/TagUpdater | Discord Forum 自动管理 | 保留不变 |
| CIPHER | 决策学习记忆 | 保留，增强 Lead 自主能力 |
| StateStore (SQLite) | 状态持久化 | 保留不变 |
