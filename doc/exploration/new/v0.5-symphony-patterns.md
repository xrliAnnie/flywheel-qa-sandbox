## Pattern 1: WORKFLOW.md — 单文件配置 + 提示词模板 {#pattern-1-workflowmd}

### 问题

Flywheel 当前的配置散落在多处：
- `packages/edge-worker/src/config.ts` — 代码级配置
- Linear issue metadata — 运行时参数
- CLAUDE.md / SKILL.md — agent 指令
- 环境变量 — secrets 和 feature flags

不同目标 repo 无法轻松定制自己的工作流。

### Symphony 的做法

**一个文件 `WORKFLOW.md`**，放在目标 repo 根目录，YAML front matter 定义 orchestrator 配置，Markdown body 是 Liquid/Jinja 模板化的 agent prompt：

```yaml
---
tracker:
  kind: linear
  project_slug: "my-project"
  active_states: [Todo, In Progress, Merging, Rework]
  terminal_states: [Closed, Cancelled, Done]
polling:
  interval_ms: 5000
workspace:
  root: ~/code/workspaces
hooks:
  after_create: |
    git clone --depth 1 git@github.com:org/repo .
    npm install
  before_run: |
    git pull origin main
agent:
  max_concurrent_agents: 10
  max_turns: 20
  max_concurrent_agents_by_state:
    Merging: 2
codex:
  command: codex --model gpt-5.3-codex app-server
  approval_policy: never
  stall_timeout_ms: 300000
---

You are working on Linear ticket `{{ issue.identifier }}`

{% if attempt %}
This is retry attempt #{{ attempt }}. Resume from current workspace state.
{% endif %}

Issue: {{ issue.identifier }} — {{ issue.title }}
Status: {{ issue.state }}
Labels: {{ issue.labels }}
URL: {{ issue.url }}

Description:
{{ issue.description | default: "No description provided." }}

Instructions:
1. This is unattended. Never ask a human for follow-up.
2. Only stop for true blockers (missing auth/secrets).
3. Follow the status workflow below...

## Status Workflow
| From | To | Trigger |
|------|-----|---------|
| Todo | In Progress | Agent starts working |
| In Progress | Human Review | PR created, tests green |
| Human Review | Merging | Approved |
| Merging | Done | Merged |
| Any | Rework | Review requests changes |
```

### 我们的实现方案：`FLYWHEEL.md`

在每个目标 repo（如 GeoForge3D）根目录放一个 `FLYWHEEL.md`：

```yaml
---
# Flywheel Project Config
project_name: geoforge3d
project_repo: xrliAnnie/GeoForge3D

# Linear mapping
linear:
  team_key: GEO
  active_states: [Todo, In Progress, In Review]
  terminal_states: [Done, Cancelled]

# Agent settings
agent:
  budget_usd: 5.0
  budget_complex_usd: 10.0
  max_turns: 10
  stall_timeout_ms: 300000

# Workspace hooks
hooks:
  before_run: |
    npm install
    npm run build
  after_run: |
    npm run lint

# Concurrency
concurrency:
  max_parallel: 3
  max_by_state:
    In Review: 1  # 同时只有 1 个 PR 在 review
---

You are an autonomous coding agent working on {{ issue.identifier }}.

Issue: {{ issue.identifier }} — {{ issue.title }}
Priority: {{ issue.priority }}
Description:
{{ issue.description }}

{% if retry_attempt %}
This is retry #{{ retry_attempt }}. Previous error: {{ last_error }}
Resume from current worktree state, do not restart from scratch.
{% endif %}

## Rules
1. Run tests before committing: `npm test`
2. Follow existing code style
3. Create atomic commits with conventional commit format
4. If blocked, update the Linear issue with a comment explaining the blocker
```

### 实现要点

1. **BlueprintBuilder 改造**：读取目标 repo 的 `FLYWHEEL.md`，解析 YAML front matter 作为 `ProjectConfig`，渲染 Markdown body 作为 initial prompt
2. **模板引擎**：用 `liquidjs`（npm 包，Liquid 兼容）或简单的 `{{ var }}` 替换
3. **Fallback**：没有 `FLYWHEEL.md` 的 repo 使用默认配置（当前行为不变）
4. **验证**：启动时验证 YAML schema，invalid 时 skip dispatch 并告警

### 优先级：P0（v0.5）

### 预估 LOC：~200（YAML parser + template renderer + BlueprintBuilder 集成）

## Pattern 2: 动态工具注入 — Agent-side Tools {#pattern-2-dynamic-tools}

### 问题

当前 Flywheel 的 agent（Claude Code session）与外部系统的交互全部由 orchestrator 代理：Blueprint 收集结果 → Decision Layer 决策 → SlackNotifier 通知。Agent 自己无法主动操作 Linear 或发 Slack 消息。

### Symphony 的做法

在 agent session 启动时注入 **dynamic client-side tools**，让 agent 用 orchestrator 的凭证直接操作外部系统：

```typescript
// Symphony 在 thread/start 时注入
{
  "method": "thread/start",
  "params": {
    "dynamicTools": [
      {
        "name": "linear_graphql",
        "description": "Execute a Linear GraphQL query/mutation",
        "parameters": {
          "query": "string — the GraphQL query",
          "variables": "object — optional variables"
        }
      }
    ]
  }
}
```

Agent 在 session 中直接调用 `linear_graphql` tool 来：
- 更新 issue 状态（Todo → In Progress）
- 发评论（workpad 进度更新）
- 关联 PR（`attachmentLinkGitHubPR` mutation）

### 我们的实现方案：MCP Server 或 SKILL.md 工具

由于 Claude Code 支持 MCP（Model Context Protocol），我们可以通过 MCP server 注入工具：

**方案 A：轻量级 MCP Server（推荐）**

```typescript
// packages/edge-worker/src/FlywheelMcpServer.ts
// 启动一个 localhost MCP server，expose 以下工具给 Claude Code session

tools = {
  "flywheel_linear_update": {
    // Agent 自己更新 issue 状态
    params: { issueId: string, stateId: string },
    handler: (params) => linearClient.updateIssue(params)
  },
  "flywheel_linear_comment": {
    // Agent 发 Linear 评论（进度/blocker）
    params: { issueId: string, body: string },
    handler: (params) => linearClient.createComment(params)
  },
  "flywheel_report_progress": {
    // Agent 报告进度给 TeamLead
    params: { executionId: string, message: string, status: string },
    handler: (params) => teamleadClient.emitProgress(params)
  }
}
```

在 CLAUDE.md 或 SKILL.md 中告知 agent 这些工具的存在和用法。

**方案 B：SKILL.md + shell 命令**

如果 MCP 太重，可以在 worktree 的 `.claude/skills/linear/SKILL.md` 中提供 `gh` 和 `curl` 命令模板，让 agent 通过 shell 操作 Linear API。Symphony 的 `linear` skill 就是这种 cookbook 模式：

```markdown
# Linear Operations

## 查询 issue
gh api graphql -f query='{ issue(id: "...") { id title state { name } } }'

## 更新状态
gh api graphql -f query='mutation { issueUpdate(id: "...", input: { stateId: "..." }) { ... } }'

## 发评论
gh api graphql -f query='mutation { commentCreate(input: { issueId: "...", body: "..." }) { ... } }'
```

### 核心价值

- Agent 自己更新 Linear 状态 → orchestrator 不需要猜 agent 的进度
- Agent 自己发评论 → 实现 workpad 模式（见 Pattern 6）
- Agent 报告进度给 TeamLead → 更实时的 Slack 通知

### 优先级：P1（v0.5）

### 预估 LOC：~300（MCP server）或 ~100（SKILL.md cookbook）

## Pattern 3: 多轮续接 — Multi-turn Continuation {#pattern-3-multi-turn}

### 问题

Flywheel 当前是 single session per issue。如果 agent 没完成（超时、部分完成），Blueprint 标 failed，需要人工重新触发。没有"继续未完成工作"的概念。

### Symphony 的做法

同一个 agent 在同一个 thread 上最多跑 `max_turns`（默认 20）轮。每轮完成后：

```
1. Agent 完成一个 turn
2. Orchestrator 检查 Linear issue 是否还在 active_states
3. 如果还 active 且 turn_number < max_turns → 发 continuation prompt 开始新 turn
4. 如果到 max_turns 或 issue 不再 active → 正常退出
5. 正常退出后 orchestrator 等 1 秒再检查 → 如果 issue 仍 active，重新 dispatch（新 session）
```

**Continuation prompt**（不是重发完整 prompt）：

```
Continuation context:
- This is retry attempt #{{ attempt }} because the ticket is still in an active state.
- Resume from the current workspace state instead of restarting from scratch.
- Do not repeat already-completed investigation or validation unless needed.
- Do not end the turn while the issue remains in an active state unless blocked.
```

### 我们的实现方案

分两层：

**Layer 1: Blueprint Continuation（v0.5）**

在 `Blueprint.run()` 完成后，如果 result 不是 terminal（issue 仍在 active state），自动 retry 同一个 worktree：

```typescript
// Blueprint.run() 改造伪代码
async run(context: BlueprintContext): Promise<BlueprintResult> {
  let turnNumber = 1;
  const maxTurns = this.config.maxTurns ?? 5; // 保守起步

  while (turnNumber <= maxTurns) {
    const prompt = turnNumber === 1
      ? this.buildFullPrompt(context)
      : this.buildContinuationPrompt(context, turnNumber, maxTurns);

    const result = await this.runner.run({ ...context, prompt });

    // 检查 issue 是否还需要工作
    const issueState = await this.tracker.getIssueState(context.issueId);
    if (this.isTerminal(issueState) || result.status === 'completed') {
      return result;
    }

    turnNumber++;
  }

  return { status: 'max_turns_reached', ... };
}
```

**Layer 2: Orchestrator Continuation Retry（v0.5+）**

Blueprint 正常退出后，DagDispatcher 等 1 秒检查 issue 状态，如果仍 active 则 re-dispatch（新 session，但复用 worktree）。

### 关键设计决策

- **共享 worktree**：continuation turns 复用同一个 worktree，agent 看到自己之前的改动
- **轻量级 prompt**：后续 turns 只发 continuation guidance，不重发完整 issue description
- **保守 max_turns**：初始设 5（不是 Symphony 的 20），逐步调高
- **terminal 检测**：每个 turn 之间检查 Linear issue 状态，issue 被人工关闭时立即停止

### 优先级：P2（v0.5 或 v0.6）

### 预估 LOC：~150

## Pattern 4: 按状态并发限制 {#pattern-4-per-state-concurrency}

### 问题

当前 DagDispatcher 只有全局并发限制。如果 3 个 issue 同时进入 merge 阶段，可能产生 merge conflicts。

### Symphony 的做法

```yaml
agent:
  max_concurrent_agents: 10
  max_concurrent_agents_by_state:
    Merging: 2    # 同时最多 2 个 agent 在 merge
    Rework: 3     # 同时最多 3 个 agent 在 rework
```

Dispatch 时检查：

```
global_slots = max_concurrent_agents - running_count
state_slots = max_concurrent_agents_by_state[issue.state] - count_running_in_state(issue.state)
available = min(global_slots, state_slots)
```

### 我们的实现方案

在 `DagDispatcher.dispatchOne()` 前加检查：

```typescript
// DagDispatcher 改造
private canDispatch(issue: Issue): boolean {
  const globalSlots = this.maxParallel - this.runningCount;
  if (globalSlots <= 0) return false;

  const stateLimit = this.config.maxByState?.[issue.state];
  if (stateLimit !== undefined) {
    const runningInState = this.countRunningInState(issue.state);
    if (runningInState >= stateLimit) return false;
  }

  return true;
}
```

配置来源：`FLYWHEEL.md` 的 `concurrency.max_by_state`。

### 优先级：P1（v0.5）

### 预估 LOC：~30

## Pattern 5: Land Skill — PR 全生命周期自动着陆 {#pattern-5-land-skill}

### 问题

Flywheel 当前流程：agent 创建 PR → Blueprint 完成 → Decision Layer 决定 approve/reject → SlackNotifier 通知 CEO → CEO 点 approve → `gh pr merge`。Agent 在 PR 创建后就结束了，merge 过程完全依赖 orchestrator。

Symphony 的 `land` skill 让 agent 自己拥有 PR 从创建到 merge 的完整生命周期。

### Symphony `land` Skill 的完整流程

```
1. 确认在 PR branch 上，working tree clean
2. 本地跑完整测试套件（validation gate）
3. 如果有未提交改动 → commit skill → push skill
4. 检查 mergeability + conflicts
5. 如果有 conflict → pull skill（merge origin/main + resolve）→ push
6. 检查 review comments → 分类处理（accept/clarify/push back）
7. 启动 async watcher（land_watch.py）监控 CI + reviews + head changes
8. CI 失败 → 读日志 → 修复 → commit → push → re-watch
9. 所有 checks green + review addressed → squash-merge
10. 更新 Linear 状态到 Done
```

**Review 处理协议**：
- 每条 review comment 分类为：correctness / design / style / clarification / scope
- **Accept**：实施修复，commit + push
- **Clarify**：回复请求更多信息
- **Push back**：acknowledge + rationale + offer alternative
- **Context guard**：不盲目实施与原始 task intent 冲突的反馈

**Async Watcher**（`land_watch.py`，~450 行）：
- 三个并发任务：CI 状态、review comments、PR head 变化
- Exit code 协议：2=有 review comments，3=CI failed，4=PR head updated，5=merge conflict
- 指数退避 + jitter 处理 GitHub API rate limit
- 线程级 review tracking（知道哪些 thread 已回复）

### 我们的实现方案：`AutoLandSkill`

**Phase 1（v0.5）：简化版 — SKILL.md 指令**

在 worktree 的 `.claude/skills/land/SKILL.md` 中提供 land 工作流指令：

```markdown
# Auto Land

## 前提条件
- `gh` CLI 已认证
- 在 PR branch 上，working tree clean

## 步骤
1. 确认 PR 存在：`gh pr view --json number,url,state`
2. 跑测试：`npm test`（或 repo 的 test 命令）
3. 检查 CI：`gh pr checks`
4. 如果 CI 失败：读日志 → 修复 → commit → push → 等待 CI
5. 检查 merge conflicts：`gh pr view --json mergeable`
6. 如果有 conflict：`git fetch origin main && git merge origin/main` → resolve → push
7. 检查 reviews：`gh pr view --json reviews,reviewRequests`
8. 如果有 review comments → 处理（见 Review Protocol）
9. 所有 green → `gh pr merge --squash --auto`

## Review Protocol
- 每条 comment 回复：acknowledge + 处理方式
- 如果 comment 与 task intent 冲突 → push back with rationale
- 如果不确定 → ask for clarification in comment

## 失败处理
- CI 失败 3 次以上 → 停止，更新 Linear 评论说明 blocker
- Merge conflict 无法自动解决 → 停止，请求人工介入
```

**Phase 2（v0.6）：Watcher 模式**

移植 `land_watch.py` 的概念，用 TypeScript 实现一个轻量级 CI/review watcher：

```typescript
// packages/edge-worker/src/skills/LandWatcher.ts
export async function watchPR(prNumber: number, repo: string): Promise<LandWatchResult> {
  // 并发监控三个信号
  const result = await Promise.race([
    watchCI(prNumber, repo),        // 等 CI 完成
    watchReviews(prNumber, repo),   // 等 review comments
    watchHead(prNumber, repo),      // 等 head 变化
  ]);
  return result; // { type: 'ci_passed' | 'ci_failed' | 'review_needed' | 'head_updated' | 'conflict' }
}
```

### 优先级：P0（v0.5 Phase 1 SKILL.md），P2（v0.6 Watcher）

### 预估 LOC：SKILL.md ~80 行，Watcher ~300 行

## Pattern 6: Workpad — Linear 评论作为 Issue 级状态存储 {#pattern-6-workpad}

### 问题

Agent 的中间进度没有持久化。如果 session 中断重来，新 session 不知道之前做到了哪一步。

### Symphony 的做法

Agent 在 Linear issue 上维护一个 **Workpad 评论**——一个结构化的 Markdown checklist，每完成一步就更新：

```markdown
## Codex Workpad

### Status: In Progress

### Completed
- [x] Read and understood the issue
- [x] Created branch `flywheel-GEO-95`
- [x] Implemented auth middleware refactor
- [x] Added 3 unit tests

### In Progress
- [ ] Running full test suite

### Blocked
(none)

### Notes
- Changed JWT validation to use RS256 instead of HS256
- Test coverage: 87% → 92%
```

**关键规则**：
- 每个 issue 只有一个 Workpad comment（找到并更新，不创建新的）
- 使用 `commentUpdate` mutation 更新（不是创建新 comment）
- 如果 session 中断，新 session 先读 Workpad 了解进度
- 状态转换时更新 Workpad status

### 我们的实现方案

**通过 SKILL.md 指令实现**（不需要 orchestrator 代码改动）：

在 agent prompt 或 SKILL.md 中加入 Workpad 指令：

```markdown
## Workpad Protocol

在开始工作前，检查 Linear issue 是否已有 Workpad 评论（以 "## Flywheel Workpad" 开头）。

如果有：读取并了解之前的进度，从 "In Progress" 项继续。
如果没有：创建一条新评论。

每完成一个重要步骤，更新 Workpad：
- `gh api graphql -f query='mutation { commentUpdate(id: "COMMENT_ID", input: { body: "..." }) { ... } }'`

Workpad 格式：
\`\`\`markdown
## Flywheel Workpad

### Status: In Progress | Completed | Blocked

### Completed
- [x] Step description (timestamp)

### In Progress
- [ ] Current step

### Blocked
- Blocker description

### Evidence
- Branch: `flywheel-GEO-95`
- PR: #123
- Tests: 15/15 passing
- Coverage: 87%
\`\`\`
```

**依赖**：Pattern 2（动态工具注入 / Linear SKILL.md），agent 需要能操作 Linear API。

### 优先级：P1（v0.5，随 Pattern 2 一起）

### 预估 LOC：~0（纯 SKILL.md 指令），或 ~50（orchestrator 在 retry 时自动读取 Workpad）

## Pattern 7: 结构化 Retry 与 Continuation 语义 {#pattern-7-retry-continuation}

### 问题

Flywheel 当前没有区分"正常完成后 issue 仍然 active"和"agent 失败"。两者都走同一个路径。

### Symphony 的做法

两种完全不同的 retry 语义：

**Continuation Retry（正常退出）**：
- Agent 正常完成 turn，但 issue 还在 active state
- 等待 **1 秒** 后重新检查 issue 状态
- 如果仍 active → 重新 dispatch（`attempt = 1`）
- 场景：agent 完成了部分工作（创建了 PR），issue 进入 Review 状态，reviewer approve 后 issue 回到 active

**Failure Retry（异常退出）**：
- Agent 失败（crash、timeout、stall）
- **指数退避**：`delay = min(10000 * 2^(attempt-1), 300000)`
  - attempt 1 → 10s
  - attempt 2 → 20s
  - attempt 3 → 40s
  - attempt 4 → 80s
  - attempt 5 → 160s
  - attempt 6+ → 300s (cap)
- 每次 retry 前重新从 Linear 拉 issue 状态，确认还 eligible

**Stall Detection**：
- 如果 agent 超过 `stall_timeout_ms`（默认 5 分钟）没有活动
- 强制 kill → 按 failure retry 处理

### 我们的实现方案

```typescript
// DagDispatcher 改造
interface RetryPolicy {
  continuationDelayMs: 1000;          // 正常完成后的 re-check 延迟
  failureBaseDelayMs: 10000;          // 失败重试基础延迟
  failureMaxDelayMs: 300000;          // 最大重试延迟（5 分钟）
  stallTimeoutMs: 300000;             // 无活动超时
}

function computeRetryDelay(attempt: number, policy: RetryPolicy): number {
  return Math.min(
    policy.failureBaseDelayMs * Math.pow(2, attempt - 1),
    policy.failureMaxDelayMs
  );
}

// Blueprint 退出后
if (result.status === 'completed' && issueStillActive) {
  // Continuation retry — 1 秒后 re-check
  setTimeout(() => reDispatch(issue, { attempt: 1 }), policy.continuationDelayMs);
} else if (result.status === 'failed') {
  // Failure retry — 指数退避
  const delay = computeRetryDelay(attempt, policy);
  setTimeout(() => reDispatch(issue, { attempt: attempt + 1 }), delay);
}
```

### 优先级：P1（v0.5）

### 预估 LOC：~80

## Pattern 8: Workspace Lifecycle Hooks {#pattern-8-workspace-hooks}

### 问题

Flywheel 的 worktree setup 硬编码在 `WorktreeManager` 中。不同项目可能需要不同的初始化步骤（npm install / pip install / make setup）。

### Symphony 的做法

四个 hook 时机，通过 WORKFLOW.md 配置 shell 脚本：

| Hook | 触发时机 | 失败行为 |
|------|----------|----------|
| `after_create` | 新 workspace 创建后（首次） | **Fatal** — 中止 workspace 创建 |
| `before_run` | 每次 agent 启动前 | **Fatal** — 中止当前 attempt |
| `after_run` | agent 完成后（成功/失败） | 记录日志，**忽略**错误 |
| `before_remove` | workspace 删除前 | 记录日志，**忽略**错误，继续删除 |

执行方式：`sh -lc <script>` with workspace as `cwd`。
超时：`hooks.timeout_ms`（默认 60000ms）。

### 我们的实现方案

在 `FLYWHEEL.md` 中定义 hooks（见 Pattern 1），`WorktreeManager` 在相应时机执行：

```typescript
// WorktreeManager 改造
```

## Pattern 9: 热重载配置 {#pattern-9-hot-reload}

### 问题

Flywheel 更改配置需要重启进程。如果 TeamLead daemon 在跑着，改 max_concurrent 需要手动重启。

### Symphony 的做法

`WorkflowStore` GenServer 每 1 秒 poll `WORKFLOW.md` 文件：
- 文件修改 → 重新解析 YAML + 验证 schema
- 验证通过 → 热替换配置（影响后续 tick 的 dispatch、concurrency 限制、prompt 模板等）
- 验证失败 → **保留上一次 valid 配置**，emit 错误日志
- 不影响正在运行的 agent sessions

### 我们的实现方案

```typescript
// packages/edge-worker/src/ConfigWatcher.ts
export class ConfigWatcher {
  private lastConfig: FlywheelConfig;
  private lastMtime: number = 0;

  constructor(private configPath: string, private onUpdate: (config: FlywheelConfig) => void) {}

  start(intervalMs: number = 5000): void {
    setInterval(() => this.check(), intervalMs);
  }

  private check(): void {
    const stat = fs.statSync(this.configPath);
    if (stat.mtimeMs === this.lastMtime) return;
    this.lastMtime = stat.mtimeMs;

    try {
      const newConfig = parseFlywheelMd(this.configPath);
      validateConfig(newConfig); // NimbleOptions 风格的 schema 验证
      this.lastConfig = newConfig;
      this.onUpdate(newConfig);
      logger.info('Config reloaded from FLYWHEEL.md');
    } catch (err) {
      logger.warn('Config reload failed, keeping last valid config', { error: err.message });
      // 不更新 lastConfig — 保留上一次 valid 的
    }
  }
}
```

### 优先级：P3（nice-to-have）

### 预估 LOC：~60

## Pattern 11: Reconciliation Loop {#pattern-11-reconciliation}

### 问题

Flywheel 当前缺少"正在运行的 agent 对应的 issue 状态可能已变化"的处理。如果 CEO 在 Linear 上手动 close 了一个正在执行的 issue，agent 不会停。

### Symphony 的做法

每个 poll tick **先** reconcile，**再** dispatch：

```
on_tick:
  1. reconcile_stalled_runs()     — kill 无活动 > 5min 的 agent
  2. reconcile_tracker_state()    — 拉所有 running issues 的最新状态
     - terminal → kill agent + clean workspace
     - active → update 内存中的 issue snapshot
     - neither → kill agent（不清 workspace）
  3. dispatch_eligible_issues()   — 正常 dispatch
```

### 我们的实现方案

在 DagDispatcher 的 tick 或 TeamLead 的 EventIngestion 中加入：

```typescript
// DagDispatcher 改造
async tick(): Promise<void> {
  // 1. Reconcile — 检查所有 running issues 的 Linear 状态
  await this.reconcileRunning();

  // 2. Dispatch
  await this.dispatchEligible();
}

private async reconcileRunning(): Promise<void> {
  const runningIds = [...this.running.keys()];
  if (runningIds.length === 0) return;

  const states = await this.tracker.fetchIssueStates(runningIds);
  for (const [id, state] of states) {
    if (this.isTerminal(state)) {
      this.killAgent(id);
      this.cleanWorktree(id);
    }
  }
}
```

### 优先级：P1（v0.5）

### 预估 LOC：~60

## Pattern 12: 异步 Watcher 委托模式 {#pattern-12-async-watcher}

### 问题

Agent 在等 CI、等 review 时浪费 turns/tokens 做 polling。

### Symphony 的做法

`land` skill 启动一个独立的 Python 脚本 `land_watch.py` 作为异步 watcher：

```bash
# Agent 在 session 中执行
python3 .codex/skills/land/land_watch.py
```

Watcher 并发监控三个信号源：
1. **CI checks**：poll `gh api repos/.../check-runs`，dedup by name，等待所有 pass
2. **Review comments**：poll issue comments + review comments + review states，过滤已回复的 thread
3. **PR head changes**：检测 head SHA 变化（表示有新 push）

通过 **exit code 协议** 与 agent 通信：
- Exit 0 → CI green, no new reviews, ready to merge
- Exit 2 → 有 review comments 需要处理
- Exit 3 → CI failed
- Exit 4 → PR head updated（有人 push 了新 commit）
- Exit 5 → Merge conflict

Agent 根据 exit code 决定下一步：

```
result = run("python3 land_watch.py")
if result.exitCode == 0 → proceed to merge
if result.exitCode == 2 → handle review comments
if result.exitCode == 3 → fix CI failures
if result.exitCode == 4 → re-check (autofix commit landed)
if result.exitCode == 5 → pull + resolve conflicts
```

### 我们的实现方案

在 `.claude/skills/land/` 目录提供一个 `watch_pr.sh`（或 TypeScript），agent 可以调用：

```bash
#!/usr/bin/env bash
# .claude/skills/land/watch_pr.sh
# Usage: watch_pr.sh <pr-number> [timeout-seconds]
# Exit codes: 0=ready, 2=reviews, 3=ci-failed, 5=conflict

PR=$1
TIMEOUT=${2:-300}
ELAPSED=0
INTERVAL=15

while [ $ELAPSED -lt $TIMEOUT ]; do
  # Check CI
  STATUS=$(gh pr checks "$PR" --json state --jq '.[].state' | sort -u)
  if echo "$STATUS" | grep -q "FAILURE"; then exit 3; fi

  # Check reviews
  REVIEWS=$(gh pr view "$PR" --json reviewDecision --jq '.reviewDecision')
  COMMENTS=$(gh api "repos/{owner}/{repo}/pulls/$PR/comments" --jq 'length')
  if [ "$COMMENTS" -gt 0 ]; then exit 2; fi

  # Check mergeability
  MERGEABLE=$(gh pr view "$PR" --json mergeable --jq '.mergeable')
  if [ "$MERGEABLE" = "CONFLICTING" ]; then exit 5; fi

  # All checks passed?
  if echo "$STATUS" | grep -qv "PENDING"; then
    if ! echo "$STATUS" | grep -q "FAILURE"; then exit 0; fi
  fi

  sleep $INTERVAL
  ELAPSED=$((ELAPSED + INTERVAL))
done

exit 1  # timeout
```

### 优先级：P2（v0.6，随 Land Skill Phase 2）

### 预估 LOC：~100（shell script）或 ~300（TypeScript with proper retry/jitter）

## 附录 B: Symphony Skills 完整参考 {#appendix-b-skills}

### commit skill

**目的**：从 session 历史和 git diff 创建规范化 commit

**关键步骤**：
1. 读 session 历史获取 intent
2. `git status` + `git diff` + `git diff --staged`
3. `git add -A`（但检查是否意外添加了 build artifacts）
4. 选择 conventional type（feat/fix/refactor/...）
5. Subject <= 72 chars，imperative mood
6. Body 包含 Summary + Rationale + Tests 三个 section
7. 用 `git commit -F <tmpfile>`（不用 `-m`，保留换行）
8. 追加 `Co-authored-by: Codex <codex@openai.com>`

### push skill

**目的**：push branch + 创建/更新 PR

**关键步骤**：
1. **Validation gate**：push 前跑完整测试
2. Push with upstream tracking
3. Push 被拒绝 → 区分 sync 问题（用 pull skill）vs auth 问题（直接报错）
4. 创建或更新 PR（处理 closed PR → new branch 场景）
5. PR body 用 repo 的 pull_request_template.md
6. 更新 PR 时刷新 body 反映完整 scope（不只是最新 commit）
7. 只在必要时用 `--force-with-lease`，绝不用 `--force`

### pull skill

**目的**：同步 origin/main + 解决 merge conflicts

**关键步骤**：
1. 确保 working tree clean（commit 或 stash）
2. 启用 `git rerere`（自动记录并复用冲突解决方案）
3. `git fetch origin`
4. 先 `git pull --ff-only origin <current-branch>`（同步远程 feature branch）
5. `git -c merge.conflictstyle=zdiff3 merge origin/main`（三路合并，更清晰的 conflict markers）
6. 解决冲突：先理解双方 intent，再编辑。一个文件一个文件来，每批后跑测试
7. 确认无 conflict markers：`git diff --check`

### land skill

见 [Pattern 5](#pattern-5-land-skill) 的详细描述。

### linear skill

**目的**：Agent 通过 `linear_graphql` tool 操作 Linear

**关键操作 cookbook**：
- 查询 issue：渐进查找（key → identifier → internal id）
- 获取 team workflow states：**状态转换前必须先查**，用 exact stateId
- 更新评论：`commentUpdate` mutation
- 创建评论：`commentCreate` mutation
- 更新状态：`issueUpdate` with stateId
- 关联 PR：`attachmentLinkGitHubPR`（优于 generic URL attachment）
- 上传视频到评论：三步（fileUpload → curl PUT → commentCreate with assetUrl）

### debug skill

**目的**：调查 stuck/failed runs

**方法论**：
1. 定位 issue 在日志中的切片（按 issue_identifier grep）
2. 提取 session_id（thread-turn pair）
3. 追踪 session 生命周期（start → events → terminal）
4. 分类问题（timeout/stall/startup failure/turn failure/retry loop）
5. 收集证据

