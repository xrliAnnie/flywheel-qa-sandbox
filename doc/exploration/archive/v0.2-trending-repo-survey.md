# GitHub Trending Repo Survey — Flywheel 可用性调研

> 日期：2026-03-01
> 目的：调研 GitHub trending 上与 Flywheel 相关的项目，找出可直接复用的代码、模式和库
> 方法论：每个 repo 做 shallow clone + subagent 深度代码分析，区分真实实现与营销宣传

## 调研范围

13 个 GitHub trending repos，按与 Flywheel 的关联度分层。

## Tier 1：直接可用（v0.1.1 - v0.1.2）

### 1. superset-sh/superset

- **链接**：https://github.com/superset-sh/superset
- **Stars**：~3k | **语言**：TypeScript | **许可**：-
- **是什么**：AI Coding Agent 的桌面 IDE。Electron + React + tRPC，同时管理 10+ 个 Claude Code / Codex session，git worktree 隔离
- **关键发现**：
  - **Git worktree 管理**（`apps/desktop/src/lib/trpc/routers/workspaces/utils/git.ts`）：生产级 TS 实现，含 macOS `.app` bundle 挂起的 rename trick、porcelain 输出解析、per-project mutex
  - **Claude Code hook 注入**（`agent-wrappers-claude-codex-opencode.ts`）：通过 `claude-settings.json` 注入 `Stop` event hook → curl 本地 HTTP 通知。与 Flywheel 的 SessionEnd hook 设计完全一致
  - **命令构建**（`packages/shared/src/agent-command.ts`）：heredoc 防转义构建 claude CLI 命令
  - **Git status 无锁查询**（`getStatusNoLock`）：`--no-optional-locks` + porcelain v1 `-z` 格式
- **可用性**：⭐⭐⭐⭐⭐ — TS 代码可直接复制到 Flywheel
- **深度研究**：→ `doc/research/new/004-superset-worktree-hook-patterns.md`

### 2. microsoft/markitdown

- **链接**：https://github.com/microsoft/markitdown
- **Stars**：~89k | **语言**：Python | **许可**：MIT
- **是什么**：微软的文件→Markdown 转换工具。支持 PDF、Word、Excel、PowerPoint、HTML、图片 OCR、音频转录、YouTube 字幕等
- **关键发现**：
  - `pip install 'markitdown[all]'` 一行安装
  - CLI：`markitdown file.pdf > output.md`
  - Python API：`MarkItDown().convert("file.pdf").markdown`
  - 独立 MCP server 包：`markitdown-mcp`（stdio + HTTP 双模式）
  - 从 TS 调用：直接 `execFile("markitdown", [filePath])`
- **可用性**：⭐⭐⭐⭐⭐ — 零修改直接用，Linear issue 附件预处理
- **用法**：PreHydrator 中处理 Linear issue 的文档附件，转成 Markdown 注入 Claude Code session

### 3. ruvnet/ruflo (claude-flow)

- **链接**：https://github.com/ruvnet/ruflo
- **Stars**：~17.5k | **语言**：TypeScript | **许可**：MIT
- **是什么**：Claude Code 多 agent 编排平台（原名 claude-flow）。60+ 专用 agent，MCP 原生集成，号称 250% 效率提升
- **关键发现**：
  - **Session Forking**（`v2/src/sdk/session-forking.ts`）：利用 `@anthropic-ai/claude-code` SDK 的 `forkSession: true`，比每次 spawn 新进程快 10-20x。**Flywheel Phase 2 并行化的关键技术**
  - **Agent 路由**：加权评分（能力匹配 40% + 历史成功率 30% + 负载 20% + 优先级 10%），真实但较简单
  - **状态管理**：Task statusHistory（完整状态转换记录）+ SQLite 持久化
  - **ReasoningBank**：Pattern 的 short-term → long-term 晋升机制（按 usageCount 和 qualityScore）
  - **"自学习"真实度**：RL 算法代码是真的（PPO/Q-Learning），但端到端闭环不存在。embedding 是字符哈希伪嵌入，不是语义向量
  - **MCP 集成**：30+ MCP 工具，动态 agent type enum 填充
- **可用性**：⭐⭐⭐⭐ — Session Forking 模式高价值；RL/学习部分是 vaporware
- **深度研究**：→ `doc/research/new/004-superset-worktree-hook-patterns.md`（Session Forking 部分）

## Tier 2：Phase 2-3 可用（Decision Layer / Memory）

### 5. NevaMind-AI/memU

- **链接**：https://github.com/NevaMind-AI/memU
- **Stars**：~12.2k | **语言**：Python | **许可**：Apache 2.0
- **是什么**：24/7 自主 agent 的持久化记忆框架。Python + Rust 混合架构
- **关键发现**：
  - **三层实体模型**：Resource（原始数据）→ MemoryItem（原子记忆）→ MemoryCategory（类别/目录）。多对多关联
  - **Salience 排序**：`similarity × log(reinforcement_count + 1) × exp(-0.693 × days / halfLife)` — 可直接移植
  - **content_hash 去重**：SHA256(type:normalized_text)，防重复学习
  - **三层分级检索**：route_intention → category recall → sufficiency check → item recall → resource recall。每层可提前终止，节省 token
  - **存储后端**：PostgreSQL + pgvector / SQLite（JSON text embedding + numpy cosine）/ InMemory
  - **OpenAI Wrapper**：拦截 `chat.completions.create()`，自动注入 `<memu_context>` 到 system prompt
  - **"24/7 proactive"真实度**：部分真实（intent routing + sufficiency check），没有后台常驻线程
- **可用性**：⭐⭐⭐⭐⭐ — Phase 3 Memory 系统的直接参考
- **深度研究**：→ `doc/research/new/005-memory-architecture-survey.md`

### 6. bytedance/deer-flow

- **链接**：https://github.com/bytedance/deer-flow
- **Stars**：~23.2k | **语言**：Python | **许可**：Apache 2.0
- **是什么**：字节跳动的 "SuperAgent harness"。基于 LangGraph，给 agent 提供沙箱、子 agent 派生、长期记忆
- **关键发现**：
  - **Memory JSON schema**：`workContext / personalContext / topOfMind`（用户层）+ `recentMonths / earlierContext / longTermBackground`（历史层）+ `facts[]`（结构化事实）
  - **`MEMORY_UPDATE_PROMPT`**：完整的 LLM prompt 模板，指导如何从对话提取事实、更新各层记忆。可直接拷贝
  - **Debounced 异步写入**：`MemoryUpdateQueue`（30s debounce + threading.Timer + 同 thread_id 去重）
  - **`DanglingToolCallMiddleware`**：修复中断 session 的 orphan tool calls（插入合成 ToolMessage）— Blueprint 重试场景必备
  - **原子文件写入**：tempfile + rename 模式
  - **Sub-agent 并发截断**：在 AIMessage 层直接丢弃超额 tool calls，比靠 prompt 限制更可靠
  - **澄清场景分类**：5 种 clarification types（missing_info / ambiguous / approach_choice / risk_confirmation / suggestion）
  - **LangGraph 用法**：实际上极浅，只用了单节点 agent + 中间件链
- **可用性**：⭐⭐⭐⭐ — 记忆 prompt + JSON schema + DanglingToolCall 修复直接可用
- **深度研究**：→ `doc/research/new/005-memory-architecture-survey.md`

### 7. Shubhamsaboo/awesome-llm-apps

- **链接**：https://github.com/Shubhamsaboo/awesome-llm-apps
- **Stars**：~98.8k | **语言**：Python | **许可**：-
- **是什么**：100+ 个 LLM 应用合集（RAG、multi-agent、voice agent 等）
- **关键发现**：

  - **AG2 Adaptive Research Team**：Triage → Route → Verify → Synthesize 流程，与 Decision Layer 的 Haiku 判断逻辑对应

  - **EvoAgentX**：动态工作流生成（LLM 根据 goal 自动创建 WorkFlowGraph），Phase 5 参考
- **可用性**：⭐⭐⭐ — 设计模式参考，代码需要 Python→TS 翻译
- **深度研究**：→ `doc/research/new/006-decision-layer-patterns.md`

## Tier 3：架构参考

### 9. alibaba/OpenSandbox

- **链接**：https://github.com/alibaba/OpenSandbox
- **Stars**：~3.6k | **语言**：Python/Go/TypeScript | **许可**：Apache 2.0
- **是什么**：阿里的通用 AI 应用沙箱平台。Docker/K8s 隔离，多语言 SDK
- **关键发现**：
  - **TS SDK**：`@alibaba-group/opensandbox`（npm），`Sandbox.create()` / `commands.run()` / `files.*`
  - **Docker 注入机制**：execd 二进制通过 volume mount 注入任意 base image，无需修改镜像
  - **Egress 网络控制**：DNS proxy + nftables sidecar（仅 Linux，macOS 不可用）
  - **Claude Code 示例**：`examples/claude-code/main.py`
- **可用性**：⭐⭐⭐ — 当前本地 Mac + tmux 不需要，但项目规模扩大后可能在 remote Mac 上运行，沙箱隔离会变得必要。SDK 现成可用，值得持续关注
- **备注**：v0.1.4 早期发布，生产经验未知。未来 remote execution 场景下可评估

### 11. X-PLUG/MobileAgent

- **链接**：https://github.com/X-PLUG/MobileAgent
- **Stars**：~7.8k | **语言**：Python | **许可**：MIT
- **是什么**：阿里通义的 GUI 自动化 agent，视觉理解 → 自动操作手机/电脑
- **关键发现**：

- **可用性**：⭐⭐⭐ — InfoPool + 错误恢复模式可用于 DagDispatcher / Decision Layer

## Follow-up Research Sessions 建议

基于本次调研，建议开设以下研究 session：

| Session | 主题 | 输入 | 输出 |
|---------|------|------|------|
| **R1** | Worktree + Hook 模式 | superset-ai git.ts + ruflo session-forking | 更新 v0.1.1 architecture（hook 实现）+ v0.2 并行化设计 |
| **R2** | Memory 架构设计 | memU 三层模型 + deer-flow JSON schema + deer-flow prompts | `.flywheel/` 记忆系统 spec（Phase 3） |
| **R3** | Decision Layer 模式 | awesome-llm-apps Trust Layer + DevPulseAI fallback + MobileAgent InfoPool | CIPHER Decision Layer spec（Phase 2） |
| **R4** | Skill 注入系统 | claude-scientific-skills 格式 + deer-flow SKILL.md | Flywheel 专用 skill 设计 + 注入机制 |
| **R5** | 沙箱隔离 + Remote Execution | OpenSandbox SDK + deer-flow sandbox | 安全执行方案评估 + remote Mac 执行架构（Phase 3+） |
| **R6** | 多机协调（远期） | ruflo Raft/BFT consensus + 分布式 agent 调度 | 多 Mac 执行时的共识机制评估（Phase 5+） |

→ 详细 guidance 见各 research 文档。

### 关于 R5/R6 的补充说明

**Remote Mac Execution（R5）**：当前 Flywheel 跑在本地 Mac + tmux，但项目规模扩大后（如多个 project 并行），可能需要在 remote Mac（如 Garage Mac Mini）上执行。此时 OpenSandbox 的隔离能力、egress 控制、lifecycle 管理都会变得有价值。

**Multi-Machine Consensus（R6）**：如果 Flywheel 未来需要跨多台机器运行（多个 Mac Mini 执行不同 project），ruflo 的 Raft/BFT 共识模式可以作为参考。这是非常远期的需求，但值得记录。
