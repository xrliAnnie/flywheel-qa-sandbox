### P0 — Must Pass (blocks ship)

| ID | AC Ref | E2E Test Case | Verification Method |
|----|--------|---------------|---------------------|
| E01 | AC-GATE-BLOCK, AC-E2E | **Full gate lifecycle**: Run `flywheel-comm gate brainstorm` in background → verify it blocks → run `flywheel-comm respond` in separate process → verify gate exits 0 | CLI exit codes + timing |
| E02 | AC-GATE-TYPE, AC-BACKWARD-COMPAT | **CommDB schema verification**: After gate command runs, query CommDB via `sqlite3` → verify `checkpoint='brainstorm'`, `type='question'`, new columns exist with defaults | `sqlite3` CLI direct query |
| E03 | AC-CONTENT-REF | **Long message content-ref**: Run gate with >2KB message → verify content_ref file exists on disk → verify CommDB has `content_type='ref'` and `content_ref` path → verify file content matches original | `sqlite3` + `ls` + `cat` on filesystem |

## Test Implementation Notes

- Tests will be implemented as **bash scripts** (not vitest) — each test is a self-contained bash function
- CommDB setup: `flywheel-comm` CLI creates DB on first `gate`/`ask` call
- Concurrency: gate runs in background (`&`), respond runs in foreground, then `wait` for gate
- Discord tests (E11-E12): require Bridge running + Chrome MCP tools available
- Cleanup: each test creates temp directory, removes on exit
